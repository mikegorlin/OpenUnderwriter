<objective>
Implement the adversarial critique engine that produces false positive, false negative, contradictory signal, and data completeness caveats -- displayed both as inline caveat badges on specific signals in the main worksheet body AND as a dedicated Devil's Advocate section, without modifying any scores. Rule-based detection finds the issues, LLM generates human-readable explanations per user decision.

Purpose: Force intellectual honesty in the worksheet by systematically challenging the recommendation. A good underwriter mentally plays devil's advocate; this makes it explicit and systematic. Rule-based detection finds the issues, LLM-generated narratives explain them in underwriter language.

Output: Adversarial Pydantic models, rule-based detection engine, YAML rule catalog, LLM narrative generation step, ScoreStage Step 18 integration, Devil's Advocate section + inline caveat badges in HTML worksheet, and comprehensive tests.

    **Step 3: Create adversarial_rules.yaml in brain/framework/.**
    Rule catalog for all 4 check types. Each rule is declarative YAML, Python code evaluates:

    false_positive_rules (8-10 rules):
    - High CEO comp + strong stock performance = comp may be justified
    - High leverage + asset-heavy sector (utilities/real estate) = leverage may be normal
    - Insider selling + planned 10b5-1 program = selling may be routine
    - High litigation + financial sector = litigation frequency normal for banks
    - Revenue decline + COVID/pandemic year = decline may be cyclical
    - Auditor change + company growth phase = change may be upgrade
    - Board turnover + M&A/restructuring = turnover may be planned
    - Short interest high + recently IPO'd = short interest normal for new listings

    false_negative_rules (8-10 rules):
    - CLEAR on governance + very small board (<5) = potential missed governance risk
    - CLEAR on litigation + high SCA-frequency sector = may have undisclosed risk
    - CLEAR on insider selling + lockup expiry approaching = risk not yet visible
    - CLEAR on financial distress + significant off-balance-sheet items = risk hidden
    - CLEAR on regulatory + heavily regulated sector = regulatory risk underweight
    - CLEAR on restatement + frequent accounting policy changes = risk building
    - CLEAR on cyber + large customer data holdings = exposure underweight
    - CLEAR on ESG + heavy environmental footprint = ESG risk underweight

    contradiction_rules (6-8 rules):
    - Revenue growing + margins declining = potential revenue quality issue
    - Strong governance score + executive departure = contradiction suggests incomplete picture
    - Low litigation risk + high legal fees = potential undisclosed matters
    - Conservative accounting + aggressive guidance = mixed signals on management approach
    - High insider ownership + no insider buying + stock decline = insiders not backing company
    - Strong balance sheet + credit downgrade = external assessment contradicts financials

    data_completeness_rules (6-8 rules):
    - No proxy data (DEF 14A) = governance conclusions unreliable
    - No recent 10-K = financial data may be stale
    - No litigation data = litigation assessment incomplete
    - No insider transaction data = insider behavior unknown
    - No peer comparison data = benchmarking unavailable
    - <50% of signals evaluated = overall assessment confidence low

    Each rule has: id, target_signal (or signal prefix), mitigating_signals/expected_signals/contradicting_signals, minimum_mitigating/minimum_evidence count, headline_template, explanation_template (used as LLM fallback), severity.

    **Step 4: Create adversarial_engine.py.**
    Four pure functions, each takes signal_results + rules + optional state/company, returns list[Caveat]:

    `check_false_positives(signal_results, rules, *, state=None)`:
    - For each rule: find target signal in results, check if TRIGGERED
    - If triggered: check mitigating signals -- if enough are CLEAR, create caveat
    - Confidence = mitigating_count / total_mitigating_signals
    - Initial explanation from template (placeholder -- LLM enriches later in runner)

    `check_false_negatives(signal_results, rules, *, state=None)`:
    - For each rule: find target signal in results, check if CLEAR
    - If clear: check company profile conditions that suggest exposure
    - Look for exposure_indicators (other signals or company attributes)
    - Confidence based on how many exposure indicators present

    `check_contradictions(signal_results, rules)`:
    - For each rule: check pair of signals with expected opposing statuses
    - If signal_a is TRIGGERED and signal_b is CLEAR (or vice versa) in a logically contradictory way
    - Confidence based on how directly contradictory (1.0 for direct contradiction, 0.5 for indirect)

    `check_data_completeness(signal_results, rules)`:
    - Count signals by data_status: EVALUATED vs DATA_UNAVAILABLE vs NOT_APPLICABLE
    - For each rule: check if specific critical data categories are missing
    - Flag if overall evaluation rate below threshold

    Each check function produces Caveat objects with template-based `explanation` as initial value (from rule's explanation_template with signal data interpolated). The runner then enriches these with LLM narratives in Step 5.
