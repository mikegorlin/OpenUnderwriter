<objective>
Create the unified post-extraction litigation classifier module and wire it into the extraction pipeline.

Purpose: Cases currently get classified differently by each extractor (SCA, derivative, LLM, web). This creates inconsistent coverage_type and legal_theories values, duplicate entries across lists, and boilerplate reserves mixed in with real cases. A single post-extraction pass fixes all of these.

Output: `litigation_classifier.py` with four public functions, wired into `extract_litigation.py` between cross-validation and summary generation.
</objective>

       - Also match on filing year (within 1 year) + court (if both present)
       - When match found: select primary = highest confidence source. Merge fields from secondary using _enrich_case pattern. Add source tracking: store all source references in a new comment or via the existing `case_name.source` field by appending sources.
       - CRITICAL: Run dedup BEFORE year disambiguation (Pitfall 2)
       - Redistribute consolidated cases back to appropriate lists based on unified classifier's type

    3. `disambiguate_by_year(landscape: LitigationLandscape) -> None` (per D-05):

       - Preserve SourcedValue metadata (source, confidence, as_of) when creating new SourcedValue for modified name

       - Do NOT make new web search API calls (ACQUIRE boundary -- Pitfall 4). Do NOT pollute `case.allegations` with system metadata.
       - Add `cases_needing_recovery: list[dict[str, Any]] = Field(default_factory=list)` to LitigationLandscape in models/litigation.py alongside unclassified_reserves.
