Create `src/do_uw/brain/brain_calibration.py` (~300-350 lines) with:

**Pydantic models:**
- `DriftReport`: signal_id, status (DRIFT_DETECTED|OK|INSUFFICIENT_DATA), n, confidence (LOW|MEDIUM|HIGH), current_threshold (float|None), observed_mean (float|None), observed_stdev (float|None), fire_rate (float), proposed_value (float|None), basis (str), projected_impact (str)
- `FireRateAlert`: signal_id, fire_rate, alert_type (HIGH_FIRE_RATE|LOW_FIRE_RATE), recommendation (str)
- `CalibrationReport`: drift_signals (list[DriftReport]), fire_rate_alerts (list[FireRateAlert]), insufficient_data (list[DriftReport]), total_signals_analyzed (int), total_with_numeric_values (int), total_proposals_generated (int)

**Core functions:**
- `get_signal_value_distribution(conn, signal_id) -> list[float]`: Query brain_signal_runs WHERE is_backtest=FALSE, try float() on each value, skip non-numeric. Defensive parsing per Pitfall 2.
- `compute_threshold_drift(signal_id, values, current_threshold, fire_rate) -> DriftReport`: Use `statistics.mean`, `statistics.stdev`, `statistics.quantiles`. Flag when abs(current_threshold - observed_mean) > 2 * stdev. Proposed value = 90th percentile (for "higher is worse" thresholds based on operator direction) or 10th percentile (for "lower is worse"). Confidence based on N: 5-9=LOW, 10-24=MEDIUM, 25+=HIGH.
- `extract_numeric_threshold(signal) -> tuple[float|None, str]`: Parse numeric value from threshold.red or threshold.yellow string. Return (value, operator_direction). For V2 signals, use evaluation.thresholds[0].value and op directly. For V1, parse strings like ">5.0", "<1.0 current ratio" using regex. Return None if unparseable.
- `compute_fire_rate_alerts(effectiveness_entries) -> list[FireRateAlert]`: Flag entries with fire_rate > 0.80 or < 0.02. Recommendation strings: "Threshold may be too sensitive -- consider raising" for high fire rate, "Threshold may be unreachable -- consider lowering or reviewing signal relevance" for low.
- `generate_calibration_proposals(conn, drift_signals) -> int`: For each DRIFT_DETECTED signal, INSERT into brain_proposals with source_type='CALIBRATION', proposal_type='THRESHOLD_CALIBRATION', proposed_changes JSON containing threshold adjustments, backtest_results JSON containing statistical evidence (n, mean, stdev, percentiles). Return count of proposals generated.
- `compute_calibration_report(conn) -> CalibrationReport`: Orchestrator. Load signals via BrainLoader, get effectiveness data, iterate signals with numeric thresholds, compute drift for each, compute fire rate alerts, generate proposals. Return complete report.

**Implementation notes:**
- For V1 signals: extract numeric threshold from threshold.red string using the regex pattern `r'[<>=!]+\s*([\d.]+)'`. For threshold direction, check operator: `>` means "higher is worse" (use p90 as proposed), `<` means "lower is worse" (use p10 as proposed). If no operator found, fall back to fire rate analysis only.
- For V2 signals (schema_version >= 2): use evaluation.thresholds[0].value and evaluation.thresholds[0].op directly.
- Qualitative signals (threshold.type not in NUMERIC_THRESHOLD_TYPES): skip numeric drift analysis, only compute fire rate alert.
- Use existing `compute_effectiveness()` from brain_effectiveness.py to get fire rate data -- don't recompute.
- Import pattern: `from do_uw.brain.brain_unified_loader import BrainLoader` for signal definitions.

**Tests:** Create `tests/brain/test_brain_calibration.py` using an in-memory DuckDB fixture. Seed brain_signal_runs with synthetic data for each test case. Use the shared DuckDB fixture pattern from existing brain tests (create tables, insert test data, run function, assert).
