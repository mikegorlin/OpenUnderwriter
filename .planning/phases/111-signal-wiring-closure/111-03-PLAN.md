    ```python
    def resolve_signal_data(sig: dict[str, Any], state: AnalysisState) -> dict[str, Any]:
        """Resolve signal data from YAML field declarations against state.

        Resolution priority:
        1. acquisition.sources[*].fields[*].path (v7.0 YAML paths)
        2. acquisition.sources[*].fields[*].computed_from (analysis-stage outputs)
        3. acquisition.sources[*].fields[*].fallback_paths (ordered alternatives)
        4. field_path (legacy direct field key)
        5. data_strategy.field_key (legacy strategy key)
        """
    ```

    Key implementation details:
    - _resolve_path(obj, dotted_path) -> Any: traverse nested attributes/dicts, unwrap SourcedValue at each level
    - _unwrap_sourced(value) -> Any: check for .value + .source attributes, return .value if SourcedValue
    - Handle both dict access (current.get(part)) and attribute access (getattr(current, part, None)) at each path segment
    - Return dict[str, Any] keyed by field name (same contract as map_signal_data)

    Wire into signal_engine.py with PHASED MIGRATION (per research recommendation):
    1. Try resolve_signal_data(sig, state) FIRST
    2. If resolver returns empty dict AND the old mapper would have provided data, fall back to map_signal_data(sig, state)
    3. Log when fallback is used (these are signals whose YAML paths need fixing)
    4. This allows incremental path correction without big-bang risk

    The phased approach means:
    - Signals with correct YAML paths use the new resolver
    - Signals with incorrect/aspirational paths fall back to old mapper
    - Each YAML path fix moves one more signal from mapper to resolver
    - Once all signals use resolver, mapper can be deleted (Phase 115 legacy purge)
