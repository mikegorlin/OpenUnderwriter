## Architecture Patterns

### Pattern 1: Brain Build Pipeline (Gap 1)

**What:** YAML is the single source of truth. `brain build` rebuilds brain.duckdb from YAML. The runtime pipeline reads only from brain.duckdb via `BrainDBLoader.load_signals()`. signals.json is a backward-compatibility export artifact.

**Data flow:**
```
YAML (source of truth)
  |
  v
brain_build_signals.py::build_checks_from_yaml()
  |
  v
brain.duckdb (runtime query cache)
  |
  v
BrainDBLoader.load_signals() -> brain_signals_active view
  |
  v
signal_engine.py -> threshold dispatch (INFO_ONLY_TYPES vs evaluate_tiered)
```

**Key insight:** `brain.duckdb` is what the pipeline reads at runtime, NOT signals.json. `BrainDBLoader` docstring (line 1-2): "JSON files are import/export format only -- not read at runtime." However, `_validate_yaml_json_sync()` in brain_build_signals.py checks that signals.json and YAML have matching signal IDs before allowing a build. So signals.json must also be updated to avoid build failures on future runs.

**Current state verified:**
```
YAML:        threshold.type = tiered  (CORRECT - updated by Phase 52-04)
brain.duckdb: threshold_type = display (STALE - not rebuilt)
signals.json: threshold.type = display (STALE - not regenerated)
```

### Pattern 2: Dual-Key State Read (Gap 2)

**What:** Post-Phase 49 state.json uses `signal_results` key. Pre-Phase 49 state.json uses `check_results` key. Any code reading state.json must try both.

**Established precedent:** `cli_brain_trace.py:188-195`:
```python
def _get_signal_results(state: dict[str, Any]) -> dict[str, Any]:
    """Extract signal results from state, supporting legacy key names."""
    analysis = state.get("analysis", {})
    for key in ("signal_results", "check_results"):
        candidate = analysis.get(key)
        if candidate:
            return candidate
    return {}
```

**Also used in:** `validation/health_summary.py:70-97` (same dual-key pattern).

**Current broken code:** `cli_feedback.py:68`:
```python
check_results = data.get("analysis", {}).get("check_results", {})
```
This ONLY reads the legacy key. For any ticker analyzed after Phase 49, the state has `signal_results` and `check_results` doesn't exist -- so the function returns 0 triggered signals.

### Pattern 3: Nomenclature Allowlist (Gap 2)

**What:** `test_signal_nomenclature.py` greps for forbidden patterns (old `check_*` terminology) in all Python files. Files that legitimately need to reference old keys for backward compatibility are listed in `ALLOWLIST_FILES`.

**Current allowlist (relevant entries):**
- `cli_brain_trace.py` -- backward-compat fallback for reading old state files
- `health_summary.py` -- backward-compat fallback for reading old state files

**Fix pattern:** After making `cli_feedback.py` use `signal_results` as primary key, the file will still contain `"check_results"` as a fallback string. Two options:
1. Add `cli_feedback.py` to ALLOWLIST_FILES (follows existing pattern)
2. Extract a shared `_get_signal_results()` helper so cli_feedback.py doesn't contain the string directly

Option 1 is simpler and matches the established precedent. However, option 2 (extracting a shared helper) would be cleaner if there are already 3+ files using the same dual-key pattern. Assessment: there are exactly 3 files (cli_brain_trace.py, health_summary.py, cli_feedback.py), and each has slightly different surrounding logic. The allowlist approach is the right choice for a gap-closure phase.

### Pattern 4: Signal Details -> Composite Data Contract (Gap 3)

**What:** `signal_details.py` enriches SignalResult.details dicts. `brain_composite_engine.py` reads those details via `_get_details()`. The key names must match.

**Current mismatch:**

- `brain_composite_engine.py:245` reads: `insider_details.get("clusters", [])`
- `brain_composite_engine.py:434` reads: `cluster_details.get("clusters", [])`

**Fix:** Rename `"cluster_events"` to `"clusters"` in signal_details.py line 130. This is a 1-line change. No other consumers of this key exist.

### Anti-Patterns to Avoid
- **Don't add new YAML columns or fields** -- these are data flow fixes, not schema changes
- **Don't modify brain_composite_engine.py** to read `"cluster_events"` -- the writer should match the reader's established contract, not vice versa
- **Don't remove signals.json** -- it's still used as a sync validation artifact by `_validate_yaml_json_sync()`

| Problem | Don't Build | Use Instead | Why |

## Common Pitfalls

### Pitfall 1: Running `brain build` but not regenerating signals.json
**What goes wrong:** `_validate_yaml_json_sync()` checks that signal IDs match between YAML and signals.json. If signals.json has stale threshold data but the same IDs, the sync check passes but the JSON is technically out of date.
**Why it happens:** `brain build` only rebuilds brain.duckdb. signals.json regeneration is a separate `export-all` step.
**How to avoid:** After `brain build`, also run the signals.json export. The Phase 52.1 plan should include both steps.
**Warning signs:** `brain build` succeeds but `diff` between pre/post signals.json shows no changes.

### Pitfall 2: Nomenclature test still failing after code fix
**What goes wrong:** The test greps for the literal string `check_results` in Python files. Even with dual-key read, the fallback string `"check_results"` will be detected.
**Why it happens:** The nomenclature test is a simple grep guard, not a semantic analysis.
**How to avoid:** Add `cli_feedback.py` to ALLOWLIST_FILES with a comment explaining the backward-compat reason.
**Warning signs:** `pytest tests/brain/test_signal_nomenclature.py -x` still fails after code change.

### Pitfall 3: Cluster key rename breaking other consumers
**What goes wrong:** If anything besides brain_composite_engine.py reads `"cluster_events"`, renaming to `"clusters"` would break it.
**Why it happens:** Implicit data contracts between modules.
**How to avoid:** Grep for `"cluster_events"` across entire codebase. Verified: only signal_details.py line 130 writes it; only brain_composite_engine.py lines 245 and 434 read the corresponding `"clusters"` key.
**Warning signs:** Test failures in modules other than brain composites.

### Pitfall 4: Not verifying the fix end-to-end
**What goes wrong:** Code change is correct but the runtime behavior doesn't change because of caching or stale state.
**Why it happens:** Brain.duckdb is the runtime cache. Code changes alone don't update it.
**How to avoid:** After Gap 1 fix, query brain.duckdb to verify `threshold_type = 'tiered'` for STOCK.TRADE.volume_patterns. After Gap 3 fix, run a pipeline and verify composite receives non-empty cluster data.
**Warning signs:** `brain build` reports success but DB query still shows old values.
