# Phase 119.1: Narrative Quality Overhaul - Research

**Researched:** 2026-03-20
**Domain:** Boilerplate elimination across D&O worksheet narrative layers
**Confidence:** HIGH

## Summary

The worksheet has three distinct layers of boilerplate that must be addressed independently: (1) Python narrative builders in `sect1_findings_neg.py` and `sect1_findings_pos.py` that produce generic D&O commentary, (2) brain YAML `do_context` templates across 562 signals where ~299 contain the exact phrase "historically correlated with increased D&O claim frequency and severity" (batch-generated in Phase 116-01), and (3) hardcoded D&O implications in `_DO_IMPLICATIONS_MAP` in `narrative.py` that use identical text regardless of company profile.

The critical insight is that the boilerplate lives at three levels -- Python functions, YAML templates, and Python constants -- but only the Python functions and the `_DO_IMPLICATIONS_MAP` are directly actionable in this phase. The 299 YAML do_context templates are consumed by the do_context engine and rendered into signal results at ANALYZE time. Fixing them requires either regenerating the YAML templates or overriding them at render time. The Python narrative builders are the most impactful target because they produce the Executive Brief key findings that the user sees first.

**Primary recommendation:** Eliminate `neg_generic()` entirely, make all `neg_*` and `pos_*` functions pull actual company data and scoring details, replace `_DO_IMPLICATIONS_MAP` static text with state-aware generation, and batch-fix the 299 YAML do_context templates to use signal-specific language instead of the "historically correlated" boilerplate.

## Architecture Patterns

### Current Narrative Architecture (What Exists)

```
AnalysisState
  |
  +-- executive_summary.key_findings.negatives/positives  (KeyFinding objects)
  |     |
  |     +-- evidence_narrative (raw string from scoring/signal engine)
  |     +-- scoring_impact (e.g., "Critical Red Flag: Ceiling 65")
  |     +-- section_origin (where finding came from)
  |
  +-- scoring.factor_scores[]  (per-factor: points_deducted, max_points, signal_contributions[])
  |
  +-- analysis.signal_results{signal_id -> {status, value, evidence, do_context, ...}}
  |
  +-- extracted.* (financials, governance, litigation -- all real data)
  |
  +-- company.* (market_cap, employees, subsidiary_count)
```

### Narrative Generation Flow

```
KeyFinding.evidence_narrative  --->  sect1_findings.py (dispatcher)
                                         |
                                         +-- pattern match on raw text
                                         |
                                         +-- dispatch to neg_*/pos_* function
                                         |
                                         +-- function pulls from AnalysisState
                                         |
                                         +-- returns list[str] sentences
                                         |
                                         +-- joined into paragraph
```

### Pattern: What Makes Narrative Company-Specific

Good (already in the codebase):
```python
# neg_prior_litigation already does this well:
f"{name} has {subs} subsidiaries across {jurs} jurisdictions"
f"Goodwill at {gw:.1f}% of total assets reflects..."
f"The current docket includes {lit['derivative']} derivative suits"
```

Bad (the boilerplate):
```python
# neg_generic -- could apply to ANY company:
f"At its scale ({', '.join(ctx_parts)}), even moderate risk factors can generate material D&O exposure."

# neg_stock_risk -- ANGI 73.7% decline = "notable decline":
f"{name}'s stock has experienced a notable decline."

# pos_stable_leadership -- identical for 15yr CEO and 2yr CEO:
"Leadership stability reduces the 'rats leaving a sinking ship' narrative..."
```

### Pattern: State-Aware D&O Implications

The `_DO_IMPLICATIONS_MAP` is static text. It should be replaced with functions that check state:

```python
# BAD (current):
("distress_indicators", "HIGH", "Financial distress indicators correlate with increased securities fraud allegation risk")

# GOOD (needed):
# Function checks actual Altman Z, D/E, going concern, then generates:
# "With Altman Z-Score of 0.98 (distress zone) and D/E of 3.2x, ANGI faces acute Zone of Insolvency
#  fiduciary duty expansion and creditor derivative suit exposure. Going concern risk at this level
#  correlates with 4x baseline SCA filing frequency."
```

### Anti-Patterns to Avoid

- **Generic D&O primer sentences**: "Audit-related issues are among the strongest predictors of securities litigation" -- this is a textbook statement, not analysis. Replace with: "ANGI's Beneish M-Score of -2.1 with 3/8 component flags triggered (DSRI 1.42, SGAI 1.31, TATA 0.09) indicates earnings manipulation probability -- this pattern precedes 67% of SEC enforcement actions in the tech sector."
- **Scale-only differentiation**: "At $3.2T market cap, even moderate..." -- inserting numbers into a template is not the same as company-specific analysis. The *interpretation* must change, not just the numbers.
- **Same narrative for opposite profiles**: The `pos_stable_leadership` function gives identical text for a 15-year CEO at a healthy company and a 2-year CEO at a distressed company. The tenure duration, company health context, and departure history should all shape the narrative.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Generating 299 unique YAML do_context templates | Manual editing per signal | Batch script with LLM-assisted generation (same pattern as Phase 116-01, but with better prompts) | 299 signals with the same boilerplate phrase; hand-editing is infeasible |
| Company-specific D&O implications | Static text map | State-aware functions that read scoring, extracted data, and signals | Static maps cannot differentiate between AAPL and ANGI |
| Severity-calibrated language | Hardcoded adjective thresholds | Calibration table mapping metric ranges to descriptor words | "notable decline" vs "catastrophic collapse" should be driven by thresholds, not if/else chains |

## Common Pitfalls

### Pitfall 1: Fixing Python But Not YAML
**What goes wrong:** The Executive Brief findings get fixed but the D&O Risk columns in every table still show "historically correlated with increased D&O claim frequency" from brain YAML do_context.
**Why it happens:** The 299 YAML templates are in a different layer (brain/) than the Python builders (render/).
**How to avoid:** Phase must address both layers. YAML templates need batch regeneration with signal-specific language.
**Warning signs:** Running the pipeline and checking any D&O Risk column in a table -- if it says "historically correlated," the YAML layer is unfixed.

### Pitfall 2: Template Variable Limitations
**What goes wrong:** Attempting to make YAML do_context templates highly company-specific, but the template engine only has access to `{company}`, `{value}`, `{zone}`, `{threshold}`, `{evidence}`, `{source}`, `{confidence}`, `{ticker}`, and `{details_*}`.
**Why it happens:** The `do_context_engine.py` `SafeFormatDict` only provides signal-level variables, not full state data.
**How to avoid:** For YAML templates, focus on making the *signal-specific* language better (explain WHAT the metric means for D&O, not just "historically correlated"). Company-specific data is already in `{value}` and `{evidence}`. The generic conclusion sentence is the problem, not the data insertion.
**Warning signs:** Templates referencing variables that don't exist (they'll silently render as empty string).

### Pitfall 3: Breaking Existing Tests
**What goes wrong:** Changing narrative text breaks string-match assertions in tests.
**Why it happens:** Tests may assert on exact narrative content.
**How to avoid:** Check: no tests currently assert on `neg_generic` or specific narrative text (confirmed by grep -- zero test files reference these functions). The `_DO_IMPLICATIONS_MAP` has tests in `test_narrative_context.py` and `test_5layer_narrative.py` that may need updating.

### Pitfall 4: Losing the Dispatch Pattern
**What goes wrong:** Attempting to replace the keyword-based dispatch in `sect1_findings.py` with something more complex.
**Why it happens:** The dispatch pattern (match "Volatility" in raw -> call `neg_stock_risk`) is a core architectural choice.
**How to avoid:** Keep the dispatch pattern. Fix the *called functions*, not the dispatcher. The dispatcher routes correctly; the functions produce bad output.

### Pitfall 5: YAML Changes Without Evidence
**What goes wrong:** Rewriting 299 YAML do_context templates without underwriting justification violates the NON-NEGOTIABLE rule: "Brain changes driven by real evidence, not YAML editing."
**Why it happens:** The original batch generation (Phase 116-01) was expedient but produced low-quality boilerplate.
**How to avoid:** Frame YAML changes as "correcting batch-generation defects" not "new brain knowledge." The signal definitions themselves are not changing -- only the presentation layer text that was auto-generated with insufficient quality. Document the evidence: "Phase 116-01 batch generation used a single template pattern that produces identical text for all 299 signals, violating QUAL-01."

## Code Examples

### Example 1: Replacing neg_generic with finding-aware generation

```python
# BEFORE (neg_generic):
def neg_generic(state: AnalysisState, name: str, raw: str) -> list[str]:
    mc = market_cap_billions(state)
    empl = employee_count(state)
    subs = subsidiary_count(state)
    clean = raw.split(":")[0].strip() if ":" in raw else raw
    sentences = [f"{name} faces risk from {clean.lower()}."]
    ctx_parts = []
    if mc: ctx_parts.append(f"${mc:.1f}B market cap")
    ...
    return sentences

# AFTER: Find the best matching neg_* function or build from signal data
def neg_from_signal_data(
    state: AnalysisState, name: str, finding: KeyFinding,
) -> list[str]:
    """Build narrative from the finding's actual signal data."""
    # Extract the signal ID and factor from the finding's origin
    signal_id = _extract_signal_id(finding)
    factor_id = _extract_factor_id(finding)

    sentences = []
    if signal_id and state.analysis and state.analysis.signal_results:
        sig = state.analysis.signal_results.get(signal_id, {})
        value = sig.get("value", "")
        evidence = sig.get("evidence", "")
        threshold_ctx = sig.get("threshold_context", "")

        # Build from actual signal data
        sentences.append(
            f"{name}'s {finding.evidence_narrative.split(':')[0]} "
            f"measured at {value} ({threshold_ctx})."
        )
        if evidence:
            sentences.append(evidence)

    # Add factor-specific D&O theory
    if factor_id:
        theory = _FACTOR_DO_THEORY.get(factor_id, "")
        if theory:
            sentences.append(theory.format(name=name))

    return sentences or [f"{finding.evidence_narrative}."]
```

### Example 2: Severity-calibrated stock language

```python
# BEFORE:
sentences.append(f"{name}'s stock has experienced a notable decline.")

# AFTER: Calibrate language to actual magnitude
def _stock_decline_descriptor(pct: float) -> str:
    """Map decline percentage to severity-appropriate language."""
    if pct >= 70: return "catastrophic collapse"
    if pct >= 50: return "severe decline"
    if pct >= 30: return "substantial decline"
    if pct >= 15: return "significant decline"
    return "moderate pullback"

# Then in neg_stock_risk:
decline_pct = _extract_decline_pct(state)  # from stock analysis data
if decline_pct:
    desc = _stock_decline_descriptor(decline_pct)
    sentences.append(
        f"{name}'s stock has undergone a {desc} of {decline_pct:.1f}% "
        f"from ${high:.2f} to ${low:.2f} over {days} trading days."
    )
```

### Example 3: State-aware D&O implications replacing static map

```python
# BEFORE (static text in _DO_IMPLICATIONS_MAP):
("distress_indicators", "HIGH",
 "Financial distress indicators correlate with increased securities fraud allegation risk")

# AFTER: Function that generates from state
def _distress_implication(state: AnalysisState) -> str | None:
    z = safe_distress(state, "altman_z_score")
    de = safe_leverage_ratio(state, "debt_to_equity")
    gc = going_concern(state)
    if z is None and de is None:
        return None
    parts = []
    if z is not None:
        if z < 1.1:
            parts.append(f"Altman Z-Score {z:.2f} places {name} in distress zone")
        elif z < 2.6:
            parts.append(f"Altman Z-Score {z:.2f} is in the gray zone")
    if de is not None and de > 2.0:
        parts.append(f"leverage at {de:.1f}x D/E")
    if gc:
        parts.append("going concern qualification active")
    if not parts:
        return None
    return (
        f"With {', '.join(parts)}, D&O exposure shifts from standard "
        f"securities fraud theories to creditor-oriented claims including "
        f"Zone of Insolvency fiduciary duty expansion."
    )
```
