    **Step 2: Build callout aggregation in chart_thresholds.py.**

    Add function `evaluate_chart_callouts(state: AnalysisState, metrics: dict[str, float | None], thresholds: dict[str, dict[str, float]]) -> dict[str, list[str]]`:

    1. Load signals via BrainLoader (same as extract_chart_thresholds uses)
    2. For each signal with `display.callout_templates`:
       - Map signal to its metric key (same mapping as thresholds)
       - Get the metric value from `metrics` dict
       - If value is None, skip
       - Compare value against thresholds for that metric
       - Determine severity (red, yellow, clear, or "positive" for green signals)
       - Select the appropriate callout_template string
       - Interpolate `{value}` with formatted metric value (format based on metric type: percentage, ratio, count)
       - Also interpolate `{threshold}` if present (using the threshold value that was breached)
       - Append to `flags` (for red/yellow) or `positives` (for clear/positive)
    3. Return `{"flags": flags, "positives": positives}`

    Include a `_FALLBACK_CALLOUTS` dict that replicates the current hardcoded callout text, so if signal loading fails, the function still returns meaningful callouts.

    Metric-to-signal mapping and evaluation logic:
    - `beta_ratio`: if > thresholds.beta_ratio.yellow -> yellow callout. Format: `{value}` = `f"{val:.2f}"`
    - `vol_ratio`: if vol_90d > sector_vol * thresholds.vol_ratio.yellow -> yellow callout. Format: `{value}` = `f"{vol_pct:.1f}"` where vol_pct is vol_90d as percentage
    - `max_drawdown`: if max_dd < thresholds.max_drawdown.red -> red callout. Format: `{value}` = `f"{val:.1f}"`
    - `decline_from_high`: if decline < -thresholds.decline_current.red -> red callout; if > -thresholds.decline_near_high.yellow -> positive callout. Format: `{value}` = `f"{abs(val):.0f}"`
    - `company_specific_drops`: if count >= thresholds.company_specific_drops.yellow -> yellow callout. Format: `{value}` = `f"{val:.0f}"`
    - `idiosyncratic_vol`: if > thresholds.idiosyncratic_vol.red -> red callout. Format: `{value}` = `f"{val:.0f}"`
    - `alpha_1y`: if > thresholds.alpha.yellow -> positive callout. Format: `{value}` = `f"{val:.1f}"`
    - `vol_at_sector`: if vol_90d <= sector_vol -> positive callout (template from STOCK.PRICE.technical clear)
    - `total_drops_zero`: if total_drop_count == 0 -> positive callout (template from STOCK.PRICE.unexplained_drops clear)

    **Step 2: Create CI lint test for numeric threshold drift.**

    Create `tests/test_threshold_lint.py`:

    ```python
    """CI lint: ensure chart/market templates do not contain hardcoded thresholds.

    Scans Jinja2 conditional expressions ({% if ... %}) for numeric literal
    comparisons, which indicate thresholds have drifted back into templates
    instead of being declared in signal YAML.

    Allowed patterns:
    - Comparisons to 0 (zero checks: == 0, > 0, != 0)
    - Comparisons to 1 (boolean-like: > 1, >= 1)
    - Format strings ('{:.1f}', '{:+.1f}')
    - Non-conditional contexts (text content, CSS classes)
    """
    ```

    Implementation:
    - Read `stock_charts.html.j2` and `stock_performance.html.j2`
    - Use regex to find `{% if ... %}` and `{% elif ... %}` blocks
    - Within those blocks, find numeric comparisons: pattern like `(>|<|>=|<=|==|!=)\s*-?\d+\.?\d+` where the number is NOT 0, 1, or -1
    - Also check for `* \d+\.` patterns (multiplication by threshold, like `sector_vol * 1.3`)
    - Report any violations with line number and matched expression
    - Test passes if zero violations found

    Also scan `{% set ... = ... %}` expressions that build lists with threshold logic (the `flags.append` pattern). These should NOT exist after the refactor — if found, flag as violation.

    Exemptions (explicit allowlist):
    - `vol_90d * 100 if vol_90d < 1` — unit conversion, not threshold
    - `'{:.1f}'.format(...)` — format strings
    - `unexplained > 0` — zero check
    - `figure_num` comparisons
    - Lines that reference `thresholds.` — these are the approved pattern

    The test should be parametrized to scan BOTH templates and can be extended to scan additional templates later.
