  <files>src/do_uw/brain/brain_build_signals.py, src/do_uw/cli_brain.py, src/do_uw/cli_brain_ext.py, src/do_uw/cli_knowledge_traceability.py, src/do_uw/brain/brain_audit.py</files>
  <action>
    **Part A: Simplify brain build (brain_build_signals.py)**

    Current `brain build` does: YAML → parse → enrich → INSERT INTO brain_signals → rebuild framework tables → export signals.json.

    New behavior:
    1. Load all signals via `load_signals()` from brain_unified_loader (already validates + enriches)
    2. Run schema validation: for each signal, validate via `BrainSignalEntry.model_validate()` — report all errors, don't stop at first
    3. Run cross-reference integrity checks: all referenced factor IDs exist in taxonomy, all peril_ids exist in framework, signal IDs are unique
    4. Run coverage matrix: check that all prefix families have signals
    5. Export `signals.json` to `brain/config/signals.json` (the new location from Plan 01)
    6. Log summary: "Validated N signals, M errors, exported signals.json"
    7. DO NOT insert into brain_signals DuckDB table
    8. DO NOT call build_framework() (no more DuckDB framework table population)

    Keep brain_changelog logging (at Claude's discretion per CONTEXT.md — recommend keeping it since it's valuable history). Write changelog entries to DuckDB for any changes detected between current YAML and last export.

    The `build_checks_from_yaml()` function is the main entry point called by `cli_brain.py`. Rewrite it to do validate + export instead of truncate + insert. Keep the function name for CLI compat.

    Remove or deprecate `build_framework()` which populates brain_perils and brain_causal_chains DuckDB tables (perils/chains now read from YAML directly).

    **Part B: Update brain CLI commands**

    In `cli_brain.py`:

    - `brain status`: Currently queries brain_signals count and brain_taxonomy from DuckDB. Change to:
      - Signal count: `len(load_signals()["signals"])` from brain_unified_loader
      - Taxonomy: `load_taxonomy()` from brain_unified_loader
      - Keep DuckDB history table counts (brain_signal_runs, brain_effectiveness) as-is

    - `brain gaps`: Currently uses BrainDBLoader. Switch to BrainLoader.

    In `cli_brain_ext.py`:

    - `brain export-docs`: Currently reads from brain_signals_active DuckDB view. Switch to `load_signals()`.
    - `brain export-all`: Currently uses BrainDBLoader (all methods). Switch to BrainLoader.
    - `brain backlog`: Currently uses BrainDBLoader.load_backlog(). Switch to BrainLoader (which reads from DuckDB brain_backlog — history table).
    - `brain stats`: Signal count from YAML, history table counts from DuckDB.
    - `brain import-json`: This imported JSON→DuckDB definition tables. Now that definitions come from YAML, this command is obsolete for definitions. Keep it functional for importing into brain_backlog or brain_proposals (if those use it), or mark as deprecated with a warning message.

    In `cli_knowledge_traceability.py`:
    - Uses BrainKnowledgeLoader (already deleted in Plan 02). Switch to BrainLoader.

    In `brain_audit.py`:
    - Already reads YAML directly for signal definitions (confirmed in research). BUT it may reference `brain_coverage_matrix` DuckDB view which reads from definition tables. If so, compute coverage from YAML signals directly instead of the DuckDB view.
