## Summary

Phase 50.1 closes the FACET-03 gap identified in Phase 49 verification: `_PREFIX_DISPLAY` (a hardcoded dict mapping signal ID prefixes to section display names) still controls how signals are grouped into sections in the HTML output. Facet definitions were added as parallel metadata but are not authoritative for section assignment.

The migration is well-scoped and low-risk because the current prefix-to-facet mapping is already 1:1 (each prefix maps to exactly one facet). No signal belongs to a facet that contradicts its prefix. The FacetSpec model already carries `id`, `name`, `display_type`, and `signals` -- everything needed to replace `_PREFIX_DISPLAY`. The work is: (1) make `_group_signals_by_section()` read facet definitions to determine section assignment, (2) update `_compute_coverage_stats()` similarly, (3) maintain backward-compatible prefix-based keys in `signal_results_by_section` since 6 Jinja2 templates access signals by prefix key (e.g., `.get('GOV', [])`), (4) update `cli_brain_trace.py` to stop importing `_PREFIX_DISPLAY`, (5) clean up stale "checks" terminology.

**Primary recommendation:** Replace `_PREFIX_DISPLAY` with facet-driven section assignment in `html_signals.py`, keeping prefix-based dict keys for template compatibility (templates use `signal_results_by_section.get('BIZ', [])` etc.). Use a derived `_facet_to_prefix()` lookup to maintain the prefix-key contract while making facets the source of truth for section names and grouping.

### Key Design Decision: Prefix Keys Must Be Preserved

The 6 section templates directly access `signal_results_by_section` using prefix keys:
- `signal_results_by_section.get('BIZ', [])`
- `signal_results_by_section.get('FIN', [])`
- `signal_results_by_section.get('GOV', [])` + `.get('EXEC', [])`
- `signal_results_by_section.get('LIT', [])`
- `signal_results_by_section.get('STOCK', [])`
- `signal_results_by_section.get('NLP', [])` + `.get('FWRD', [])`

Changing these keys would require modifying 6+ Jinja2 templates AND re-testing HTML output visually. That is scope creep for this phase. The correct approach: **keep prefix-based keys but derive them FROM facets** instead of from `_PREFIX_DISPLAY`.

### Pattern: Facet-Derived Prefix Map

Build the prefix-to-section mapping dynamically from facet definitions:

```python
def _build_facet_section_map() -> dict[str, str]:
    """Build prefix -> display_name from facet definitions (authoritative)."""
    facets = _get_facets()
    prefix_map: dict[str, str] = {}
    for facet_id, facet in facets.items():
        if not facet.signals:
            continue  # Skip red_flags (no signals)
        # Infer prefix from first signal
        first_prefix = facet.signals[0].split(".")[0]
        # All signals in a facet share the same prefix (validated by CI)
        prefix_map[first_prefix] = facet.name
    return prefix_map
```

This replaces the hardcoded `_PREFIX_DISPLAY` with a dynamic derivation from YAML. The `_PREFIX_TO_FACET` bridge dict can also be derived similarly.

### Pattern: Signal-to-Facet Direct Lookup for Grouping

Instead of grouping by `signal_id.split(".")[0]` (prefix extraction), group by facet membership:

```python
def _group_signals_by_section(signal_results, filing_date_lookup=None):
    facets = _get_facets()
    # Build signal_id -> prefix mapping from facet declarations
    signal_to_prefix: dict[str, str] = {}
    for facet_id, facet in facets.items():
        if not facet.signals:
            continue
        prefix = facet.signals[0].split(".")[0]
        for sig_id in facet.signals:
            signal_to_prefix[sig_id] = prefix

    grouped: dict[str, list[dict]] = {}
    for signal_id, result_data in signal_results.items():
        # Primary: facet-driven lookup
        prefix = signal_to_prefix.get(signal_id)
        if prefix is None:
            # Fallback: infer from signal ID prefix (undeclared signals)
            prefix = signal_id.split(".")[0] if "." in signal_id else "OTHER"
        # ... rest of grouping logic
```

This makes facets authoritative while gracefully handling signals not declared in any facet (fallback to prefix inference).

### Anti-Patterns to Avoid

- **Changing template dict keys:** The section templates use hardcoded `.get('BIZ', [])` keys. Do NOT change these to facet IDs (`business_profile`) -- that is scope creep requiring 6+ template modifications.
- **Removing prefix fallback:** Some signals may not be in any facet (future additions, edge cases). Always fall back to prefix extraction.
- **Name mismatches breaking coverage table:** `_compute_coverage_stats` currently uses `_PREFIX_DISPLAY` names. 4 of 8 names differ between `_PREFIX_DISPLAY` and facet `name`:

| Prefix | `_PREFIX_DISPLAY` (current) | Facet `name` (target) |
|--------|-----------------------------|-----------------------|
| STOCK | "Market & Trading" | "Market Activity" |
| GOV | "Governance" | "Governance Assessment" |
| NLP | "NLP / Filing Analysis" | "Filing Analysis" |
| FWRD | "Forward-Looking" | "Forward-Looking Indicators" |

These name changes will appear in the coverage appendix. This is EXPECTED and CORRECT -- facet names are the authoritative display names. The `qa_audit.html.j2` template applies `{{ section_name | replace('_', ' ') | title }}` to the prefix key, which currently renders "Biz", "Fin", etc. After migration the display remains prefix-based in that template since the dict key stays as prefix.

## Common Pitfalls

### Pitfall 1: Template Key Breakage
**What goes wrong:** Changing `signal_results_by_section` dict keys from prefixes to facet IDs breaks 6 section templates that use `.get('BIZ', [])`.
**Why it happens:** Natural instinct to use facet_id as the canonical key.
**How to avoid:** Keep prefix as dict key. The facet is authoritative for *which signals go where* and *section display name*, but the dict key stays as prefix for template compatibility.
**Warning signs:** Any template showing empty check sections after the migration.

### Pitfall 2: Name Changes in Coverage Table
**What goes wrong:** Coverage appendix shows "Market Activity" instead of "Market & Trading" -- user thinks something broke.
**Why it happens:** `_compute_coverage_stats` switches from `_PREFIX_DISPLAY` names to facet names.
**How to avoid:** This is correct behavior. Document it as expected. Visually verify the coverage table in output HTML.
**Warning signs:** None -- this is desired. Just needs visual confirmation.

### Pitfall 3: cli_brain_trace.py Import Breakage
**What goes wrong:** `cli_brain_trace.py` imports `_PREFIX_DISPLAY` from `html_signals.py` at lines 245 and 313. If `_PREFIX_DISPLAY` is removed, these imports fail.
**Why it happens:** The trace command uses `_PREFIX_DISPLAY` to show "section_name" in the render stage display.
**How to avoid:** Migrate `cli_brain_trace.py` to use facet definitions directly (it already loads facets at line 441-448 for the trace header). Replace `_PREFIX_DISPLAY.get(prefix, prefix)` with facet name lookup.
**Warning signs:** `do-uw brain trace <ID>` crashing after migration.

### Pitfall 4: red_flags Facet Edge Case
**What goes wrong:** The `red_flags` facet has an empty `signals` list. Any logic that iterates facets and derives prefixes from the first signal will crash.
**Why it happens:** red_flags is a special display-only facet sourced from scoring results, not signal evaluations.
**How to avoid:** Skip facets with empty `signals` lists (already done in `render-audit` command at line 544-549 of cli_brain_trace.py).
**Warning signs:** KeyError or IndexError on red_flags facet.

### Pitfall 5: md_renderer.py Parallel Structure
**What goes wrong:** `md_renderer.py` has its own `_PREFIX_DISPLAY_NAMES` dict (lines 282-291) that is NOT part of this migration scope but will diverge.
**Why it happens:** Markdown renderer has parallel prefix mapping logic.
**How to avoid:** Out of scope for this phase. Note as tech debt for future alignment but do NOT change md_renderer.py in this phase. Keep scope tight.
**Warning signs:** None for this phase -- md_renderer already works independently.

## Open Questions

1. **Should `_PREFIX_DISPLAY` be deleted or replaced with a deprecation comment?**
   - Recommendation: Delete it entirely. The facet-derived map replaces it. No external consumers (only internal to html_signals.py + cli_brain_trace.py).
   - Confidence: HIGH

2. **Should facet section order be deterministic?**
   - Currently, `_group_signals_by_section` returns a plain dict. Section order in qa_audit.html.j2 depends on dict insertion order (Python 3.7+ preserves insertion order).
   - The facets are loaded via `sorted(facets_dir.glob("*.yaml"))` -- alphabetical by filename. The signal_results iteration order determines grouped dict order.
   - Recommendation: Not a concern for this phase. Current behavior is acceptable.
   - Confidence: HIGH

3. **Should `__all__` export `_PREFIX_TO_FACET` after migration?**
   - It is currently exported. If no external consumers remain after cli_brain_trace.py migration, remove from `__all__`.
   - Recommendation: Remove from `__all__` and delete the dict if it becomes unused.
   - Confidence: HIGH
