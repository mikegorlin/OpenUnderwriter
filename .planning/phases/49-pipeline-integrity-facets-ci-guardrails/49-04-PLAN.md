Trace command design (from CONTEXT.md decisions):
- Default mode: shows results from last completed analysis run (reads state.json/output)
- Blueprint mode (--blueprint): shows theoretical route from YAML only
- Step-by-step vertical flow with stage markers (checkmark/cross)
- Shows actual data values + status at each stage
- Errors if no run exists and --blueprint not specified

Trace data sources per stage:
| Stage | Source | Data |
|-------|--------|------|
| YAML Definition | brain/signals/**/*.yaml | id, name, threshold, required_data, field_key, facet, display |
| Extraction | state.json ExtractionManifest | extractor, source filing, raw value |
| Mapping | signal_field_routing + data_strategy | mapper function, field_key, mapped_value |
| Evaluation | state.signal_results[signal_id] | status, value, threshold_level, evidence, trace_* |
| Rendering | facet definition + html_signals | section, facet, display format |

1. Load all facet definitions -> collect declared signal IDs per facet
2. Load state.json -> collect signal_results where status != SKIPPED
3. Report: per-facet declared vs populated, overall coverage %

Add a `trace` command to the brain CLI app. Two modes: default (live) and --blueprint.

**Command signature:**
```python
@brain_app.command("trace")
def brain_trace(
    signal_id: str = typer.Argument(help="Signal ID to trace (e.g., GOV.BOARD.independence)"),
    blueprint: bool = typer.Option(False, "--blueprint", help="Show theoretical route without requiring a run"),
    ticker: str = typer.Option("", "--ticker", help="Ticker for live mode (defaults to most recent)"),
) -> None:
```

**Implementation - Common (both modes):**
1. Load the signal YAML entry by ID:
   - Use brain loader to find the signal by ID across all YAML files
   - If signal not found, print error and exit(1): `Signal '{signal_id}' not found in brain/signals/`
2. Load the signal's facet definition (from brain/facets/*.yaml)
3. Print header:
   ```
   Signal: {id} -- {name}
   Facet:  {facet_id} ({facet_name})
   ```

**Implementation - YAML Definition stage (both modes):**
Print:
```
--- YAML Definition ---
  work_type:    {work_type}
  threshold:    {threshold.type} ({threshold.red} / {threshold.yellow} / {threshold.green or "clear"})
  required_data: {data_strategy.required_data or data_locations}
  field_key:    {data_strategy.field_key}
```

**Implementation - Blueprint mode (`--blueprint`):**
For stages 2-5, show theoretical routes:
- Extraction Route: infer extractor from data_strategy.required_data (e.g., SEC_DEF14A -> proxy_extractor)
- Mapping Route: show data_strategy.field_key and which mapper handles it (from signal_field_routing)
- Evaluation Route: show threshold type and levels
- Rendering Route: show facet and display spec

Print each stage with "(theoretical)" label. No state.json needed.

**Implementation - Live mode (default):**
1. Find the most recent run output:
   - If --ticker specified, look in `output/{TICKER}/` for most recent date directory
   - If not specified, find most recent output directory across all tickers
   - If no output found, print error: `No analysis run found. Use --blueprint for theoretical route, or run 'do-uw analyze <TICKER>' first.`
   - Load the state from the output (state.json or serialized AnalysisState)

2. Extraction stage: from state, find what was extracted for this signal's data source
   - Look at ExtractionManifest or the acquired data to find the extractor and source filing
   - Show raw extracted value if available

3. Mapping stage: show the mapper function and mapped value from signal_results trace fields
   - SignalResult has trace_mapper, trace_field_key, trace_mapped_value (or similar)
   - If trace fields not available, show "N/A" with note

4. Evaluation stage: from state.analysis.signal_results[signal_id]
   - Show status (TRIGGERED/CLEAR/SKIPPED), value, threshold_level, evidence
   - Use checkmark/cross markers: CLEAR -> checkmark, TRIGGERED -> warning, SKIPPED -> cross

5. Rendering stage: show facet and display spec, section assignment

**Output formatting:**
Use Rich console for colored output:
- Stage headers in bold
- Status markers: green checkmark for OK, yellow warning for TRIGGERED, red cross for SKIPPED/missing
- Values aligned with consistent indentation

**Error cases:**
- Invalid signal ID -> clear error message
- No run found (live mode without --blueprint) -> suggest using --blueprint
- Signal found in YAML but not in run results -> show "Not evaluated in this run" with reason

```python
@brain_app.command("render-audit")
def brain_render_audit(

) -> None:
```

**Implementation:**
1. Find the most recent run output for the ticker (same logic as trace command)
2. Load all facet definitions using `load_all_facets()` from brain_facet_schema.py
3. Load signal_results from the run's state
4. For each facet (excluding red_flags which is dynamic):

   - Missing signals: declared but SKIPPED or not in results
5. Print per-facet summary:
   ```
   Facet: {facet_name} ({declared_count} declared signals)
     Rendered:  {rendered_count} of {declared_count}
     Missing:   {list of missing signal IDs with reason (SKIPPED, not evaluated, etc.)}
   ```
6. Print overall summary:
   ```
   Overall: {total_rendered} / {total_declared} active signals rendered ({pct}%)
            {total_skipped} SKIPPED (not rendered)
            {total_inactive} INACTIVE (excluded from audit)
   ```
