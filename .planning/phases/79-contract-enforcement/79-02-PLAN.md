  <behavior>
    - ManifestFacet accepts optional requires: list[str] field (default empty list)
    - validate_requires_populated(manifest, context_dict) returns list of warnings (not violations -- missing data is a warning, not a contract break)
    - Each warning identifies facet_id, missing field path, and what data is expected
    - Facets with empty requires list produce no warnings
    - Dot-notation paths resolved against nested dict (e.g., "financials.statements" checks context["financials"]["statements"])
    - None, empty list, and missing key all count as "not populated"
    - Existing manifest YAML without requires field still loads (backwards compatible)
  </behavior>
  <action>
    1. Update ManifestFacet in manifest_schema.py:
       - Add `requires: list[str] = Field(default_factory=list, description="Data field paths required for this facet to render meaningfully (dot-notation, e.g. 'financials.statements')")`

    2. Add to contract_validator.py:
       - DataWarning(BaseModel): facet_id, section_id, field_path, detail
       - validate_requires_populated(manifest: OutputManifest, context: dict[str, Any]) -> list[DataWarning]:
         - For each section/facet with non-empty requires, resolve each field path against context
         - Resolution: split on '.', traverse dict keys, check final value is not None/empty
         - Return DataWarning for each field that is missing/empty
       - This is a WARNING system, not a hard failure -- the pipeline should still render, just log what is missing

    3. Write tests:
       - Facet with requires=["financials.statements"] + context has it -> no warnings
       - Facet with requires=["financials.statements"] + context missing it -> warning with field path
       - Facet with requires=[] -> no warnings (backwards compatible)
       - Nested path resolution: "a.b.c" checks context["a"]["b"]["c"]
       - None value at path -> warning
       - Empty list at path -> warning
  </action>
