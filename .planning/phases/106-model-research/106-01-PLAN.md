Create a comprehensive YAML design document specifying the multiplicative scoring model. This is a DESIGN ARTIFACT, not code. Structure the document with these sections:

**Section 1: H/A/E Composite Computation**
- For each of the 3 categories (Host, Agent, Environment), specify how subcategory scores aggregate into a composite
- Each subcategory score = f(signal evaluations within that subcategory)
- Signal evaluation outcomes (RED/YELLOW/CLEAR/SKIPPED) map to numeric values
- Subcategory aggregation method: weighted average of signal scores within the subcategory
- Define subcategory weights within each category (e.g., host.financials may weigh more than host.data_foundation)
- Cite rationale for weights: reference SCAC filing data (securities claims correlate most with financial misrepresentation = agent.financial_conduct and agent.disclosure dominate), Cornerstone settlement data, and D&O underwriting practice
- Composite score normalized to 0-1 range

**Section 2: Weighting Rationale**
- Each weight must cite its basis:
  - SCAC data: ~65% of SCAs involve financial misstatement (agent.financial_conduct, agent.accounting_events)
  - Cornerstone Research: market cap and stock drop are top 2 settlement predictors (host.identity, environment.market_conditions)
  - NERA: insider trading present in ~15% of SCAs but dramatically increases settlement (agent.insider_conduct)
  - ISS governance data: board independence and audit committee correlate with fewer claims (host.governance_structure)
  - Beneish/Altman: forensic accounting scores predict restatement risk (agent.accounting_events)
  - D&O underwriting practice for weights without published research
- Include a weight_source field for each subcategory weight

**Section 3: Multiplicative Interaction Model**
- P = H_composite x A_composite x E_composite
- Define normalization: raw composites in [0,1], product also in [0,1]
- Explain why multiplicative beats additive: interaction effects (high H + high A = much worse than sum suggests; low A with high H = still manageable)
- Define the specific formula with any scaling/calibration constants
- Document how the model handles the "all low" case (P near 0) vs "one very high" case
- Include calibration approach: shadow mode comparing additive (legacy) vs multiplicative (new) across test tickers

**Section 4: CRF Discordance Formalization**
- Reference ELECTRE III multi-criteria decision analysis framework
- Define concordance index: composite score agreement with tier assignment
- Define discordance threshold: CRF presence creates veto regardless of concordance
- List CRF categories and their veto targets:
  - Active SEC enforcement action -> PROHIBITED
  - Announced restatement of material magnitude -> minimum ELEVATED
  - Zone of insolvency (Altman Z < 1.1 or current ratio < 0.5) -> PROHIBITED
  - Active fraud investigation -> PROHIBITED
  - 3+ CRFs of any type -> minimum ADVERSE
- Formalize the override mechanism: CRF discordance overrides multiplicative score
- Reference decision_framework.yaml tier definitions for consistency

**Section 5: Decision Tier Mapping**
- Map composite score ranges to the 6 decision tiers from decision_framework.yaml
- Specify thresholds (conceptual, to be calibrated in Phase 107)
- Document Liberty calibration adjustments:
  - At high attachment (>$50M): increase Host weight, decrease Agent weight
  - For Side A product: increase Agent weight (individual conduct matters more)
  - ABC at low attachment: all dimensions equally weighted
- Include attachment-dependent weight adjustment formulas

**Section 6: Shadow Calibration Specification**
- Define shadow mode: run BOTH legacy additive 10-factor scoring AND new multiplicative H/A/E on same tickers
- Output comparison table: ticker, legacy_score, legacy_tier, hae_score, hae_tier, delta, interpretation
- Calibration criteria: new model should not be systematically more lenient or more harsh than legacy
- Test ticker selection: mix of known good risks, known bad risks, and edge cases
- Specify what "calibrated" means: rank-order correlation > 0.7, no tier disagreements > 1 level on >80% of tickers

Do NOT write any Python code. This is a YAML design document only.
Use real D&O industry data and research citations where available. Where specific empirical data is unavailable, note "calibration_required: true" and explain the basis for the initial estimate. Attribute practitioner knowledge to "D&O underwriting practice" per Phase 103 convention.

Create a comprehensive YAML design document specifying the severity estimation model. This is a DESIGN ARTIFACT, not code. Structure the document with these sections:

**Section 1: Damages Estimation Formula**
- Base damages = market_cap x |class_period_return| x turnover_rate
- turnover_rate approximation from CRSP/TAQ (academic standard) or simplified as shares_traded_in_class_period / shares_outstanding
- Reference Cornerstone Research annual settlement reports for the market cap / settlement relationship
- Include modifier for allegation type:
  - Financial restatement: 1.5x base (stronger scienter evidence)
  - Insider trading: 1.3x base (intent evidence)
  - Regulatory action: 1.2x base (government investigation adds credibility)
  - Guidance miss only: 0.7x base (weakest claim theory)
- Note: these modifiers are from D&O underwriting practice and should be calibrated against actual settlement data in Phase 108

**Section 2: Settlement Regression Features**
- Define the feature vector for settlement prediction based on SCAC/Cornerstone empirical data:
  - market_cap_at_filing (log-transformed) -- strongest predictor per Cornerstone
  - max_stock_decline_pct -- second strongest predictor
  - allegation_type (categorical: financial restatement, insider trading, regulatory, offering, guidance miss)
  - jurisdiction (categorical: SDNY, NDCal, DCt, other federal, state)
  - lead_plaintiff_type (categorical: institutional, individual, government)
  - class_period_length_days -- longer = higher settlement
  - restatement_present (binary) -- strong scienter indicator
  - sec_investigation_present (binary) -- government investigation amplifier
  - number_of_named_defendants -- more defendants = higher settlement
  - company_sector (categorical from GICS sectors)
- For each feature, cite the research source (Cornerstone annual reports, NERA trends, SCAC data)
- Define expected coefficient direction and approximate magnitude
- Note: actual regression coefficients will be fit in Phase 108 implementation if training data is available; initial estimates from published research summaries

**Section 3: Severity Amplifier Catalog**
- Design the complete catalog of severity amplifiers (these will be YAML files validated against the SeverityAmplifier Pydantic schema from Phase 103)
- Each amplifier has: id, name, multiplier (1.0-5.0), trigger_condition, signal_ids, rap_class, epistemology
- Required amplifiers with research basis:
  - media_notoriety: 1.5x -- Cornerstone data shows media-covered cases settle 40-60% higher
  - government_investigation: 2.0x -- SEC/DOJ parallel investigation dramatically increases settlement
  - fraud_type_scienter: 1.8x -- Intentional fraud vs. negligence affects settlement and defense cost
  - jurisdiction_plaintiff_friendly: 1.3x -- SDNY, NDCal, Delaware Chancery have higher settlements
  - restatement_magnitude: 1.5x -- Larger restatement = larger class period damages
  - class_period_length: 1.2x -- Longer class periods = more affected shareholders
  - institutional_lead_plaintiff: 1.4x -- Institutional leads get larger settlements (Cornerstone)
  - multi_defendant: 1.2x -- More named defendants increases total liability exposure
  - prior_securities_litigation: 1.3x -- Recidivist companies face higher settlements
  - whistleblower_involvement: 1.5x -- Whistleblower cases settle higher and faster
- For each, include epistemology with rule_origin and threshold_basis citing real sources

**Section 4: P x S Matrix Structure**
- Define the probability-severity matrix for expected loss visualization
- Probability axis: from multiplicative scoring model (H x A x E composite, 0-1)
- Severity axis: from damages estimation / settlement regression (dollar amount, log-scaled for display)
- Expected loss = P x S
- Matrix zones:
  - GREEN: low P AND low S (standard risk)
  - YELLOW: moderate P OR moderate S (cautious, monitor)
  - ORANGE: high P OR high S (elevated, referral)
  - RED: high P AND high S (adverse/prohibited)
- Define zone boundaries (conceptual, calibrated in Phase 108)
- Specify chart rendering requirements: 2D scatter/heatmap, log-scale severity axis, color-coded zones
- Include specification for how individual accounts plot on the matrix
- Reference chart_registry.yaml integration for Phase 105

**Section 5: Liberty Excess-Only Severity Adjustments**
- Severity as seen by Liberty differs from total claim severity
- At high attachment, Liberty's exposure is only the slice above attachment
- Formula: liberty_severity = max(0, total_estimated_settlement - attachment_point)
- Layer erosion probability: what fraction of claims at this severity level would reach Liberty's layer?
- Reference tower data (from uw_db) for real attachment points
- Side A severity considerations: individual defendant liability vs entity-level settlement
