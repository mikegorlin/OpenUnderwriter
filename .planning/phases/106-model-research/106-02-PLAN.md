Create a comprehensive YAML design document specifying four pattern detection engine algorithms. This is a DESIGN ARTIFACT, not code. Structure the document with these sections:

**Engine 1: Conjunction Scan**
- Purpose: Detect 3+ individually CLEAR/normal signals that, when co-occurring, indicate elevated risk invisible to single-signal evaluation
- Algorithm:
  1. Group evaluated signals by rap_subcategory
  2. For each cross-subcategory combination (e.g., host.financials x agent.disclosure x environment.market_conditions), count signals that are individually CLEAR but have known co-fire correlation from brain_correlations table
  3. When 3+ individually CLEAR signals co-fire AND their combined historical co-fire_rate exceeds threshold (e.g., 0.15), flag as conjunction pattern
  4. Confidence = co_fire_rate x number_of_co_firing_signals / expected_co_fire_count
- Data requirements: signal evaluation results, brain_correlations table, rap_subcategory classification
- Complexity: O(n^2) for pairwise correlation lookup, O(n^3) for 3-way conjunctions; mitigated by only checking cross-domain pairs (host x agent, host x environment, agent x environment)
- Thresholds: minimum 3 signals, co_fire_rate > 0.15, confidence > 0.5
- Output: list of conjunction patterns with participating signals, cross-domain span, confidence score
- Integration: results feed into Phase 110 conjunction signal rules (MECH-01)

**Engine 2: Peer Outlier**
- Purpose: Detect companies that are statistical outliers across multiple financial dimensions relative to sector peers, using SEC XBRL Frames data
- Algorithm:
  1. For each company, extract key XBRL metrics: revenue_growth, profit_margin, debt_ratio, cash_ratio, inventory_turnover, receivable_days, capex_ratio, SGA_ratio
  2. Pull sector peer data from SEC Frames API (same GICS sector, similar market cap tier)
  3. Compute z-scores for each metric relative to peers
  4. A company is a peer outlier if: (a) 3+ metrics have |z| > 2.0 (multi-dimensional outlier), OR (b) any single metric has |z| > 3.0 AND it's in a risk-relevant direction
  5. Risk-relevant direction: higher debt, lower profitability, faster revenue decline, inventory buildup, receivable stretch, lower cash
  6. Confidence = sum(abs(z_i) for outlier metrics) / count(outlier_metrics) -- normalized
- Data requirements: company XBRL data (from ACQUIRE stage), sector peer XBRL data (from SEC Frames), GICS sector classification
- Complexity: O(peers x metrics) for z-score computation -- fast
- Thresholds: |z| > 2.0 for multi-dimensional, |z| > 3.0 for single-metric, minimum 10 peers for statistical validity
- Output: list of outlier dimensions, z-scores, peer comparison context, overall outlier score
- Reference: Beneish M-Score (1999) for precedent of multi-metric fraud detection via statistical outliers

**Engine 3: Migration Drift**
- Purpose: Detect gradual cross-domain deterioration that individual quarterly snapshots miss -- the "boiling frog" pattern
- Algorithm:
  1. For each rap_category (H, A, E), track the subcategory-level scores across 4-8 quarters of XBRL data
  2. Compute per-subcategory trend (linear regression slope over quarters)
  3. A migration drift occurs when: (a) 2+ subcategories in DIFFERENT rap_categories are simultaneously deteriorating (negative slope > threshold), AND (b) the deterioration is gradual (no single-quarter jump > 2x the slope)
  4. Cross-domain requirement: must span at least 2 of {host, agent, environment} -- single-domain drift is just a trend, not a migration
  5. Confidence = magnitude_of_slopes x quarters_observed / 8 (max for 2-year lookback)
- Data requirements: quarterly XBRL data (from ACQUIRE stage) for 4-8 quarters, rap_subcategory classification, signal evaluation history if available
- Complexity: O(subcategories x quarters) for trend computation -- fast
- Thresholds: slope > 0.05 per quarter (5% deterioration/quarter), minimum 4 quarters of data, cross 2+ rap_categories
- Output: list of drifting subcategories with slopes, cross-domain span, quarters observed, overall drift score
- D&O relevance: gradual deterioration across domains is the classic pre-filing pattern -- companies don't fail suddenly, they erode across multiple dimensions simultaneously

**Engine 4: Precedent Match**
- Purpose: Find historical D&O cases with similar signal profiles to the current company, providing precedent-based risk assessment
- Algorithm:
  1. Compute the company's "signal fingerprint": a vector of [signal_id: evaluation_status] for all 514 signals
  2. Encode as binary vector: 1 = RED/YELLOW (fired), 0 = CLEAR/SKIPPED
  3. For each case library entry, compare fingerprints using Jaccard similarity on the fired-signal sets
  4. Weight the similarity by signal importance: CRF signals get 3x weight, ELEVATED-threshold signals get 2x weight
  5. Rank matches by weighted Jaccard similarity
  6. Confidence = similarity_score x case_outcome_severity (cases with known bad outcomes are more informative matches)
- Data requirements: current company signal evaluations, case library entries (from case_library_design.yaml), signal importance weights
- Complexity: O(cases x signals) for similarity computation -- manageable with <1000 cases
- Thresholds: similarity > 0.3 for "notable match", > 0.5 for "strong match", > 0.7 for "very strong match"
- Output: top-5 matching cases with similarity scores, outcome data, key signal overlaps, divergences
- Reference: k-nearest-neighbors (kNN) classification adapted for binary signal vectors

**General Section: Engine Coordination**
- Engines are independent -- each runs separately and produces its own output
- No engine's output is input to another engine (parallel, not sequential)
- Engine firing panel aggregates results: for each engine, show fired/not-fired, confidence level, key finding summary
- Aggregate pattern score: optional weighted sum of engine confidences for overall pattern risk assessment

**General Section: Data Requirements Summary**
- List all data sources each engine needs
- Map to existing ACQUIRE stage outputs
- Identify any new data requirements not currently acquired
