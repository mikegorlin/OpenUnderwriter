# Roadmap: v12.0 Output Quality & Architectural Integrity

## Overview

Eliminate the 5 root causes behind all CUO audit failures through architectural guarantees. Seven phases follow a dependency-driven order: build the CanonicalMetricsRegistry first (Phase 137) because it is the foundation for cross-section consistency, then introduce Typed Context Models (Phase 138) that depend on canonical metrics. Four independent phases -- Contextual Signal Validation (139), Litigation Classification (140), Output Sanitization (141), and Stock Charts (143) -- can execute in any order relative to each other. Quality Gates (Phase 142) come after the registry and typed models are in place, as consistency checking is most effective against canonical values.

## Phase Details

### Phase 137: Canonical Metrics Registry
**Goal**: Every metric that appears in more than one section (revenue, market cap, CEO, exchange, growth rates) is computed exactly once from the highest-confidence source and consumed everywhere
**Depends on**: Nothing (foundation for v12.0)
**Requirements**: METR-01, METR-02, METR-03, METR-04
**Success Criteria** (what must be TRUE):
  1. Running the pipeline for any ticker produces zero cross-section contradictions for revenue, net income, market cap, stock price, employees, exchange, CEO name, and growth rates
  2. Every metric in the registry carries a visible source attribution and as-of date in the worksheet
  3. Context builders import metrics from the registry rather than independently navigating state paths
  4. Changing a metric's source priority (e.g., swapping XBRL for yfinance) requires editing one line in the registry, not hunting through multiple builders
**Plans**: 2 plans
Plans:

- [x] 137-02-PLAN.md — Wire registry into build_html_context and migrate 5 builders

### Phase 138: Typed Context Models
**Goal**: The 5 highest-leakage context builders (executive brief, financial, market, governance, litigation) return Pydantic-validated models instead of raw dicts, preventing untyped data from reaching templates
**Depends on**: Phase 137 (typed models reference canonical metrics from the registry)
**Requirements**: TYPE-01, TYPE-02, TYPE-03, TYPE-04, TYPE-05
**Success Criteria** (what must be TRUE):
  1. Each of the 5 priority context builders has a corresponding Pydantic BaseModel with all fields typed
  2. All fields use Optional defaults -- no pipeline crash when any data is missing
  3. Validation wrapper catches model errors and falls back to the existing untyped dict, so the render pipeline never breaks
  4. Template output is byte-identical before and after typed model introduction (model_dump produces same dict keys)
**Plans**: 2 plans
Plans:
- [x] 138-01-PLAN.md — Define 5 Pydantic context models + validation wrapper with real-state tests
- [ ] 138-02-PLAN.md — Wire _validate_context into md_renderer.py for all 5 builders

### Phase 139: Contextual Signal Validation
**Goal**: Triggered signals are cross-checked against company state so that false positives (IPO signals on 30-year-old companies, insolvency signals on mega-caps with safe Z-Scores) get annotated with explanatory context rather than silently suppressed
**Depends on**: Nothing (independent of Phases 137-138; inserts post-ANALYZE, different pipeline point)
**Requirements**: SIG-01, SIG-02, SIG-03, SIG-04, SIG-05, SIG-06, SIG-07
**Success Criteria** (what must be TRUE):
  1. Running AAPL (public 45+ years) shows zero unannotated IPO/offering signals in the worksheet
  2. Validation rules live in YAML files (not Python if/else), following brain portability principle
  3. Signals are annotated with context (e.g., "Note: Z-Score and O-Score both in safe zone"), never silently suppressed
  4. Negation patterns ("do not have", "no holdings") in signal evidence are detected and the signal is annotated as negated
  5. Signals referencing departed executives carry the departure date annotation
**Plans**: 1 plan
Plans:
- [ ] 139-01-PLAN.md — YAML validation rules + contextual_validator engine + pipeline wiring

### Phase 140: Litigation Classification & Consolidation
**Goal**: Every litigation entry in the worksheet is classified by legal theory (not data source), deduplicated across sources, disambiguated by year, and tagged with D&O coverage side
**Depends on**: Nothing (independent)
**Requirements**: LIT-01, LIT-02, LIT-03, LIT-04, LIT-05
**Success Criteria** (what must be TRUE):
  1. Each case in the litigation section shows its type (SCA, derivative, SEC enforcement, regulatory) derived from legal theories and named defendants, not from which data source found it
  2. Substantially similar cases from different sources appear as a single consolidated entry with all related case names listed
  3. Same-company cases from different years are disambiguated with year suffix (e.g., "In re Fastly (2020)" vs "In re Fastly (2024)")
  4. Each case shows its coverage side classification (Side A / B / C) derived from case type and named defendants
  5. Cases missing critical fields (case number, court, class period) are flagged in the output and acquisition is attempted from available sources
**Plans**: 3 plans
Plans:

- [x] 140-02-PLAN.md — Context builder display integration + visual verification checkpoint

### Phase 141: Output Sanitization
**Goal**: A single sanitization pass on completed HTML eliminates all markdown artifacts, raw Python serialization, and system jargon that leaked through existing filters
**Depends on**: Nothing (independent; operates post-render on final HTML)
**Requirements**: SAN-01, SAN-02, SAN-03, SAN-04, SAN-05
**Success Criteria** (what must be TRUE):
  1. No markdown formatting (`**`, `---`, `###`) appears in the rendered HTML worksheet for any ticker
  2. No raw Python serialization (`['...']`, `{'...'}`, debug evaluation strings) appears in the final output
  3. The sanitizer logs every substitution it makes, building evidence for upstream prevention rules
  4. All existing scattered sanitization functions are consolidated into a single OutputSanitizer class
**Plans**: 2 plans
Plans:
- [x] 141-01-PLAN.md — OutputSanitizer class with markdown/python/jargon/debug pattern categories + tests
- [x] 141-02-PLAN.md — Wire OutputSanitizer into html_renderer.py post-render + integration tests

### Phase 142: Quality Gates
**Goal**: Automated gates catch cross-section inconsistencies, suppress broken sections, validate template types against schemas, and replace unspec'd MagicMocks with real-state tests for context builders
**Depends on**: Phase 137 (consistency checking most effective against canonical values), Phase 138 (template type validation requires typed models)
**Requirements**: GATE-01, GATE-02, GATE-03, GATE-04, GATE-05
**Success Criteria** (what must be TRUE):
  1. Running the consistency checker against a rendered worksheet reports every instance where the same fact (revenue, CEO, exchange, case counts) differs between sections
  2. Sections with >50% N/A values are either suppressed or show an "Insufficient Data" banner instead of rendering broken tables
  3. Context builder integration tests run against real state.json files and fail when field paths change
  4. CI gate validates that template variables match Pydantic output schema field names
  5. Consistency checker starts in report-only mode and is promotable to blocking after zero inconsistencies are achieved
**Plans**: 2 plans
Plans:
- [x] 142-01-PLAN.md — Cross-section consistency checker + section completeness gate with Insufficient Data banners
- [x] 142-02-PLAN.md — Real-state integration tests for context builders + template type validation CI gate
