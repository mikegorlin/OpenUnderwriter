## Phase Details

### Phase 49: Pipeline Integrity, Facets & CI Guardrails
**Goal**: The brain is fully self-describing -- every check declares its data needs, display format, and facet membership -- with traceable data routes, verified rendering completeness, and CI tests that enforce the contract
**Depends on**: v1.1 (Phase 48)
**Requirements**: NOM-01, INT-01, INT-02, INT-03, FACET-01, FACET-02, FACET-03, FACET-04, QA-03
**Success Criteria** (what must be TRUE):
  1. "check" renamed to "signal" throughout codebase -- BrainCheckEntry -> BrainSignalEntry, all CLI commands, output, and internal references use consistent terminology
  2. User runs `do-uw brain trace <SIGNAL_ID>` and sees the complete data route from YAML definition through extraction, mapping, evaluation, to rendered output -- for any signal
  3. Every brain signal has a `facet` field and complete `display` spec; HTML rendering is organized by facet definitions, not hardcoded prefix mappings; `do-uw brain render-audit` confirms all evaluated signals appear in output
  4. SKIPPED count drops from 68 toward ~34 as Population B DEF14A signals gain working extraction and evaluate instead of skipping
  5. CI test suite (`test_brain_contract.py`) fails the build if any ACTIVE signal lacks a data route, threshold, v6_subsection_ids, or factor/peril mapping
  6. All facet signal IDs are validated correct -- every declared signal maps to a real brain signal with matching ID
**Plans**: 5 plans

Plans:
- [ ] 49-01-PLAN.md -- Big-bang rename check -> signal (Wave 1)
- [ ] 49-02-PLAN.md -- Facet metadata + definitions for all 8 domains (Wave 2)
- [ ] 49-03-PLAN.md -- DEF14A Population B signal fixes + INACTIVE triage (Wave 2)
- [ ] 49-04-PLAN.md -- brain trace + render-audit CLI commands + facet-aware rendering (Wave 3)
- [ ] 49-05-PLAN.md -- CI contract tests + nomenclature lint guard (Wave 3)

### Phase 50: Automated QA & Anomaly Detection
**Goal**: The pipeline self-validates after every run and the user has full visibility into brain health and cross-run behavioral changes
**Depends on**: Phase 49
**Requirements**: QA-01, QA-02, QA-04, QA-05
**Success Criteria** (what must be TRUE):
  1. Every `do-uw analyze` run ends with an automated health summary showing evaluated/TRIGGERED/SKIPPED counts and anomaly warnings (e.g., 0 TRIGGERED when litigation data is present)
  2. User runs `do-uw brain health` and sees unified coverage %, fire rate distribution, top never-fire/always-fire checks, data freshness, and feedback queue status
  3. User runs `do-uw brain delta <TICKER>` and sees which checks flipped CLEAR to TRIGGERED or TRIGGERED to CLEAR since the last run for that ticker
  4. User runs `do-uw brain audit` and gets a report on staleness, coverage imbalance across perils, threshold conflicts, and orphaned checks
**Plans**: 4 plans

Plans:
- [ ] 50-01-PLAN.md -- Signal result enrichment + post-analyze health summary (Wave 1)
- [ ] 50-02-PLAN.md -- Brain health + brain audit CLI commands (Wave 1)
- [ ] 50-03-PLAN.md -- Brain delta cross-run comparison CLI command (Wave 2)
- [ ] 50-04-PLAN.md -- Signal Composites foundation + facet evolution (Wave 2)

### Phase 51: Feedback Loop
**Goal**: Underwriter reactions captured via CLI flow end-to-end through auto-generated calibration proposals to brain YAML modification with git commit -- the feedback loop closes completely
**Depends on**: Phase 50
**Requirements**: FEED-01
**Success Criteria** (what must be TRUE):
  1. User runs `do-uw feedback <TICKER>` to capture structured reactions to check results (agree/disagree/severity adjustment), and feedback is stored with check ID references
  2. User runs `do-uw feedback process` and pending feedback entries are batch-processed into brain calibration proposals with before/after impact projections
  3. User runs `do-uw brain apply-proposal <id>` and the system modifies the brain YAML file directly, runs `brain build` to resync DuckDB, validates regression, and creates a git commit -- the proposal reaches brain YAML, not just DuckDB
**Plans**: 3 plans

Plans:
