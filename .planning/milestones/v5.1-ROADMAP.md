## Phase Details

### Phase 88: Data Foundation
**Goal**: Every stock analysis in the system operates on 2 years of daily data, every return is decomposed into market + sector + company-specific components, and the underwriter can see how a company's maximum drawdown compares to its sector
**Depends on**: Nothing (first phase of v5.1)
**Requirements**: STOCK-07, STOCK-01, STOCK-03
**Success Criteria** (what must be TRUE):
  1. Running `underwrite AAPL` acquires 2 years of daily stock data (not 1 year) -- visible in the stock charts spanning a full 2-year window and in the data manifest
  2. The stock analysis section shows every major return period (1Y, 5Y) broken into three labeled components: market contribution, sector contribution, and company-specific residual -- with the three summing to the total return
  3. The MDD ratio (company MDD / sector ETF MDD) is computed and displayed for both 1Y and 5Y periods, showing whether the company's worst drawdown is normal or outsized relative to its sector
  4. All downstream stock metrics (drops, volatility, scoring) use the 2-year daily dataset as their input
**Plans:** 2/2 plans complete
Plans:
- [x] 88-01-PLAN.md — 2Y acquisition, model fields, return decomposition + MDD ratio computation with tests
- [x] 88-02-PLAN.md — Display decomposition and MDD ratio in charts and HTML template

### Phase 89: Statistical Analysis
**Goal**: The system quantifies DDL/MDL dollar exposure from stock drops, identifies statistically significant abnormal returns on event days, and classifies market volatility into regimes that inform underwriting risk assessment
**Depends on**: Phase 88
**Requirements**: STOCK-02, STOCK-04, STOCK-05
**Success Criteria** (what must be TRUE):
  1. The pricing section shows DDL exposure (market_cap x worst_drop_pct) and an estimated settlement amount (DDL x 1.8%) for the company's worst stock drop -- giving the underwriter a concrete dollar figure for potential exposure
  2. Event days with abnormal returns are flagged with t-statistics, and only days where |t| >= 1.96 are highlighted as statistically significant -- distinguishing real anomalies from noise
  3. The stock analysis shows both simple rolling volatility and EWMA (lambda=0.94) volatility, with periods classified into regimes (low/normal/elevated/crisis) -- the underwriter can see whether current volatility is abnormal
  4. Volatility regime transitions are visible in the output, showing when the company moved between regimes and how long each regime lasted
**Plans:** 2/2 plans complete
Plans:
- [x] 89-01-PLAN.md — Computation functions (DDL, abnormal return, EWMA, regime) + model fields + unit tests
- [x] 89-02-PLAN.md — Wire computations into extraction pipeline + render in charts and pricing section

### Phase 90: Drop Enhancements
**Goal**: Stock drop scoring reflects recency (recent drops matter more), each individual drop is decomposed to show whether the company or the market drove it, and unexplained drops trigger automatic investigation for contemporaneous disclosures
**Depends on**: Phase 88, Phase 89
**Requirements**: STOCK-06, STOCK-08, STOCK-09
**Success Criteria** (what must be TRUE):
  1. A drop from 3 months ago scores higher than an identical drop from 18 months ago -- the time-decay function visibly affects the drop severity ranking in the output
  2. Each identified drop event shows its own 3-component decomposition (market / sector / company-specific) -- the underwriter can see at a glance whether a 15% drop was company-driven or a market-wide sell-off
  3. Drops without a known catalyst trigger a reverse lookup that searches for contemporaneous 8-K filings and news -- any findings are attached to the drop as potential corrective disclosures
  4. The corrective disclosure enhancement surfaces previously unlinked 8-K/news items that coincide with significant drops
**Plans:** 2/2 plans complete
Plans:
- [ ] 90-01-PLAN.md — Model fields, decay computation, per-drop decomposition, corrective disclosure reverse lookup + tests
- [ ] 90-02-PLAN.md — Wire into extraction pipeline, F2 scoring modifiers, HTML/Word rendering columns

### Phase 91: Display Centralization
**Goal**: All chart evaluation thresholds and risk callout text are declared in signal YAML (not hardcoded in templates), and the rendering pipeline reads presentation configuration from signal data rather than inline template literals
**Depends on**: Phase 88
**Requirements**: DISP-01, DISP-02, DISP-03, DISP-04
**Success Criteria** (what must be TRUE):
  1. Changing a chart evaluation threshold (e.g., beta "elevated" threshold from 1.5 to 1.8) requires editing only the signal YAML file -- no Jinja2 template changes needed
  2. The amber/green risk evaluation callout boxes in the stock charts section are generated from signal presentation templates with D&O underwriting context -- not from strings embedded in the HTML template
  3. A chart type registry YAML file declares all available charts, their data requirements, which signals they visualize, and which output formats support them (HTML/Word/PDF)
  4. Adding a new chart type requires only a registry entry and a rendering function -- no template surgery to wire it in
**Plans:** 3/3 plans complete
Plans:
- [ ] 91-01-PLAN.md — Signal YAML threshold extension + chart_thresholds.py context builder + template threshold variables
- [ ] 91-02-PLAN.md — Chart type registry YAML + registry loader/validator with function resolution
- [ ] 91-03-PLAN.md — Signal callout_templates + callout aggregation + template simplification + CI lint test

### Phase 92: Rendering Completeness
**Goal**: The system guarantees that every piece of extracted data reaches the output, with CI enforcement, post-pipeline audit trails, and cross-ticker validation proving nothing is silently lost
**Depends on**: Phase 88
**Requirements**: REND-01, REND-02, REND-03, REND-04
**Success Criteria** (what must be TRUE):
  1. A CI contract test validates that every non-empty field in the extracted model has a corresponding render path (template reference or context builder mapping) -- adding a new extraction field without a render path fails the build
  2. After a pipeline run, an audit report lists any extracted-but-not-displayed fields for that ticker -- the underwriter can verify nothing important was silently dropped
  3. A post-pipeline health check flags data quality issues in the rendered output: empty percentage values, 0.0 placeholders, raw LLM text leaking through templates
  4. Running cross-ticker QA on multi-segment companies (e.g., RPM, AAPL) validates that business profile fields (revenue segments, customer/supplier concentration, geographic footprint) are populated
**Plans:** 2/2 plans complete
Plans:
- [ ] 92-01-PLAN.md — CI render-path contract + render exclusions config + post-pipeline audit report + state.json injection
- [ ] 92-02-PLAN.md — Health check heuristics (LLM text, 0.0 placeholders, empty %) + cross-ticker QA business profile validation
