### v2.0 Brain-Driven Architecture (Phases 53-57)

**Milestone Goal:** Make the brain YAML the complete contract driving acquisition, evaluation, AND rendering. Eliminate hardcoded acquisition clients, procedural mapper code, English-text threshold parsing, and hardcoded section renderers. A human can read any signal YAML and understand: why we check this, where we get the data, how we evaluate it, and how we display it.

- [x] **Phase 53: Data Store Simplification** - YAML direct read at runtime, config consolidation, single loader, DuckDB scoped to history only
- [x] **Phase 54: Signal Contract V2** - Extend BrainSignalEntry with optional acquisition, evaluation, presentation sections; migrate 10-15 representative signals (completed 2026-03-01)
- [x] **Phase 55: Declarative Mapping & Structured Evaluation** - Dotted-path field resolution from ExtractedData, operator-based threshold evaluation, migrate first prefix end-to-end (completed 2026-03-01)
- [x] **Phase 56: Facet-Driven Rendering** - Facets define subsections with layout specs, renderer dispatches from facet YAML, migrate first facet section (completed 2026-03-02)
- [x] **Phase 57: Closed Learning Loop** - Statistical calibration, co-occurrence mining, fire rate alerts, signal lifecycle state machine (completed 2026-03-02)

## Phase Details

### Phase 53: Data Store Simplification
**Goal**: Single source of truth for signal definitions. YAML files read directly at runtime (no DuckDB intermediary for definitions). Config consolidated. One loader replaces four. DuckDB retains run history and analytics only.
**Depends on**: v1.2 (Phase 52.1)
**Requirements**: STORE-01, STORE-02, STORE-03, STORE-04, STORE-05
**Success Criteria** (what must be TRUE):
  1. `BrainYAMLLoader.load_signals()` reads 400 signals from `brain/signals/*.yaml` in < 1 second — verified by benchmark test
  2. `config/` directory deleted; all 21 duplicate configs merged into `brain/config/`; all existing import paths work via re-exports or updates
  3. `BrainDBLoader`, `BrainKnowledgeLoader`, `BackwardCompatLoader`, `ConfigLoader` replaced by single `BrainLoader` — grep confirms zero remaining imports of old loaders
  4. brain.duckdb signal definition tables (`brain_signals`, `brain_taxonomy`) no longer read at pipeline runtime — only `brain_signal_runs`, `brain_effectiveness`, `brain_feedback`, `brain_changelog` are read
  5. `brain build` no longer migrates YAML→DuckDB for signal definitions; signals.json export preserved; all 3,967+ tests pass
  6. Pipeline produces identical output on SNA before/after (HTML diff)
**Plans**: 3 plans
  - [x] 53-01-PLAN.md -- BrainLoader creation + config consolidation (STORE-01, STORE-02)
  - [x] 53-02-PLAN.md -- Import site migration + old code deletion (STORE-03, STORE-04)
  - [x] 53-03-PLAN.md -- Brain CLI + brain build simplification + SNA verification (STORE-04, STORE-05)

### Phase 54: Signal Contract V2
**Goal**: Extend signal YAML with machine-readable acquisition, evaluation, and presentation sections. Add `schema_version` for dual-path dispatch and field registry for declarative resolution. Schema is additive — old signals keep working unchanged. 10-15 signals migrated as proof of concept.
**Depends on**: Phase 53
**Requirements**: SCHEMA-01, SCHEMA-02, SCHEMA-03, SCHEMA-04, SCHEMA-05, SCHEMA-06
**Success Criteria** (what must be TRUE):
  1. BrainSignalEntry Pydantic model accepts optional `acquisition` (sources list with dotted paths, fallback chain), `evaluation` (formula, structured thresholds with operators), and `presentation` (facet, subsection, format, detail_levels) sections
  2. `schema_version` field on BrainSignalEntry (default: 1) — signal engine dispatches to legacy or declarative path based on this field
  3. `brain/field_registry.yaml` maps logical field names to ExtractedData/CompanyProfile dotted paths — fields classified as DIRECT_LOOKUP or COMPUTED
  4. Existing 400 signals load without modification — V2 fields are optional with sensible defaults
  5. 10-15 signals across FIN, GOV, LIT, STOCK, BIZ prefixes have V2 fields populated and `schema_version: 2` — Pydantic-validated at load time
  6. V2 fields are stored but NOT consumed by pipeline yet — evaluation results identical before/after
**Plans**: TBD

### Phase 55: Declarative Mapping & Structured Evaluation
**Goal**: Signals with V2 paths resolve data declaratively via field registry. Signals with V2 thresholds evaluate via structured operators. Shadow evaluation proves parity. At least one full prefix migrated end-to-end.
**Depends on**: Phase 54
**Requirements**: MAP-01, MAP-02, MAP-03, MAP-04, EVAL-01, EVAL-02, EVAL-03, EVAL-04, EVAL-05
**Success Criteria** (what must be TRUE):
  1. `declarative_mapper.py` resolves DIRECT_LOOKUP fields from field registry paths — SourcedValue-aware auto-unwrap with confidence propagation. COMPUTED fields dispatch to named Python functions.
  2. `structured_evaluator.py` evaluates signals using `{op, value}` operators — supports >, <, >=, <=, ==, !=, between, contains. Handles clear_conditions for qualitative-clear signals.
  3. Shadow evaluation mode: both legacy and V2 evaluators run on migrating signals, discrepancies logged. Switch only at zero discrepancy across 3 ticker runs.
  4. Multi-field formulas support arithmetic and built-in functions; composite formulas (Altman Z) remain procedural via named evaluator dispatch
  5. Signals without V2 fields use legacy mapper/evaluator unchanged — no signal produces a different result
  6. At least one full prefix (e.g., FIN.LIQ.* — 5 signals) migrated: V2 YAML populated, declarative mapper used, FIELD_FOR_CHECK entries removed, shadow evaluation confirms identical results
  7. Edge cases: None → SKIPPED, empty list → 0 for counts, "N/A" → missing, boolean coercion handled
**Plans**: TBD

### Phase 56: Facet-Driven Rendering
**Goal**: Facets control what appears in each document section. Subsections with layout specs drive template dispatch. At least one facet fully migrated with no hardcoded renderer involvement.
**Depends on**: Phase 55
**Requirements**: RENDER-01, RENDER-02, RENDER-03, RENDER-04
**Success Criteria** (what must be TRUE):
  1. Facet YAML files have `subsections` list with `id`, `name`, `render_as`, `signals`, `columns` — validated by Pydantic schema at load time
  2. `facet_renderer.py` orchestrates existing Jinja2 templates based on `render_as` type (kv_table, scorecard, narrative, chart) — facets add structure around existing 4,000+ lines of renderers, not replace them
  3. At least one facet (e.g., Financial Health) renders from facet YAML subsections with dispatcher orchestrating existing templates
  4. Facets without `subsections` field use existing hardcoded renderers unchanged — no visual regression
  5. Side-by-side HTML comparison confirms zero visual regression for fully-migrated facet section
**Plans**: TBD

### Phase 57: Closed Learning Loop
**Goal**: Every pipeline run makes the brain smarter. Signal effectiveness is tracked, thresholds drift-checked, correlations discovered, and signal lifecycle managed.
**Depends on**: Phase 55
**Requirements**: LEARN-01, LEARN-02, LEARN-03, LEARN-04
**Success Criteria** (what must be TRUE):
  1. `brain audit` includes statistical threshold analysis — compares observed values from `brain_signal_runs` against thresholds, flags signals >2σ from observed distribution
  2. `brain audit` identifies co-occurring signals (>70% co-fire rate) and populates `correlated_signals` field on signal YAML
  3. `brain audit` flags signals with extreme fire rates (>80% or <2%) as calibration candidates
  4. Signal lifecycle states (INCUBATING → ACTIVE → MONITORING → DEPRECATED → ARCHIVED) implemented with transition proposals from `brain audit` and confirmation via `feedback process`
