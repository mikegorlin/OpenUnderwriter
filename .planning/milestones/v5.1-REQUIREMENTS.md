# Requirements: D&O Underwriting System

**Defined:** 2026-03-08
**Core Value:** Surface every red flag and risk signal that exists in public data for any publicly traded company, benchmarked against industry peers, so an underwriter can make the most informed D&O decision possible.

## v5.1 Requirements

Requirements for Stock Analysis Engine + Display Centralization. Each maps to roadmap phases.

### Stock Analysis Engine (STOCK)

- [x] **STOCK-01**: Every stock return (1Y, 5Y, per-drop) decomposed into 3 components: market contribution + sector contribution + company-specific residual
- [x] **STOCK-02**: DDL/MDL exposure computed as market_cap × worst_drop_pct with settlement estimate (DDL × 1.8%), displayed in pricing section
- [x] **STOCK-03**: Peer-relative MDD ratio (company MDD / sector ETF MDD) computed and displayed for both 1Y and 5Y periods
- [x] **STOCK-04**: Abnormal returns on event days computed via market model (AR = R_actual - α̂ - β̂·R_market) with t-stat significance (|t| ≥ 1.96), flagging statistically significant days
- [x] **STOCK-05**: EWMA volatility (λ=0.94) computed alongside simple rolling vol, with regime detection classifying periods as low/normal/elevated/crisis volatility
- [x] **STOCK-06**: Corrective disclosure detection enhanced with reverse lookup — unexplained drops trigger search for contemporaneous 8-K filings and news
- [x] **STOCK-07**: Daily stock data acquisition extended from 1-year to 2-year lookback for all metrics (drops, volatility, returns)
- [x] **STOCK-08**: Time-decay weighting applied to stock drops — recent drops score higher than older ones using exponential decay function
- [x] **STOCK-09**: Per-drop return decomposition showing market/sector/company-specific attribution for every identified drop event

### Display Centralization (DISP)

- [x] **DISP-01**: Chart evaluation thresholds (beta, volatility, MDD, decline) declared in signal YAML presentation blocks, not hardcoded in Jinja2 templates
- [x] **DISP-02**: Signal presentation blocks wired to chart rendering — template reads thresholds and evaluation text from signal data, not inline literals
- [x] **DISP-03**: Chart type registry YAML declares available charts, their data requirements, signal linkage, and render targets (HTML/Word/PDF)
- [x] **DISP-04**: Evaluation callout text (amber/green risk boxes) generated from signal presentation templates with D&O context, not template-embedded strings

### Rendering Completeness (REND)

- [x] **REND-01**: CI contract test validates every non-empty extracted model field has a render path (template reference or context builder mapping)
- [x] **REND-02**: Post-pipeline audit report lists any extracted-but-not-displayed fields for the current ticker run
- [x] **REND-03**: Post-pipeline health check flags data quality issues in rendered output — empty percentages, 0.0 placeholder values, raw LLM text leaking through
- [x] **REND-04**: Cross-ticker QA validates business profile fields (revenue segments, customer/supplier concentration, geographic footprint) populated for multi-segment companies

## Out of Scope

Explicitly excluded. Documented to prevent scope creep.

| Feature | Reason |
|---------|--------|
| Dynamic signal creation at runtime | Breaks auditability and reproducibility; signals must be declared and versioned |
| ML-based sector classification | Over-engineered for ~4,000 SEC filers; SIC codes already assigned by SEC |
| Real-time sector risk scores | System does point-in-time analysis; D&O policies are annual |
| Competitor-by-competitor analysis | 5-10x cost for deep peer analysis; Frames API percentile ranking suffices |
| Full chart rewrite to declarative engine | Over-engineered; centralize config in YAML, keep matplotlib rendering code |
| Factor model beyond 2-factor | SMB/HML adds complexity without D&O underwriting value; market+sector sufficient |
