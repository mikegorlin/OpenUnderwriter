# Requirements: D&O Underwriting System

**Defined:** 2026-02-26
**Core Value:** Surface every red flag and risk signal that exists in public data for any publicly traded company, benchmarked against industry peers, so an underwriter can make the most informed D&O decision possible.

## v1.2 Requirements

Requirements for v1.2 System Intelligence milestone. Each maps to roadmap phases.

### Nomenclature

- [x] **NOM-01**: "check" renamed to "signal" throughout codebase -- BrainCheckEntry -> BrainSignalEntry, all CLI commands, output, docs, and internal references use consistent "signal" terminology

### Pipeline Integrity

- [x] **INT-01**: System closes SKIPPED gap -- Population B DEF14A signals evaluate instead of SKIP, reducing SKIPPED from 68 toward ~34
- [x] **INT-02**: User can trace any signal's full data route with `do-uw brain trace <SIGNAL_ID>` -- one command shows YAML -> extraction -> mapper -> evaluator -> renderer
- [x] **INT-03**: System verifies every evaluated signal appears in HTML output -- `do-uw brain render-audit` reports declared vs populated signals per facet

### Facets & Signal Completeness

- [x] **FACET-01**: Facet definitions exist for every signal domain (BIZ, FIN, GOV, LIT, STOCK, EXEC, NLP, FWRD, MKT, etc.) with declared signals
- [x] **FACET-02**: Every brain signal has `facet` field and complete `display` spec (value_format, source_type, threshold_context)
- [x] **FACET-03**: HTML rendering is driven by facet definitions, not hardcoded prefix mappings -- facets control section organization and signal grouping
- [x] **FACET-04**: All facet signal IDs are correct -- every declared signal maps to a real brain signal with matching ID

### Quality Assurance

- [x] **QA-01**: Every `do-uw analyze` run ends with automated health summary -- evaluated/TRIGGERED/SKIPPED counts, anomaly warnings (e.g., 0 TRIGGERED with litigation data present)
- [x] **QA-02**: User can view unified system health with `do-uw brain health` -- coverage %, fire rate distribution, top never-fire/always-fire signals, data freshness, feedback queue
- [x] **QA-03**: CI tests enforce brain contract -- every ACTIVE signal has data route, threshold, v6_subsection_ids, and factor/peril mapping; new signal without these fails tests
- [x] **QA-04**: User can detect cross-run changes with `do-uw brain delta <TICKER>` -- shows checks that flipped CLEAR<->TRIGGERED since last run
- [x] **QA-05**: User can audit brain health with `do-uw brain audit` -- staleness, coverage imbalance across perils, threshold conflicts, orphaned checks

### Feedback

- [x] **FEED-01**: Underwriter feedback loop works end-to-end -- feedback CLI captures reactions, auto-generates calibration proposals, `do-uw brain apply-proposal` writes changes to brain YAML with git commit

### Data Quality

- [x] **DQ-01**: Board directors array populated from DEF 14A -- individual names, independence, committees, qualifications extracted (not just aggregate stats)
- [x] **DQ-02**: Guidance vs consensus correctly distinguished -- FIN.GUIDE.* signals check `provides_forward_guidance` before evaluating; non-guiding companies show N/A
- [x] **DQ-03**: Litigation extraction rejects boilerplate -- LLM requires named parties/court/docket for SCA; post-extraction filter drops hollow CaseDetail records
- [x] **DQ-04**: Volume spike detection and event correlation -- STOCK.TRADE.volume_patterns upgraded to evaluative with tiered thresholds; spikes trigger targeted news search

## Out of Scope

| Feature | Reason |
|---------|--------|
| Auto-apply brain changes without human review | System proposes, human disposes -- all mutations require explicit approval |
| Real-time pipeline monitoring (Prometheus/Grafana) | CLI tool, not a service -- no persistent process to monitor |
| Web UI for brain CRUD operations | Dashboard stays read-only; CLI is the mutation surface |
| ML threshold optimization | Catastrophically insufficient data (3-4 tickers); need 50+ companies first |
| Cross-company correlation | Premature with < 10 companies analyzed |
| Push notifications / alerting | Single-operator CLI tool; console output is sufficient |
| Scheduling infrastructure (APScheduler/Celery) | System cron or manual invocation is correct for CLI tool |
