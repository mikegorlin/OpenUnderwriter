# Requirements: D&O Underwriting System v2.0

**Defined:** 2026-02-28
**Core Value:** Surface every red flag and risk signal that exists in public data for any publicly traded company, benchmarked against industry peers, so an underwriter can make the most informed D&O decision possible.

## v2.0 Requirements

Requirements for v2.0 Brain-Driven Architecture milestone. Each maps to roadmap phases. The brain YAML becomes the complete contract driving acquisition, evaluation, and rendering — eliminating hardcoded pipeline behavior and intermediary databases for signal definitions.

### Data Store Simplification (STORE)

- [x] **STORE-01**: Signals load from YAML at runtime — `BrainYAMLLoader` reads 400 signals from `brain/signals/` directly, no DuckDB intermediary for signal definitions. Load time < 1 second.
- [x] **STORE-02**: Config directory consolidated — 21 overlapping config files between `config/` and `brain/config/` merged into single `brain/config/` directory (per-file diff review to select canonical version, since files have diverged). `config/` directory deleted. Import paths updated or re-exported.
- [x] **STORE-03**: Single loader replaces 4 — `BrainDBLoader`, `BrainKnowledgeLoader`, `BackwardCompatLoader`, and `ConfigLoader` replaced by one `BrainLoader` that reads YAML/JSON from brain/ directory.
- [x] **STORE-04**: DuckDB scoped to history only — brain.duckdb retains `brain_signal_runs`, `brain_effectiveness`, `brain_feedback`, `brain_changelog`, `brain_proposals`. Signal definitions, scoring factors, patterns, red flags, sectors read from YAML/JSON at runtime.
- [x] **STORE-05**: `brain build` simplified — no longer migrates YAML→DuckDB for signal definitions. Only rebuilds signals.json export and validates schema. Existing `brain build` workflow preserved for history/analytics tables.

### Signal Contract V2 (SCHEMA)

- [x] **SCHEMA-01**: BrainSignalEntry extended with optional `acquisition` section — sources list with type, sections, fields (dotted paths into ExtractedData), and fallback chain. Old signals without `acquisition` keep working unchanged.
- [x] **SCHEMA-02**: BrainSignalEntry extended with optional `evaluation` section — formula field, structured thresholds with operators (`op`, `value`, `window_years`, `label`). Old signals with English-text thresholds keep working unchanged.
- [x] **SCHEMA-03**: BrainSignalEntry extended with optional `presentation` section — facet, subsection, format (label/value_type/context_template), detail_levels (glance/standard/deep). Old signals with existing `display` spec keep working unchanged.
- [x] **SCHEMA-04**: `schema_version` field on BrainSignalEntry — default 1 (legacy), set to 2 for V2 signals. Signal engine dispatches to legacy or declarative mapper/evaluator based on this field.
- [x] **SCHEMA-05**: Field registry YAML — single `brain/field_registry.yaml` maps logical field names to ExtractedData dotted paths and CompanyProfile paths. Replaces 371-line FIELD_FOR_CHECK Python dict as the source of truth for field resolution. Fields classified as DIRECT_LOOKUP or COMPUTED.
- [x] **SCHEMA-06**: 10-15 representative signals migrated to V2 format — at least 2 signals from each of FIN, GOV, LIT, STOCK, BIZ prefixes. V2 fields validated by Pydantic schema at load time.

### Declarative Data Mapping (MAP)

- [x] **MAP-01**: Declarative mapper resolves signal data from field registry paths — DIRECT_LOOKUP fields resolved by traversing ExtractedData/AnalysisState via dotted paths. COMPUTED fields dispatch to named Python functions. No manual mapper entry needed for either type.
- [x] **MAP-02**: SourcedValue-aware path resolver — dotted paths auto-unwrap SourcedValue at each traversal step, propagating source and confidence metadata. Supports dual roots (`extracted.*` for ExtractedData, `company.*` for CompanyProfile).
- [x] **MAP-03**: Legacy fallback preserved — signals without V2 `acquisition` paths use existing mapper chain (signal_mappers.py → signal_field_routing.py). No signal evaluates differently after migration.
- [x] **MAP-04**: At least one full prefix migrated — all signals in one prefix (e.g., FIN.LIQ.*) use declarative mapping exclusively. Corresponding FIELD_FOR_CHECK entries removed. Regression test confirms identical results.

### Structured Threshold Evaluation (EVAL)

- [x] **EVAL-01**: Structured evaluator handles operator-based thresholds — when a signal has `evaluation.thresholds` with `{op, value}`, evaluation uses direct comparison instead of English-text parsing. Supported operators: `>`, `<`, `>=`, `<=`, `==`, `!=`, `between`, `contains`.
- [x] **EVAL-02**: Formula evaluation for single-field and multi-field signals — `evaluation.formula` supports field references, arithmetic expressions, and built-in functions (`count_within`, `years_since`, `pct_change`). Composite formulas (Altman Z-Score) stay procedural via named evaluator dispatch.
- [x] **EVAL-03**: Edge cases handled — None/missing data returns SKIPPED (not error), empty lists evaluate as 0 for count operators, "N/A" strings treated as missing.
- [x] **EVAL-04**: Legacy fallback preserved — signals without V2 `evaluation` section use existing check_evaluators.py. No signal produces a different result after migration.
- [x] **EVAL-05**: Shadow evaluation mode — during migration, both legacy and V2 evaluators run on every migrating signal. Discrepancies logged to console and DuckDB. Shadow runs permanently for ongoing regression detection. Auto-promotion logic (switching to V2 primary after N clean runs) deferred to Phase 57 (LEARN-04 signal lifecycle).

### Facet-Driven Rendering (RENDER)

- [x] **RENDER-01**: Facets define subsections with layout specs — each facet YAML gains `subsections` list with `id`, `name`, `render_as` (kv_table/scorecard/narrative/chart), `signals`, `columns`.
- [x] **RENDER-02**: Facet renderer orchestrates existing templates — `render_facet_section()` dispatches to existing Jinja2 section templates (not replacing them) based on `render_as` type. Facets add structure around existing renderers, not replace 4,000+ lines of narrative/conditional/chart logic.
- [x] **RENDER-03**: At least one facet fully migrated — one facet section (e.g., Financial Health) renders entirely from facet YAML subsections with no hardcoded renderer involvement. Visual output matches existing.
- [x] **RENDER-04**: Legacy fallback preserved — facets without `subsections` field use existing hardcoded section renderers. No visual regression in HTML output.

### Closed Learning Loop (LEARN)

- [x] **LEARN-01**: Statistical threshold calibration — `brain audit` analyzes observed values from `brain_signal_runs` against current thresholds, flags signals where threshold is >2σ from observed distribution, proposes adjustments.
- [x] **LEARN-02**: Co-occurrence mining — `brain audit` identifies signals that fire together on >70% of analyzed companies, auto-populates `correlated_signals` field on signal YAML.
- [x] **LEARN-03**: Fire rate alerts — signals firing on >80% or <2% of companies flagged as potential calibration candidates in `brain audit` output.
- [x] **LEARN-04**: Signal lifecycle state machine — formal states (INCUBATING → ACTIVE → MONITORING → DEPRECATED → ARCHIVED) with transitions proposed by `brain audit` based on fire rate + feedback + age, confirmed by underwriter.

## Future Requirements

Deferred to v2.1+. Tracked but not in current roadmap.

### Acquisition Intelligence (deferred)

- **ACQ-01**: Signal `acquisition` specs influence ACQUIRE stage behavior — pre-scan signals to build acquisition plan, skip unused data clients when no active signals need them
- **ACQ-02**: Automatic fallback chain execution — if primary source returns no data, automatically try fallback sources defined in signal YAML

### Advanced Learning (deferred)

- **ADV-01**: A/B threshold testing — new threshold proposals tested against next N runs before committing
- **ADV-02**: Cross-company correlation engine — detect sector-wide risk patterns across 50+ analyzed companies
- **ADV-03**: ML-based threshold optimization — train on historical runs to optimize per-signal thresholds

### Full Rendering Migration (deferred)

- **REND-01**: All 9 facets fully migrated to facet-driven rendering — zero hardcoded section renderers remain
- **REND-02**: Detail level rendering — HTML output supports glance/standard/deep views switchable in browser

## Out of Scope

| Feature | Reason |
|---------|--------|
| Full ACQUIRE stage rewrite | Acquisition intelligence deferred — v2.0 focuses on mapping/eval/render |
| Real-time YAML file watching | Pipeline runs are batch; hot-reload adds complexity without benefit |
| Web UI for signal editing | CLI is the mutation surface; YAML editing via text editor or feedback CLI |
| Auto-apply brain changes without review | System proposes, human disposes — all mutations require explicit approval |
| GraphQL/REST API for brain queries | CLI tool; DuckDB direct queries sufficient for analytics |
