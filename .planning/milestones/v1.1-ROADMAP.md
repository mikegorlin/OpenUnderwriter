## Phase Details

### Phase 46: Brain-Driven Gap Search
**Goal**: The pipeline closes the brain-to-acquire feedback loop — checks with no structured-source data get targeted web searches, and their results feed back into re-evaluation so SKIPPED checks become TRIGGERED or CLEAR rather than silently absent.
**Depends on**: Phase 45 (v1.0 complete codebase)
**Requirements**: GAP-01, GAP-02, GAP-03, GAP-04, GAP-05, GAP-06
**Success Criteria** (what must be TRUE):
  1. After running `do-uw analyze AAPL`, the total SKIPPED check count is lower than the v1.0 baseline of 68, with the difference attributable to web-sourced re-evaluations visible in the QA audit table showing `confidence=LOW` and `source="WEB (gap)"`
  2. No new TRIGGERED findings appear on AAPL (a known-clean company) after Phase 46 — the evidence quality gate prevents unscoped web snippets from directly triggering checks
  3. The pipeline completes within the existing 50-search budget cap, with gap searches consuming at most 15 of those searches per run (budget gating visible in pipeline logs)
  4. Re-evaluated checks have their results persisted to `AcquiredData.brain_targeted_search` so that a second run of the same ticker does not fire redundant gap searches (results survive pipeline stages)
  5. L1 acquisition-tier checks (XBRL financial data — must come from structured sources) remain ineligible for web fallback and are never re-evaluated via gap search
**Plans**: 4 plans

Plans:
- [ ] 46-01-PLAN.md — Triage: classify 68 SKIPPED checks, add gap_bucket + gap_keywords to brain YAMLs, rebuild brain DB
- [ ] 46-02-PLAN.md — State model: add AcquiredData.brain_targeted_search + CheckResult.confidence fields
- [ ] 46-03-PLAN.md — Gap searcher: gap_searcher.py + gap_query_generator.py + orchestrator Phase E wiring
- [ ] 46-04-PLAN.md — Re-evaluation: gap_revaluator.py + AnalyzeStage wiring + QA audit summary

### Phase 47: Check Data Mapping Completeness
**Goal**: Every SKIPPED check with an existing structured data field in `ExtractedData` has a routing entry, and the residual SKIPPED population consists only of intentionally-unmapped qualitative checks — reducing the floor from 68 to ~20-30 permanently, independent of gap search.
**Depends on**: Phase 46 (gap search audit classifies the 68 checks into routing-gap, intentionally-unmapped, aspirational buckets — that classification drives this phase)
**Requirements**: MAP-01, MAP-02, MAP-03, QA-03
**Success Criteria** (what must be TRUE):
  1. All ~40 bucket-a checks (routing gap — field exists in `ExtractedData` but `FIELD_FOR_CHECK` entry is missing) have routing entries and no longer appear as "Data mapping not configured" in the QA audit table
  2. All ~28 bucket-b checks (extraction gap — required data exists in source filings but no extractor reads it into `ExtractedData`) have new extractor fields populated on AAPL, RPM, and TSLA
  3. DEF 14A extraction on AAPL, RPM, and TSLA populates board diversity, director tenure, expertise, and attendance fields in `DEF14AExtraction` at a rate of 80%+ across the regression set
  4. Every TRIGGERED finding in the QA audit table shows a human-readable threshold criterion (e.g., "red: Prior SEC enforcement action within 5 years") sourced from the brain check's `threshold.red`/`threshold.yellow` text — the `threshold_context` field on `CheckResult` is populated at evaluation time
  5. Regression on AAPL, RPM, and TSLA confirms no false triggers introduced by routing changes — TRIGGERED count on clean AAPL does not increase
**Plans**: 4 plans

Plans:
- [ ] 47-01-PLAN.md — Re-audit SKIPPED checks + regression baseline snapshot + Wave 0 test scaffolds
- [ ] 47-02-PLAN.md — threshold_context field on CheckResult + _apply_traceability() injection (QA-03)
- [ ] 47-03-PLAN.md — Bucket-a routing fixes: brain YAML data_strategy + FIELD_FOR_CHECK + brain build
- [ ] 47-04-PLAN.md — DEF 14A schema expansion + convert_board_profile() + mapper None placeholder fixes

### Phase 48: Output Quality Hardening
**Goal**: The QA audit table is accurate and trustworthy — source column shows actual filing references or web URLs (never "—"), value column shows the datum that was evaluated, and every TRIGGERED finding in the red flags section displays its threshold criterion alongside the finding value.
**Depends on**: Phase 47 (threshold context field on `CheckResult` must exist before templates can render it)
**Requirements**: QA-01, QA-02, QA-04, QA-05
**Success Criteria** (what must be TRUE):
  1. The QA audit table source column shows the actual filing reference (e.g., "10-K 2024-09-28") or "WEB (gap)" URL for every evaluated check — no row shows "—" as source
  2. The QA audit table value column shows the raw datum evaluated for each check (e.g., "1.23" for a current ratio check, not blank) — the `result.value` field is populated by all threshold evaluator types
  3. Every TRIGGERED finding in the red flags HTML section displays the human-readable threshold criterion alongside the finding (e.g., "red: Prior SEC enforcement action within 5 years") sourced from `threshold_context`
  4. Full regression on AAPL, RPM, and TSLA confirms: SKIPPED count is lower than the v1.0 baseline of 68, TRIGGERED count on AAPL does not increase versus v1.0 baseline, and output is approved on human review of the HTML worksheet
**Plans**: 4 plans

Plans:
- [ ] 48-01-PLAN.md — Signal display: schema (DisplaySpec in brain_check_schema.py) + Wave 0 test scaffolds (RED baseline for QA-01/02/04)
- [ ] 48-02-PLAN.md — QA-01 source date fix (_format_check_source + _build_filing_date_lookup) + QA-02 coerce_value() boolean fix
- [ ] 48-03-PLAN.md — QA-04 threshold_context in red_flags.html.j2 + Population A deprecation_note backfill
- [ ] 48-04-PLAN.md — QA-05 regression: fresh AAPL + RPM run, baseline threshold update, human review checkpoint
