  <files>src/do_uw/stages/extract/sca_extractor.py, tests/test_sca_extractor.py</files>
  <action>
  Create `sca_extractor.py` following the established extractor pattern (see audit_risk.py, stock_performance.py for reference).

  **Function signature:**
  ```python
  def extract_securities_class_actions(
      state: AnalysisState,
  ) -> tuple[list[CaseDetail], ExtractionReport]:
  ```

  **Data sources (in priority order -- per LOCKED decision: Stanford SCAC primary, 10-K Item 3 supplement):**
  1. EFTS litigation data from `state.acquired_data.litigation_data` -- sec_references (closest proxy for Stanford SCAC; captures SCA data from ACQUIRE stage's LitigationClient)
  2. 10-K Item 3 (Legal Proceedings) -- supplement for company's own characterization of pending cases
  3. Web search results from `state.acquired_data.litigation_data` -- web_results
  4. Blind spot results from `state.acquired_data.blind_spot_results`

  **Extraction logic:**
  1. Parse EFTS sec_references FIRST: check `state.acquired_data.litigation_data` for "sec_references" key. Parse each reference for SCA keywords. For each SCA match, create a CaseDetail with MEDIUM confidence (structured data source).
     - Extract case name if pattern `In re .+ (Securities|Litigation)` matches
     - Extract court if pattern `(S\.D\.N\.Y\.|N\.D\.\s*Cal\.|D\.\s*Del\.|C\.D\.\s*Cal\.)` matches
     - Extract filing date if date pattern `(?:filed|commenced|initiated)\s+(?:on\s+)?(\w+ \d+, \d{4}|\d{1,2}/\d{1,2}/\d{4})` matches
     - Extract settlement amount if `\$[\d,]+(?:\.\d+)?\s*(?:million|billion)` near settlement keywords
     - Classify with two-layer system: coverage_type from CoverageType enum, legal_theories from matched patterns
     - Look up lead counsel against lead_counsel_tiers.json using substring matching: load via ConfigLoader, check if any tier_1 or tier_2 firm name is a substring of the counsel string
  2. Supplement with 10-K Item 3: Get 10-K full text via `get_filing_document_text(state, "10-K")`. Extract Item 3 section via `extract_section()` with the item3 patterns from SECTION_DEFS. Parse for SCA indicators using regex patterns:
     - `r"class\s+action"`, `r"securities\s+(?:fraud|class\s+action|litigation)"`, `r"10b-5|Rule\s+10b-5"`, `r"Section\s+11"`, `r"Section\s+14\(a\)"`
     For each new SCA mention NOT already found via EFTS, create a CaseDetail. For existing matches, enrich with the company's characterization (status, description context).
  3. Cross-reference web search results: check litigation_data.web_results and blind_spot_results for SCA keywords. Add LOW confidence cases not already found.
  4. Deduplicate by case name similarity (fuzzy: if >80% of words match, treat as same case). When merging, prefer EFTS data as primary with Item 3 data enriching.
  5. Set `status` from keywords: "dismissed" -> DISMISSED, "settled" -> SETTLED, "pending" or "active" -> ACTIVE, "appeal" -> APPEAL.
  6. Compute `class_period_days` if both class_period_start and class_period_end are found.

  **ExtractionReport:**
  - expected fields: ["case_name", "case_number", "court", "filing_date", "allegations", "status", "lead_counsel", "coverage_type", "legal_theories", "settlement_amount"]
  - found: whichever fields were actually populated on any case
  - source_filing: "EFTS/SCAC + 10-K Item 3 + web search"

  **Time horizon:** Only include cases from the last 10 years (3650 days). Compare filing_date or mention context date against today.

  Keep file under 500 lines. If approaching limit, extract regex pattern constants to module level.

  **Tests** (`tests/test_sca_extractor.py`):
  - Test EFTS sec_references parsed as primary source -> returns CaseDetail list with MEDIUM confidence
  - Test two-layer classification populates coverage_type and legal_theories
  - Test lead counsel tier lookup matches substring
  - Test deduplication logic (EFTS preferred when merging with Item 3 matches)
  - Test empty EFTS + empty Item 3 returns empty list with low-coverage report
  - Test Item 3 supplement enriches existing EFTS cases with company characterization
  - Test Item 3 adds new cases not found in EFTS
  - Test time horizon filtering
  </action>

  **sec_enforcement.py** -- SEC enforcement pipeline position mapping.

  Function signature:
  ```python
  def extract_sec_enforcement(
      state: AnalysisState,
  ) -> tuple[SECEnforcementPipeline, ExtractionReport]:
  ```

  Logic:
  1. Initialize SECEnforcementPipeline with all defaults.
  2. Search 10-K Item 3 AND Item 1A for enforcement signal patterns (ordered most severe to least):
     - `r"enforcement\s+action|complaint\s+filed|civil\s+penalty|consent\s+decree"` -> ENFORCEMENT_ACTION
     - `r"[Ww]ells\s+[Nn]otice"` -> WELLS_NOTICE
     - `r"formal\s+(?:order\s+of\s+)?investigation|HO-\d+"` -> FORMAL_INVESTIGATION
     - `r"informal\s+(?:inquiry|investigation)|(?:voluntary\s+)?information\s+request"` -> INFORMAL_INQUIRY
     - `r"comment\s+letter|CORRESP|staff\s+comment"` -> COMMENT_LETTER
     Each match creates a SourcedValue[str] pipeline_signal with the matched text context (sentence containing the match, truncated to 500 chars).
  3. Set `highest_confirmed_stage` to the most severe stage found. Use the EnforcementStage enum ordering.
  4. Get CORRESP count from audit_risk extractor data if available in state -- check `state.extracted.financials.audit.sec_comment_letter_count` if populated. Otherwise extract from EFTS references.
  5. Set `comment_letter_count` and parse comment letter topics from EFTS references.
  6. Search for industry sweep indicators: check web search results for patterns like "[industry] SEC sweep", "[industry] SEC enforcement" involving peer companies from state.extracted.financials.peer_group (if available). Set `industry_sweep_detected` as SourcedValue[bool].
  7. Generate `enforcement_narrative`: 2-3 sentence summary of findings.
  8. Cross-reference 8-K filings for enforcement-related disclosures (Item 8.01 keywords).

  ExtractionReport expected: ["highest_stage", "pipeline_signals", "comment_letter_count", "comment_letter_topics", "industry_sweep", "enforcement_narrative", "actions", "aaer_count"]

  **derivative_suits.py** -- Derivative suit and fiduciary claim extraction.

  Function signature:
  ```python
  def extract_derivative_suits(
      state: AnalysisState,
  ) -> tuple[list[CaseDetail], ExtractionReport]:
  ```

  Logic:
  1. Search 10-K Item 3 for derivative suit patterns:
     - `r"derivative\s+(?:action|suit|complaint|claim|litigation)"`
     - `r"[Ss]ection\s+220\s+(?:books?\s+and\s+records?|demand)"`
     - `r"[Cc]aremark|oversight\s+duty|fiduciary\s+duty\s+(?:claim|breach)"`
     - `r"demand\s+(?:refusal|futility|refused|rejected)"`
     - `r"Court\s+of\s+Chancery|Delaware\s+(?:Chancery|court)"`
  2. For each match, create CaseDetail with:
     - coverage_type: DERIVATIVE_SIDE_A or DERIVATIVE_SIDE_B
     - legal_theories: [DERIVATIVE_DUTY] plus any additional theories found
     - status: from context keywords
     - Extract court (often Delaware Court of Chancery)
  3. Cross-reference web search results for derivative suit mentions.
  4. Section 220 demands are tracked separately but flagged as precursors -- add a key_ruling noting "Section 220 books-and-records demand" when detected.
  5. Deduplicate by case name similarity.
  6. Time horizon: 5 years for derivative suits.

  ExtractionReport expected: ["case_name", "court", "filing_date", "allegations", "status", "coverage_type", "section_220_demands"]

  **Tests:**
  - `test_sec_enforcement.py`: Test highest-stage-wins logic, test each stage pattern, test industry sweep detection, test narrative generation, test empty state returns NONE stage.
  - `test_derivative_suits.py`: Test derivative pattern matching, test Section 220 detection, test Caremark claim detection, test coverage type assignment, test empty text returns empty list.

  Keep each file under 500 lines.
