**Part B: Reorder clear signal check in check_evaluators.py**

In both `evaluate_tiered()` and `evaluate_numeric_threshold()`, move the `_check_clear_signal()` call to BEFORE the `try_numeric_compare()` call. This is a defensive reordering: semantically, "is this a known-clear value?" should be checked before attempting numeric parsing.

Current order in evaluate_tiered (lines 146-199):
```python
# Try numeric comparison
numeric_result = try_numeric_compare(data_value, threshold)
if numeric_result is not None:
    ...return numeric result...
# Check for known clear signals in qualitative data
clear_signal = _check_clear_signal(data_value, data_key, check_id)
if clear_signal is not None:
    ...return clear result...
# Qualitative tiered check -- report as INFO
return CheckResult(status=INFO, ...)
```

New order:
```python
# Check for known clear signals FIRST (defensive: prevents numeric parsing
# from accidentally matching qualitative values like 0.0 for wells_notice=False)
check_id = str(check.get("id", ""))
clear_signal = _check_clear_signal(data_value, data_key, check_id)
if clear_signal is not None:
    status, level, evidence = clear_signal
    return CheckResult(status=status, ...)
# Try numeric comparison
numeric_result = try_numeric_compare(data_value, threshold)
if numeric_result is not None:
    ...return numeric result...
# Qualitative tiered check -- report as INFO
return CheckResult(status=INFO, ...)
```

Apply the same reordering to `evaluate_numeric_threshold()` (lines 24-85).

NOTE: The `check_id` variable extraction (line 170 in evaluate_tiered, line 58 in evaluate_numeric_threshold) needs to move up before the clear signal call since it was previously after the numeric comparison.
