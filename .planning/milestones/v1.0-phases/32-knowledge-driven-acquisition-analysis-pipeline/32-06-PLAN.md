<objective>
Build pipeline gap detection infrastructure and check effectiveness tracking.

Purpose: SC-3 (Pipeline Gap Detection) requires that adding a new check automatically surfaces missing pipeline links — which data sources aren't acquired, which extraction fields don't exist, which mappers are missing. SC-6 (Check Effectiveness Measurement) requires tracking which checks always fire, never fire, or have high skip rates. These are the analytical tools that make the brain self-improving.

Output: RequirementsAnalyzer (manifest from check declarations), PipelineGapDetector (gap report), EffectivenessTracker (fire rate analysis from brain_check_runs).
</objective>

1. **Create src/do_uw/knowledge/requirements.py**: Build the RequirementsAnalyzer following the pattern from 32-RESEARCH.md:

   a. `AcquisitionManifest` Pydantic model with fields:
      - `required_sources: set[str]` — source types needed (SEC_10K, MARKET_PRICE, etc.)
      - `required_sections: dict[str, set[str]]` — per-source section requirements
      - `source_to_checks: dict[str, list[str]]` — which checks need each source
      - `checks_by_depth: dict[int, int]` — distribution by depth level
      - `checks_by_content_type: dict[str, int]` — distribution by content type
      - `total_checks: int` — total analyzed

   b. `build_manifest(checks: list[dict[str, Any]]) -> AcquisitionManifest`:
      - Filter to execution_mode == "AUTO" checks
      - For each check, read `required_data` (list of source types) and `data_locations` (dict of source -> sections)
      - Aggregate into the manifest fields
      - Count depth and content_type distributions

2. **Create src/do_uw/knowledge/gap_detector.py**: Build the PipelineGapDetector:

   a. `PipelineGap` Pydantic model: check_id, check_name, gap_type, detail, severity
   b. `GapReport` Pydantic model: total_checks, fully_supported, gaps list, by_type dict, by_severity dict

   c. `ACQUIRED_SOURCES` constant: Introspect from existing ACQUIRE clients or hardcode the known set:
      `{"SEC_10K", "SEC_10Q", "SEC_DEF14A", "SEC_8K", "SEC_FORM4", "SEC_13DG", "MARKET_PRICE", "MARKET_SHORT", "SCAC_SEARCH", "SEC_ENFORCEMENT"}`

   d. `detect_gaps(checks, manifest) -> GapReport`:
      - **Level 1 (SOURCE)**: For each check's required_data, verify the source is in ACQUIRED_SOURCES. Missing = CRITICAL gap.
      - **Level 2 (FIELD)**: For each check, verify it has either a `data_strategy.field_key` OR an entry in FIELD_FOR_CHECK (from check_field_routing.py) OR is handled by a Phase 26+ dedicated mapper. Missing all three = WARNING gap.
      - **Level 3 (MAPPER)**: For checks handled by prefix routing (check_mappers.py), verify the prefix has a registered mapper function. Missing = INFO gap (the mapper chain has fallback behavior).
      - Aggregate gaps by type and severity.

   e. To detect mapper coverage, use prefix-based coverage checking. There is NO MAPPER_REGISTRY — `check_mappers.py` uses if/elif prefix routing in `map_check_data()`. Instead:
      - Define the set of prefixes that have registered handlers: `HANDLED_PREFIXES = {"BIZ", "STOCK", "FIN", "LIT", "GOV", "FWRD", "EXEC", "NLP"}`
      - For each check, extract its prefix (`check_id.split(".")[0]`) and verify it's in HANDLED_PREFIXES
      - Phase 26+ dedicated mappers (EXEC, FIN.TEMPORAL, FIN.FORENSIC, FIN.QUALITY, NLP, FWRD) are checked first via `map_phase26_check()` in the routing chain
      - A check with an unhandled prefix = INFO gap (the mapper chain returns empty dict as fallback)

3. **Tests for requirements.py** (test_requirements.py):

   - Test with empty checks list -> empty manifest

   - Verify WARNING count matches expected (checks without field_key AND without FIELD_FOR_CHECK)
   - Test with synthetic check requiring unknown source -> CRITICAL gap detected

1. **Create src/do_uw/brain/brain_effectiveness.py**: Build the EffectivenessTracker:

   a. `EffectivenessReport` Pydantic model:
      - `total_checks_analyzed: int`
      - `total_runs: int`
      - `always_fire: list[dict]` — checks firing in every run (check_id, name, fire_rate, run_count) — too sensitive?
      - `never_fire: list[dict]` — checks never firing (check_id, name, run_count) — miscalibrated?
      - `high_skip: list[dict]` — checks skipped >50% (check_id, name, skip_rate, run_count) — data gap?
      - `consistent: list[dict]` — checks with stable fire rates (check_id, name, fire_rate, stddev)
      - `confidence_note: str` — "LOW (N<5 runs)" or "MEDIUM (5-20 runs)" or "HIGH (20+ runs)"
      - `by_content_type: dict[str, dict]` — stats grouped by content_type

   b. `compute_effectiveness(conn: duckdb.DuckDBPyConnection, min_runs: int = 1) -> EffectivenessReport`:
      - Query brain_check_runs: `SELECT check_id, status, COUNT(*) FROM brain_check_runs WHERE is_backtest = FALSE GROUP BY check_id, status`
      - For each check, compute: fire_rate (TRIGGERED / total), skip_rate (SKIPPED / total), clear_rate, info_rate
      - Classify: always_fire (fire_rate == 1.0), never_fire (fire_rate == 0.0 AND skip_rate < 1.0), high_skip (skip_rate > 0.5)
      - Determine confidence from total distinct run_ids
      - Return structured report

   c. `update_effectiveness_table(conn: duckdb.DuckDBPyConnection, period: str = "all_time") -> int`:
      - Compute effectiveness metrics from brain_check_runs
      - Upsert rows into brain_effectiveness table (one row per check per period)
      - Set flagged_always_fires, flagged_never_fires, flagged_high_skip, flagged_low_discrimination
      - Return count of rows written

   d. `record_check_run(conn: duckdb.DuckDBPyConnection, run_id: str, check_id: str, check_version: int, status: str, value: str | None, evidence: str | None, ticker: str, is_backtest: bool = False) -> None`:
      - Insert a single row into brain_check_runs
      - Used by the pipeline to record each check result after ANALYZE

   e. `record_check_runs_batch(conn: duckdb.DuckDBPyConnection, rows: list[dict]) -> int`:
      - Batch insert multiple brain_check_runs rows
      - More efficient for post-ANALYZE recording of all 381+ results
