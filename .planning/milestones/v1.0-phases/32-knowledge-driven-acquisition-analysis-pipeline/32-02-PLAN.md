Create `enrichment_data.py` containing the static mapping tables derived from BRAIN-DESIGN.md Section 9. This module holds data only — no logic.

Key data structures:

1. `PREFIX_TO_REPORT_SECTION: dict[str, str]` — maps check ID prefixes to report sections:
   ```python
   PREFIX_TO_REPORT_SECTION = {
       "BIZ": "company",
       "FIN": "financials",
       "GOV": "governance",
       "EXEC": "governance",
       "LIT": "litigation",
       "STOCK": "market",
       "NLP": "disclosure",
       "FWRD": "forward",
   }
   ```

2. `CHECK_TO_RISK_QUESTIONS: dict[str, list[str]]` — maps check_id to risk question IDs (Q1-Q25). Build this from BRAIN-DESIGN.md Section 9 tables. Use prefix-based defaults for checks not explicitly listed:
   - BIZ.CLASS.*, BIZ.SIZE.* -> ["Q1"]
   - BIZ.MODEL.*, BIZ.DEPEND.* -> ["Q2"]
   - BIZ.COMP.* -> ["Q3"]
   - BIZ.UNI.* -> ["Q25"]
   - FIN.LIQ.*, FIN.DEBT.*, FIN.PROFIT.*, FIN.TEMPORAL.*, FIN.SECTOR.* -> ["Q6"]
   - FIN.ACCT.*, FIN.FORENSIC.*, FIN.QUALITY.* -> ["Q7"]
   - FIN.GUIDE.* -> ["Q8"]
   - GOV.BOARD.*, GOV.EFFECT.* -> ["Q9"]
   - GOV.EXEC.*, EXEC.PROFILE.*, EXEC.CEO.*, EXEC.CFO.*, EXEC.PRIOR_LIT.* -> ["Q10"]
   - GOV.INSIDER.*, EXEC.INSIDER.*, STOCK.INSIDER.* -> ["Q11"]
   - GOV.PAY.* -> ["Q12"]
   - GOV.RIGHTS.* -> ["Q13"]
   - GOV.ACTIVIST.* -> ["Q14"]
   - LIT.SCA.*, STOCK.LIT.* -> ["Q4"]
   - LIT.REG.*, LIT.OTHER.* -> ["Q5"]
   - STOCK.PRICE.*, STOCK.PATTERN.* -> ["Q15"]
   - STOCK.SHORT.* -> ["Q16"]
   - STOCK.OWN.* -> ["Q17"]
   - STOCK.ANALYST.*, STOCK.VALUATION.*, STOCK.TRADE.* -> ["Q18"]
   - NLP.RISK.*, NLP.MDA.*, NLP.DISCLOSURE.*, NLP.FILING.*, NLP.WHISTLE.* -> ["Q19"]
   - FWRD.DISC.* -> ["Q23"]
   - FWRD.NARRATIVE.* -> ["Q24"]
   - FWRD.EVENT.* -> ["Q20"]
   - FWRD.WARN.* -> ["Q21"] (except financial warnings which also map to Q6)
   - FWRD.MACRO.* -> ["Q22"]

   Where BRAIN-DESIGN.md specifies a check contributes to MULTIPLE questions, include all (e.g., STOCK.SHORT.position -> ["Q16", "Q6"]).

3. `CHECK_TO_HAZARDS: dict[str, list[str]]` — maps check_id to hazard codes. Build from BRAIN-DESIGN.md:
   - LIT.SCA.* -> ["HAZ-SCA"]
   - LIT.REG.sec_* -> ["HAZ-SEC"]
   - LIT.REG.doj_* -> ["HAZ-DOJ"]
   - FIN.* distress checks -> ["HAZ-BANK"]
   - GOV.INSIDER.*, EXEC.INSIDER.* -> ["HAZ-SCA", "HAZ-SEC"]
   - BIZ.UNI.ai_* -> ["HAZ-AI"]
   - BIZ.UNI.cyber_* -> ["HAZ-CYBER"]
   - Default for most checks: [] (no specific hazard mapping — they're risk characteristics)

4. `CHECK_TO_RISK_FRAMEWORK_LAYER: dict[str, str]` — override mappings for checks that aren't "risk_characteristic":
   - BIZ.CLASS.*, BIZ.SIZE.*, BIZ.MODEL.*, BIZ.COMP.* -> "inherent_risk"
   - LIT.SCA.*, LIT.REG.*, LIT.OTHER.*, FWRD.EVENT.* -> "hazard"
   - Everything else defaults to "risk_characteristic"

5. `CHECK_TO_CHARACTERISTIC: dict[str, tuple[str, str]]` — (direction, strength) for key risk characteristics, from BRAIN-DESIGN.md Section 1 "Risk Characteristics by Strength" table.

The mappings don't need to be 100% perfect for every single check on the first pass. Focus on getting the prefix-based defaults right (covers ~90% of checks) and add specific overrides for the important checks listed in BRAIN-DESIGN.md Section 9. The enrichment can be refined in later iterations.

  <action>
Create `brain_enrich.py` with a function that applies enrichment data to brain_checks rows in DuckDB.

Function: `enrich_brain_checks(conn: duckdb.DuckDBPyConnection) -> dict[str, int]`
Returns counts like {"enriched": 388, "with_questions": 324, "with_hazards": 120, ...}

Logic:
1. Read all check_ids from brain_checks_current
2. For each check, look up enrichment from enrichment_data.py:
   - report_section: PREFIX_TO_REPORT_SECTION[prefix] (derive prefix from check_id.split('.')[0])
   - risk_questions: CHECK_TO_RISK_QUESTIONS.get(check_id) or derive from prefix default
   - hazards: CHECK_TO_HAZARDS.get(check_id, [])
   - risk_framework_layer: CHECK_TO_RISK_FRAMEWORK_LAYER.get(check_id) or derive from prefix
   - characteristic_direction, characteristic_strength: CHECK_TO_CHARACTERISTIC.get(check_id, (None, None))
3. Create version 2 rows for every check that has new enrichment data (append-only — don't update version 1, insert version 2)
4. Auto-populate brain_changelog for each update: change_type="MODIFIED", change_description="Phase 32 enrichment: added risk questions, hazards, report section, framework layer", triggered_by="phase_32_enrichment"

IMPORTANT: Use batch INSERT rather than per-check INSERT for performance (388 rows is small but good practice).
