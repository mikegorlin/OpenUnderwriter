Create `brain_schema.py` with functions to initialize the brain DuckDB database. Use the `duckdb` Python package (already in project dependencies).

Key implementation:
1. `get_brain_db_path() -> Path` — returns `src/do_uw/brain/brain.duckdb` (alongside checks.json)
2. `connect_brain_db(path: Path | None = None) -> duckdb.DuckDBPyConnection` — connects to brain.duckdb, creates if not exists
3. `create_schema(conn: duckdb.DuckDBPyConnection) -> None` — executes all DDL

Tables to create (use the exact SQL from BRAIN-DESIGN.md Section 4):
- `brain_checks` — append-only check definitions (PRIMARY KEY check_id, version)
- `brain_taxonomy` — risk questions, report sections, factors, hazards (PRIMARY KEY entity_type, entity_id, version)
- `brain_backlog` — potential checks not yet implemented (PRIMARY KEY backlog_id)
- `brain_changelog` — auto-generated from version diffs (PRIMARY KEY changelog_id with auto-increment sequence)
- `brain_effectiveness` — fire rates, override tracking (PRIMARY KEY check_id, measurement_period)
- `brain_industry` — industry supplements and thresholds (PRIMARY KEY industry_code, version)
- `brain_check_runs` — per-run check results (PRIMARY KEY run_id, check_id)

Views to create:
- `brain_checks_current` — latest version of each check
- `brain_checks_active` — non-retired current checks
- `brain_taxonomy_current` — latest version of each taxonomy entity

Indexes from BRAIN-DESIGN.md Section 4 "Index Strategy".

IMPORTANT: DuckDB does NOT support auto-increment or SERIAL types. For brain_changelog, use a SEQUENCE:
```sql
CREATE SEQUENCE changelog_seq START 1;
CREATE TABLE brain_changelog (
    changelog_id INTEGER DEFAULT nextval('changelog_seq'),
    ...
);
```

DuckDB does NOT support GENERATED ALWAYS AS ... STORED for computed columns. For brain_effectiveness fire_rate and skip_rate, omit the generated columns — compute them in views or at query time instead.

DuckDB does NOT support FOREIGN KEY constraints. Omit all FOREIGN KEY clauses. Document the logical relationships in comments instead.

DuckDB VARCHAR[] is supported. JSON type is supported. Use these as specified in the schema.

Create `brain_migrate.py` that migrates data FROM brain/checks.json INTO brain.duckdb. This is a one-time migration that can be re-run idempotently (drops and recreates tables).

Function: `migrate_checks_to_brain(conn: duckdb.DuckDBPyConnection | None = None) -> dict[str, int]`
Returns counts: {"checks": 388, "taxonomy_questions": 25, "taxonomy_factors": 10, "taxonomy_hazards": 15, "taxonomy_sections": 7, "backlog": N}

Steps:
1. **Load checks.json** — read all 388 checks from `src/do_uw/brain/checks.json`
2. **Insert into brain_checks** — for each check, create a version 1 row:
   - `check_id` = check["id"]
   - `name` = check["name"]
   - `content_type` = check.get("content_type", "EVALUATIVE_CHECK")
   - `lifecycle_state` = determine from check: if check has defined threshold criteria -> "SCORING"; if check has tiered type with placeholder text or no criteria -> "INVESTIGATION"; all CONTEXT_DISPLAY -> "MONITORING"
   - `depth` = check.get("depth", 2)
   - `execution_mode` = check.get("execution_mode", "AUTO")
   - `report_section` = derive from check prefix (BIZ->company, FIN->financials, GOV/EXEC->governance, LIT->litigation, STOCK->market, NLP->disclosure, FWRD->forward)
   - `risk_questions` = [] (empty initially — Plan 02 enriches these)
   - `risk_framework_layer` = "risk_characteristic" (default; Plan 02 refines)
   - `factors` = check.get("factors", [])
   - `hazards` = [] (empty initially — Plan 02 enriches)
   - `threshold_type` = check.get("threshold_type", "tiered")
   - `threshold_red` = extract RED criterion from check thresholds
   - `threshold_yellow` = extract YELLOW criterion from check thresholds
   - `threshold_clear` = extract CLEAR criterion from check thresholds
   - `question` = check["name"] (placeholder — Plan 02 enriches)
   - `rationale` = check.get("rationale", None)
   - `field_key` = check.get("data_strategy", {}).get("field_key", None) or check.get("field_key", None)
   - `required_data` = check.get("required_data", [])
   - `data_locations` = check.get("data_locations", {})
   - `acquisition_type` = "structured" (default)
   - `industry_scope` = "universal" (default)
   - `created_by` = "migration_v1"
   - `version` = 1

3. **Populate brain_taxonomy** with:
   - 25 risk questions (Q1-Q25) from BRAIN-DESIGN.md Section 9 — entity_type='risk_question', entity_id='Q1' through 'Q25', name=question text, description=section description, domain= derived, weight=null
   - 10 scoring factors (F1-F10) from config/scoring.json — entity_type='factor', entity_id='F1' through 'F10', name=factor name, weight=factor weight
   - 15 hazard codes (HAZ-SCA through HAZ-IP) from BRAIN-DESIGN.md Section 1 hazard taxonomy — entity_type='hazard', entity_id='HAZ-SCA', name='Securities Class Actions', frequency_trend, severity_range
   - 7 report sections — entity_type='report_section', entity_id='company'|'financials'|'governance'|'litigation'|'market'|'disclosure'|'forward'

4. **Seed brain_backlog** with gap items from BRAIN-DESIGN.md Section 8 Migration Phase C:
   - G1: SPAC/De-SPAC detection (5 checks needed)
   - G2: Going concern explicit flag
   - G3: Altman Z-Score
   - G4: AGR Score
   - G5: Revenue fraud pattern taxonomy
   - G6: Derivative risk category assessment
   - G7: AI-specific hazard checks
   Create 7 backlog entries with priority=HIGH, status=OPEN, source='BRAIN-DESIGN.md'

For lifecycle_state derivation: use the content_type already on checks.json. If content_type=="MANAGEMENT_DISPLAY" -> "MONITORING". If check has actual threshold criteria (not just "tiered" with generic text) -> "SCORING". If placeholder checks with no evaluatable criteria -> "INVESTIGATION".
