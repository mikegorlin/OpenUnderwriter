<task type="auto">
  <name>Task 1: Create learning infrastructure for check effectiveness tracking</name>
  <files>
    src/do_uw/knowledge/learning.py
    tests/knowledge/test_learning.py
  </files>
  <action>
    Create `src/do_uw/knowledge/learning.py`:

    NOTE: This module does NOT add new SQLAlchemy models/tables. It uses the existing knowledge store tables (checks, notes) plus simple JSON-encoded metadata on checks for tracking. Analysis run data is stored as notes with a specific tag pattern. This avoids schema changes while providing learning capability.

    1. **AnalysisOutcome** dataclass:
       - ticker: str
       - run_date: datetime
       - checks_fired: list[str] (check IDs that triggered/failed)
       - checks_clear: list[str] (check IDs that passed)
       - quality_score: float
       - tier: str

    2. **CheckEffectiveness** dataclass:
       - check_id: str
       - check_name: str
       - total_runs: int
       - times_fired: int
       - fire_rate: float (times_fired / total_runs)
       - co_firing_partners: list[tuple[str, float]] -- (check_id, co_occurrence_rate)
       - last_fired: datetime | None

    3. **record_analysis_run(store: KnowledgeStore, outcome: AnalysisOutcome) -> None**:
       - Store as a note with tag "analysis_run" and content as JSON
       - Include ticker, run_date, checks_fired IDs, quality_score, tier
       - This provides raw data for effectiveness computation

    4. **get_check_effectiveness(store: KnowledgeStore, check_id: str) -> CheckEffectiveness**:
       - Query all analysis_run notes
       - Count total runs and times this check_id appears in checks_fired
       - Compute fire_rate
       - Find co-firing partners (other checks that fired in same runs)
       - Sort co-firing partners by co-occurrence rate descending

    5. **find_redundant_pairs(store: KnowledgeStore, threshold: float = 0.85) -> list[tuple[str, str, float]]**:
       - Analyze all analysis runs
       - For each pair of checks that co-fire, compute co-occurrence rate
       - Return pairs above threshold, sorted by rate descending
       - Return as list of (check_id_a, check_id_b, co_occurrence_rate)

    6. **get_learning_summary(store: KnowledgeStore) -> dict[str, Any]**:
       - Total analysis runs recorded
       - Top 10 most-fired checks (by fire rate)
       - Top 10 redundant pairs (by co-occurrence)
       - Average quality score across all runs
       - Distribution of tiers across runs

    Create `tests/knowledge/test_learning.py`:
    - Test record_analysis_run stores data correctly
    - Test get_check_effectiveness computes fire rate
    - Test co-firing partners are identified
    - Test find_redundant_pairs with threshold
    - Test get_learning_summary with multiple runs
    - Test effectiveness with zero runs returns zero fire rate
    - Use in-memory SQLite with pre-populated test runs
  </action>

<task type="auto">
  <name>Task 2: Create narrative composition for risk story grouping</name>
  <files>
    src/do_uw/knowledge/narrative.py
    tests/knowledge/test_narrative.py
  </files>
  <action>
    Create `src/do_uw/knowledge/narrative.py`:

    1. **NarrativeStory** dataclass:
       - story_id: str (e.g., "restatement_risk")
       - title: str (e.g., "Restatement Risk Narrative")
       - description: str (what this risk story represents)
       - checks: list[str] (check IDs composing this story)
       - factors_affected: list[str] (scoring factors touched)
       - allegation_types: list[str] (A/B/C/D/E)
       - severity: str (HIGH/MEDIUM/LOW based on worst check)
       - evidence_summary: str (template narrative combining check descriptions)

    2. **NARRATIVE_TEMPLATES**: dict[str, dict] -- Pre-defined risk story templates:
       - "restatement_risk": Aggressive revenue recognition + insider selling + auditor change
       - "event_driven_claim": Stock drop + earnings miss + disclosure gap
       - "governance_failure": Board independence + compensation excess + oversight deficit
       - "regulatory_exposure": SEC pipeline + enforcement history + compliance gaps
       - "financial_distress": Debt covenant risk + liquidity deterioration + going concern signals
       - "insider_trading_pattern": Cluster selling + 10b5-1 plan abuse + timing anomalies
       - "acquisition_risk": M&A activity + premium concerns + integration issues
       - Each template has: title, description, required_check_patterns (regex or check ID prefixes), factors, allegation_types

    3. **compose_narrative(store: KnowledgeStore, fired_checks: list[str]) -> list[NarrativeStory]**:
       - Take list of check IDs that fired in an analysis
       - Match fired checks against NARRATIVE_TEMPLATES
       - A template is "activated" when >= 2 of its required checks fired
       - Build NarrativeStory with the actual checks that matched
       - Generate evidence_summary by combining check descriptions from knowledge store
       - Sort by severity (HIGH first)

    4. **get_available_narratives() -> list[dict[str, Any]]**:
       - Return all NARRATIVE_TEMPLATES with their requirements
       - Useful for understanding what stories the system can compose

    5. **suggest_new_narrative(store: KnowledgeStore, co_firing_data: list[tuple[str, str, float]]) -> list[dict[str, Any]]**:
       - Given co-firing analysis results from learning module
       - Identify clusters of 3+ checks that frequently co-fire
       - Suggest potential new narrative stories
       - Return suggestions with check IDs and co-occurrence rates

    6. **Wire narrative output into CLI** (minimal consumption path):
       - Add a `do-uw knowledge narratives <ticker>` CLI command (in cli_knowledge.py or cli.py knowledge sub-app)
       - Command loads the latest analysis run for the given ticker, extracts fired check IDs, calls compose_narrative, and prints the resulting risk stories as formatted text
       - This provides immediate utility and validates the narrative composition pipeline
       - NOTE: Full RENDER integration (embedding narratives into the Word document) is deferred to Phase 10+ when RENDER is extended. The CLI command serves as the v1 output path.

    Create `tests/knowledge/test_narrative.py`:
    - Test compose_narrative with fired checks matching restatement_risk template
    - Test compose_narrative with no matching template returns empty
    - Test activation threshold (< 2 matching checks = not activated)
    - Test severity ordering (HIGH before MEDIUM)
    - Test get_available_narratives returns all templates
    - Test suggest_new_narrative identifies clusters
    - Use in-memory SQLite with checks matching template patterns
  </action>
