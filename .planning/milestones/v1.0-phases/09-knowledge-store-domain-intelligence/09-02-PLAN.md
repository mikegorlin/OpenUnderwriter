  <action>
    Create `src/do_uw/knowledge/store.py` with KnowledgeStore class (under 500 lines):

    1. **KnowledgeStore** class:
       - `__init__(self, db_path: str | Path | None = None)`: Default path is `src/do_uw/knowledge/knowledge.db`. Creates engine + sessionmaker. Calls `Base.metadata.create_all(engine)` to ensure tables exist.
       - Uses `contextmanager` pattern for session management: `_session()` yields Session, handles commit/rollback.

    2. **Query methods** (all return Pydantic-compatible dicts or lists):
       - `query_checks(section: int | None, status: str | None, factor: str | None, severity: str | None, pillar: str | None, limit: int = 100) -> list[dict[str, Any]]`: Filter checks by any combination of criteria. Return as list of dicts.
       - `get_check(check_id: str) -> dict[str, Any] | None`: Single check by ID.
       - `search_checks(query: str, limit: int = 10) -> list[dict[str, Any]]`: Full-text search across check names and metadata. If FTS5 unavailable, fall back to LIKE queries.
       - `query_patterns(category: str | None = None, allegation_type: str | None = None, status: str | None = None) -> list[dict[str, Any]]`: Filter patterns.
       - `get_scoring_rules(factor_id: str | None = None) -> list[dict[str, Any]]`: Get scoring rules, optionally filtered by factor.
       - `get_red_flags(status: str | None = None) -> list[dict[str, Any]]`: Get red flag gates.
       - `get_sector_baselines(sector_code: str | None = None) -> dict[str, Any]`: Get sector baselines, returns nested dict structure matching sectors.json format.
       - `search_notes(query: str, limit: int = 10) -> list[dict[str, Any]]`: FTS5 search on notes.
       - `add_note(title: str, content: str, tags: str | None, source: str | None, check_id: str | None) -> int`: Insert note, return ID.
       - `get_playbook(playbook_id: str) -> dict[str, Any] | None`: Get industry playbook by ID.
       - `get_playbook_for_sic(sic_code: str) -> dict[str, Any] | None`: Find matching playbook by SIC code range.

    3. **FTS5 detection** at init time:
       - `self._fts_available: bool` -- check via PRAGMA compile_options
       - If FTS5 not available, search methods use LIKE fallback with warning log

    4. **Private helpers**:
       - `_check_to_dict(check: Check) -> dict[str, Any]`: Convert ORM object to dict
       - `_ensure_fts_index()`: Create FTS5 virtual tables if not exists and FTS5 is available

    Create `tests/knowledge/test_store.py`:
    - Test query_checks with various filter combinations
    - Test get_check returns correct check
    - Test search_checks with FTS5 (if available) and LIKE fallback
    - Test query_patterns filtering
    - Test get_scoring_rules filtering
    - Test add_note and search_notes
    - Test get_playbook_for_sic with matching and non-matching codes
    - Use in-memory SQLite with pre-populated test data
  </action>

  <action>
    **Sub-step A: Migration (migrate.py)**

    1. Create `src/do_uw/knowledge/migrate.py` with `migrate_from_json(brain_dir: Path, store: KnowledgeStore) -> MigrationResult`:

       Migration logic for each JSON file:

       **checks.json** (359 checks):
       - Parse each check from the "checks" list
       - Map fields: id, name, section, pillar, severity, execution_mode, threshold_type/threshold_value from check's threshold config, required_data (JSON list), data_locations (JSON dict)
       - Set origin="BRAIN_MIGRATION", status="ACTIVE", created_at=now, modified_at=now
       - Extract scoring_factor and scoring_rule from check's "scoring" field if present
       - Extract output_section from check's "output" field if present

       **scoring.json** (10 factors with rules):
       - Parse each factor from "factors" dict
       - For each rule in factor["rules"], create ScoringRule(id=rule["id"], factor_id=factor_key, condition=rule["condition"], points=rule["points"], triggers_crf=rule.get("triggers_crf"))
       - Store the full scoring config as metadata for reconstruction

       **patterns.json** (19 patterns):
       - Parse each pattern from "patterns" list
       - Map: id, name, category, description, allegation_types (JSON list), trigger_conditions (JSON list), score_impact (JSON dict)

       **red_flags.json** (11 triggers):
       - Parse each trigger from "escalation_triggers" list
       - Map: id, name (from id), condition, detection_logic, max_tier, max_quality_score

       **sectors.json** (sector baselines):
       - Parse each section (short_interest, volatility_90d, leverage_debt_ebitda, claim_base_rates, market_cap_filing_multipliers)
       - For each sector_code in each section, create Sector(sector_code, metric_name=section_name, baseline_value)
       - Store raw JSON as metadata for complex nested structures (percentiles, etc.)

       Return `MigrationResult` dataclass: checks_migrated, patterns_migrated, rules_migrated, flags_migrated, sectors_migrated, errors list.

    **Sub-step B: Backward-Compatible Loader (compat_loader.py)**

    2. Create `src/do_uw/knowledge/compat_loader.py` with `BackwardCompatLoader`:
       - Same interface as ConfigLoader: `load_all() -> BrainConfig`, `load_checks()`, `load_scoring()`, `load_patterns()`, `load_sectors()`, `load_red_flags()`
       - Internally uses KnowledgeStore to query data
       - Reconstructs the exact same dict structure that ConfigLoader returns:
         - `load_checks()` -> returns dict with "checks" list, "total_checks" count, "data_sources" dict, same as checks.json
         - `load_scoring()` -> returns dict with "factors" dict, "tiers" list, "total_points", same as scoring.json
         - `load_patterns()` -> returns dict with "patterns" list, "total_patterns" count, same as patterns.json
         - `load_sectors()` -> returns dict with "short_interest", "volatility_90d", etc., same as sectors.json
         - `load_red_flags()` -> returns dict with "escalation_triggers" list, same as red_flags.json
       - CRITICAL: The reconstruction must produce output that passes the EXACT same validation that ConfigLoader performs. The scoring.json reconstruction is the trickiest because it has nested structure (factors with rules, tiers, probability ceilings).
       - For scoring.json reconstruction: store the full original JSON alongside structured data during migration. BackwardCompatLoader returns the stored original for scoring (hybrid approach: structured for queries, original for backward compat).

    **Sub-step C: Exports and Tests**

    3. Update `src/do_uw/knowledge/__init__.py` exports to include KnowledgeStore, BackwardCompatLoader, migrate_from_json.

    4. Create `tests/knowledge/test_migrate.py`:

    5. Create `tests/knowledge/test_compat_loader.py`:

       - This is the CRITICAL regression test: if BackwardCompatLoader output differs from ConfigLoader output, downstream consumers will break
