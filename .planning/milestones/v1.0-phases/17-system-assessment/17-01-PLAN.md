**Bug diagnosis (confirmed):**
The SEC client's `acquire()` method returns a dict where `result["filing_documents"]` contains the fetched documents. The orchestrator stores this entire dict into `acquired.filings` via `setattr(acquired, "filings", data)`. So filing_documents ends up nested at `acquired.filings["filing_documents"]` instead of `acquired.filing_documents` (the dedicated Pydantic field).

The `get_filing_documents()` in sourced.py has a fallback that checks `filings.get("filing_documents")`, but the data at that path is the raw FilingDocument dicts from filing_fetcher -- they need to be properly routed.

**Fix in orchestrator.py (`_acquire_structured_data` or `_run_client`):**

After the SEC client runs, extract `filing_documents` from the filings dict and set it on `acquired.filing_documents` directly. The cleanest approach:

In `_run_client()`, after `setattr(acquired, target_field, data)`, add a special case: if `target_field == "filings"` and `data` is a dict containing `"filing_documents"`, pop `filing_documents` from data and set `acquired.filing_documents = filing_docs`. This way the data lives at the correct Pydantic field AND is not double-stored in filings.

Also pop `company_facts` from the data dict before setting it on acquired.filings (this supports the state size reduction in Task 2). Store a reference key like `{"_cache_key": "sec:company_facts:{cik}"}` so extractors know how to retrieve it.

**Keep backward compat in sourced.py:**
Keep the fallback in `get_filing_documents()` that checks `filings.get("filing_documents")` -- this handles any cached state.json files from before the fix.

**Tests:** Add a test in `tests/test_filing_documents.py` that:
1. Mocks SEC client to return a dict with filing_documents populated
2. Runs the orchestrator
3. Asserts `acquired.filing_documents` has the expected form types
4. Asserts `get_filing_documents(state)` returns the data
5. Asserts the filings dict does NOT contain filing_documents (no double-store)

**Problem:** Company Facts XBRL blob is 3.7MB for TSLA. It is stored in `acquired.filings["company_facts"]` and serialized into state.json. This is wasteful -- the data is already cached in SQLite by the SEC client.

**Strategy:** Keep company_facts in the filings dict during the pipeline run (extractors need it), but exclude it from serialization. The SEC client already caches it at cache key `sec:company_facts:{cik}`.

Two approaches (pick the simpler one):

**Approach A (Pydantic exclude):** Add a model_post_init or custom serializer on AcquiredData that strips company_facts from the filings dict when serializing to JSON. When deserializing, company_facts would need to be re-fetched from cache on next run.

**Approach B (Pop before serialize, simpler):** In the pipeline's state save path (likely in pipeline.py or cli.py where `state.model_dump_json()` is called), strip `company_facts` from `state.acquired_data.filings` before writing. Add a comment explaining why.

Go with Approach B -- it's simpler and matches how the pipeline already works (write state at session end). Find where state.json is written (likely `cli.py` or `pipeline.py`), and add:

```python
# Remove Company Facts XBRL blob before serialization (~4MB).
# Data is cached in SQLite and re-fetched on next run.
if state.acquired_data and "company_facts" in state.acquired_data.filings:
    del state.acquired_data.filings["company_facts"]
```

Also strip `filing_texts` (legacy Phase 3 data) and `filing_documents` (now stored at dedicated field, don't double-store in filings dict) from the filings dict before serialization. This should bring state.json from 15MB to ~2MB.

**Ensure extractors still work during pipeline run:** Company Facts must be in `filings["company_facts"]` while extractors run (it's only removed at serialization time). Verify the pipeline order: ACQUIRE sets it -> EXTRACT reads it -> RENDER writes state -> clean up before write. This is already the correct order.

**Test:** Verify `get_company_facts(state)` still works during the pipeline run (returns data from filings dict). Write a test that serializes state after the cleanup and verifies company_facts is absent from the JSON.
