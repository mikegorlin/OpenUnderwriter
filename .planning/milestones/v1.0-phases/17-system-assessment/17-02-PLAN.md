**Problem:** The regex in `_extract_name_near_title()` matches any two adjacent capitalized words near a C-suite title. In real proxy text, this captures section headers like "Interim Award", company description fragments like "Space Exploration", and compensation terminology like "Performance Award" as person names.

**The `_NON_NAME_WORDS` blocklist approach is incomplete** -- it's whack-a-mole. For every word you add, another garbage phrase slips through.

**Fix approach -- strengthen `_is_valid_person_name()` in leadership_parsing.py:**

1. **Add a broader blocklist** including common SEC filing terms that get matched as names:
   - Award/Grant/Plan/Program/Agreement terms: "Interim", "Performance", "Annual", "Incentive", "Award", "Grant", "Equity", "Deferred", "Restricted", "Option", "Plan", "Program", "Agreement", "Target", "Threshold", "Maximum"
   - Industry/business terms: "Space", "Exploration", "Technologies", "Holdings", "Industries", "Capital", "Global", "International", "Services", "Solutions", "Systems", "Group", "Energy", "Motors", "Financial"
   - Filing structure terms: "Summary", "Total", "Base", "Fiscal", "Current", "Former", "Independent", "Non", "Lead", "Section", "Item", "Part", "Schedule", "Exhibit"

2. **Add a structural validation check:** A real person name almost always has:
   - First word 2+ chars that is NOT a common English word (check against a small set)
   - Last word 2+ chars
   - Minimum 3 chars per word on average (rejects abbreviation combos)
   - No word is purely numeric
   - At most 4 words (rejects "Chief Executive Officer And Director")

3. **Also add a "name looks like a name" positive signal check:**
   - If ANY word appears in common SEC compensation/filing vocabulary, reject
   - Keep the existing "every word starts uppercase" check

4. **In `extract_executives_from_proxy()`:** After extraction, log how many candidates were rejected by the validation, to track false positive rate.

5. **Update tests in test_leadership_comp.py:**
   - Add test cases with known garbage names that MUST be rejected: "Interim Award", "Performance Award", "Space Exploration", "Annual Incentive", "Restricted Stock", "Total Compensation"
   - Add test cases with known VALID names that MUST pass: "Elon Musk", "Tim Cook", "Jamie Dimon", "Mary Barra", "Robyn Denholm", "Zachary Kirkhorn"
   - Test that the extractor returns empty list when run on compensation table text (which is full of these false matches)

**Problem:** The SCA extractor finds 18 cases for TSLA but 97% of fields are empty. Most cases have only a partial case name from EFTS search results, with no court, no filing date, no class period, no legal theories, no settlement data. These hollow records inflate case counts and make the litigation section appear more populated than it actually is.

**Fix in sca_extractor.py -- add a quality filter function:**

```python
def _is_case_viable(case: CaseDetail) -> bool:
    """Check if a case record has enough detail to be useful.

    A case must have at minimum:
    - A case name (non-empty, non-generic)
    - At least ONE of: court, filing_date, status, class_period_start
    Without these, the record is a fragment from keyword search
    and adds noise rather than signal.
    """
```

Apply this filter after all case construction in `extract_securities_class_actions()`. Cases that fail the filter are:
- NOT included in the returned case list
- Counted in the extraction report as a warning: "Filtered {N} fragment cases with insufficient detail"

**Also fix the extraction report in sca_extractor.py:**
The `found_fields` in the ExtractionReport should count fields that have REAL data, not just "field exists". Currently `create_report()` takes a list of field names. Change the found_fields to only include fields where the actual CaseDetail has populated values. For example, if `case.court` is None, don't count "court" as found.

Build a `_count_populated_fields(case: CaseDetail) -> list[str]` helper that checks which fields on a CaseDetail actually have data (non-None SourcedValue with non-empty value).

**Update the extraction report in sca_extractor.py** to use the populated field counts rather than just the field existence check.

**Dedup logic:** Also add dedup by case_name -- EFTS search may return the same case from different filing references. Normalize case names (lowercase, strip "in re" prefix) before comparing.

**Update tests in test_sca.py:**
- Add a test with a mix of viable and fragment cases, verifying only viable ones are returned
- Add a test that the extraction report warns about filtered fragments
- Add a dedup test (same case name from two sources = one case)
