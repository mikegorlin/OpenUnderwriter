Create a test file that validates extracted state data against ground truth.

**Design:**

```python
"""Ground truth validation tests.

Compares extracted AnalysisState fields against hand-verified
expected values. Reports field-level accuracy.

These tests are parametrized by company and field category.
They run against saved state.json files in output/{TICKER}/.
If no state file exists for a ticker, the test is skipped.

Usage:
    pytest tests/test_ground_truth_validation.py -v
    pytest tests/test_ground_truth_validation.py -k tsla
"""
```

**Test structure:**

1. `load_state(ticker: str) -> AnalysisState | None`: Load from `output/{ticker}/state.json`. Return None if not found.

2. `validate_identity(state, truth)`: Compare identity fields (legal_name, cik, sic_code, sector, exchange). These should be exact matches.

3. `validate_financials(state, truth)`: Compare financial values with tolerance (within 1% for XBRL-sourced data). Use relative tolerance: `abs(actual - expected) / expected < 0.01`.

4. `validate_governance(state, truth)`: Compare governance fields. Mark as expected-to-fail for fields known to be LOW quality with current regex extraction. Use `pytest.mark.xfail` for fields that will only work after LLM extraction.

5. `validate_litigation(state, truth)`: Compare litigation fields (has_active_sca, approximate case count).

6. **Accuracy report function:** After all validations, print a summary: `{ticker}: {passed}/{total} fields match ground truth ({pct}% accuracy)`.

**Parametrize across companies:**
```python
@pytest.mark.parametrize("ticker", ["TSLA", "AAPL", "JPM"])
def test_identity_matches_ground_truth(ticker: str) -> None:
    ...
```

**Key design decisions:**
- Tests are SKIPPED (not failed) if no state file exists -- this lets the test suite pass in CI without requiring pipeline runs
- Financial comparisons use 1% relative tolerance (XBRL rounding)
- Governance/litigation fields expected to fail with current regex are marked `xfail(reason="Requires LLM extraction (Phase 19)")`
- The test file doubles as a quality dashboard -- running it shows exactly which fields are accurate and which aren't
