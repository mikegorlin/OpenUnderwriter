## 1. Overview
Brief statement: "Every check in brain/checks/*.yaml is a self-describing knowledge unit across three orthogonal axes."

## 2. Three-Axis Model
Formally define the 3 axes — these are NON-NEGOTIABLE (locked decisions):

**Axis 1 — work_type** (what the check does):
- `extract` — pulls structured data from filings; no judgment, just retrieval (maps from MANAGEMENT_DISPLAY)
- `evaluate` — applies threshold logic to extracted values; produces rating (maps from EVALUATIVE_CHECK)
- `infer` — combines multiple signals into higher-order finding; dot-connecting (maps from INFERENCE_PATTERN)

**Axis 2 — risk_position** (where in the D&O framework):
- `layer`: hazard | signal | peril_confirming
  - hazard = root cause vulnerability (inherent company characteristic)
  - signal = observable indicator that hazard is activating
  - peril_confirming = direct evidence a peril has materialized
- `factors`: list of F1-F10 scoring factor IDs
- `peril_ids`: list of peril IDs from framework/perils.yaml (empty list [] if unlinked)
- `chain_roles`: dict mapping chain_id → role (trigger/amplifier/mitigator/evidence); {} if unlinked

**Axis 3 — acquisition_tier** (how hard to get the data):
- `L1` — structured XBRL/filing (SEC_10K, SEC_10Q, SEC_8K, XBRL_* in required_data)
- `L2` — filing text items (item_1, item_7, item_9a, etc. — prose extraction required)
- `L3` — web/market data (MARKET_PRICE, NEWS, COURT_RECORDS, WEB_*)
- `L4` — derived (computed from other checks or composites)

## 3. Required Fields
List ALL required fields with type, valid values, and description:
- id (str): Domain prefix drives file assignment. Format: DOMAIN.SUBDOMAIN.name
- name (str): Human-readable check name
- work_type (enum): extract | evaluate | infer
- layer (enum): hazard | signal | peril_confirming
- acquisition_tier (enum): L1 | L2 | L3 | L4
- required_data (list[str]): Data sources needed
- worksheet_section (str): Semantic section name (e.g., "governance", "financial", "litigation")
- display_when (str): "always" | "has_data" | "fired" | "critical_only"
- provenance (dict): Origin metadata — see Section 5

## 4. Optional Fields
List optional fields with when to include:
- factors (list[str]): F1-F10 scoring factor IDs; include if check affects scoring
- peril_ids (list[str]): Peril IDs from perils.yaml; empty list [] if unlinked
- chain_roles (dict): chain_id → role; {} if unlinked
- unlinked (bool): True if chain_roles is {}; set automatically by brain build
- threshold (dict): type + red/yellow/clear bands; required for evaluate checks
- data_locations (dict): Where in filing to find data; key = source, value = list of sections
- execution_mode (str): AUTO | MANUAL; default AUTO
- claims_correlation (float): 0.0–1.0; include when known from empirical data
- tier (int): 1–3 criticality; default 2
- depth (int): 1–3 analysis depth; default 2
- pattern_ref (str): Pattern ID from brain_patterns table; ONLY for work_type: infer
- plaintiff_lenses (list[str]): SHAREHOLDERS | BONDHOLDERS | REGULATORS | EMPLOYEES
- v6_subsection_ids (list[str]): Links to v6 taxonomy subsection IDs
- amplifier (str | null): Amplifier check ID; ONLY for scoring amplifier checks
- amplifier_bonus_points (float): ONLY when amplifier is set
- sector_adjustments (dict): Industry-specific threshold overrides
- extraction_hints (dict): NLP/regex patterns; ONLY for NLP-domain checks
- critical_red_flag (bool): True if this check triggers a score ceiling rule
- data_strategy (dict): field_key + primary_source for pipeline routing

## 5. Provenance Block
Explain the provenance dict structure:
```yaml
provenance:
  origin: migrated_from_json       # migrated_from_json | manual | brain_add | ingest_pipeline
  confidence: inherited             # inherited | high | medium | low
  last_validated: null              # ISO date string or null
  source_url: null                  # URL of article/filing that originated this check
  source_date: null                 # ISO date of source document
  source_author: null               # Author of source article (if applicable)
  added_by: null                    # "brain add" CLI user annotation
```
All migrated checks get: origin=migrated_from_json, confidence=inherited, all others null.
Checks added via `brain add` CLI MUST have source_url and source_date populated.

## 6. File Layout Rules
Explain the subdirectory structure:
- Domain prefix drives file assignment: GOV.* → checks/gov/*.yaml
- 500-line limit (CLAUDE.md anti-context-rot rule) requires gov/ and fwrd/ to use subdirectories
- gov/ subdirs: board.yaml, audit.yaml, exec.yaml, activist.yaml, pay.yaml, rights.yaml, effect.yaml
- fwrd/ subdirs: guidance.yaml, spac.yaml, ma.yaml, transform.yaml (split by ~20 checks each)
- All other domains: single flat file (biz.yaml, fin.yaml, exec.yaml, lit.yaml, stock.yaml, nlp.yaml)
- brain build reads checks/**/*.yaml glob — transparent to all consumers

## 7. Deprecated Fields
List fields from checks.json that are ELIMINATED in unified YAML:
- `pillar` → removed (redundant with work_type + layer)
- `category` → removed (was CONTEXT_DISPLAY/DECISION_DRIVING — proxy for work_type)
- `signal_type` → removed (was STRUCTURAL/PATTERN — redundant after work_type)
- `hazard_or_signal` → replaced by `layer` (kept in DuckDB for backward compat until callers updated)
- `content_type` → replaced by `work_type` (kept in DuckDB for backward compat until callers updated)
- `section` (int) → replaced by `worksheet_section` (str)

## 8. Complete Annotated Example

## 9. Article → Brain Decomposition Guide
Step-by-step workflow for adding new knowledge from an article:
1. Read article: identify the specific claim (not "governance problems" but "boards with >3 insiders have 2.3x higher securities class action rate")
2. Decision: Is this a new check, a threshold update to an existing check, or evidence for an existing chain?
3. New check path: determine work_type (extract/evaluate/infer), layer (hazard/signal/peril_confirming), acquisition_tier (L1/L2/L3/L4)
4. Write YAML entry with provenance.source_url and provenance.source_date populated
5. Determine chain linkage: does this fit an existing causal chain? Update chain_roles.
6. Run `brain build` to validate and load into DuckDB
7. Run `brain validate` to confirm no schema errors
8. Verify: `uv run pytest tests/brain/ -x` passes
