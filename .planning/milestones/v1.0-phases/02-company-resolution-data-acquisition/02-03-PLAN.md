       - Add `search_budget_used: int = Field(default=0, description="Web searches used in this analysis")` -- tracks Brave Search budget consumption

    2. Create `src/do_uw/stages/acquire/orchestrator.py`:
       - `AcquisitionOrchestrator` class:
         - `__init__(self, cache: AnalysisCache | None = None, search_budget: int = 50)`
         - `run(self, state: AnalysisState) -> AcquiredData`: Main entry point. Orchestrates the full acquisition flow:

           **Phase A: Pre-acquisition blind spot sweep (per user decision: before AND after structured)**
           a. Run `WebSearchClient.blind_spot_sweep(company_name, ticker)` with a subset of budget (~20% = 10 searches)
           b. Store results in `acquired_data.blind_spot_results["pre_structured"]`
           c. Log any early signals found (litigation, short seller reports, etc.)

           **Phase B: Structured data acquisition (parallel-ready, but sync for now)**
           d. Run `SECFilingClient.acquire(state, cache)` -> store in `acquired_data.filings`
           e. Run `MarketDataClient.acquire(state, cache)` -> store in `acquired_data.market_data`
           f. Run `LitigationClient.acquire(state, cache)` -> store in `acquired_data.litigation_data`
           g. Run `NewsClient.acquire(state, cache)` -> store in `acquired_data.web_search_results`
           h. For each client: wrap in try/except, log errors, store error info in `acquisition_metadata`, continue to next client (don't halt on individual client failure -- gates determine halt behavior)
           i. Tag each data source with: timestamp, confidence level, fallback tier used (if any)

           **Phase C: Post-acquisition blind spot sweep**
           j. Run a second `blind_spot_sweep` with remaining budget, using knowledge from structured results to target gaps
           k. Store in `acquired_data.blind_spot_results["post_structured"]`

           **Phase D: Gate checking**
           l. Run `check_gates(acquired_data, ACQUISITION_GATES)`
           m. For any HARD gate failure: retry the specific client once after a 2-second delay
           n. Re-check gates after retry
           o. If HARD gate still fails: raise `DataAcquisitionError` with details of what's missing
           p. If SOFT gate fails: log warning, add to gate_results, continue
           q. Store all gate results in `acquired_data.gate_results`

         - Store `search_budget_used` count in acquired_data.
         - Return the populated `AcquiredData`.
       - Keep under 300 lines. If approaching limit, extract retry logic into a helper function.

    IMPORTANT architectural notes:
    - The orchestrator receives `state` which has `state.company` populated by RESOLVE stage (Plan 01).
    - The orchestrator creates and populates `AcquiredData`, which gets assigned to `state.acquired_data` by the AcquireStage.
    - **MCP/Search boundary:** MCP tools (Brave Search, etc.) are NOT available inside Python pipeline code -- they run in Claude's main context only. The pipeline code cannot directly call Brave Search or any MCP tool. The design is:
      (a) `WebSearchClient` has a pluggable `search_fn: Callable[[str], list[dict]]` parameter with a default no-op that returns empty results and logs a warning.
      (b) In tests and standalone pipeline execution, the no-op is used.
      (c) When run through Claude (the actual production usage), the ACQUIRE stage orchestrator in the main context injects the real MCP search function before running the pipeline. This injection happens in `AcquireStage.__init__()` or via a `set_search_fn()` method.
      (d) The orchestrator MUST log clearly when web search returns empty results due to no search function being injected (distinguishing "no results found" from "no search function available").
    - All data stored in AcquiredData is RAW. No parsing, no structuring, no analysis. That is Phase 3.
