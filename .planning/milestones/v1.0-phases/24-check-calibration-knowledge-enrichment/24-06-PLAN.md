  <action>
Create `FixApplicator` class that processes anomalies and either auto-fixes or flags for review.

**CheckFix** Pydantic model:
- check_id: str
- fix_type: str ("factor_reassignment", "threshold_adjustment", "data_mapping_fix", "evidence_improvement")
- description: str
- before_value: str (JSON of the original check field)
- after_value: str (JSON of the fixed check field)
- auto_applied: bool (True if auto-fixed, False if needs review)
- reason: str (why this fix was applied or needs review)

**FixApplicator methods:**

1. `__init__(checks_path: Path | None = None, scoring_path: Path | None = None)` -- load checks.json and scoring.json

2. `audit_factor_mapping() -> dict[str, Any]` -- analyze check-to-factor distribution:
   - Count checks per factor
   - Identify F10 over-mapping: checks in F10 whose section prefix suggests they belong elsewhere (e.g., SECT5.GOV.* checks mapped to F10 should probably map to F9 Governance)
   - Identify F8 under-mapping: financial distress checks (SECT3.FIN.* or SECT3.DIST.*) that map to F3/F7 instead of F8
   - Return report dict with: distribution (factor -> count), proposed_reassignments (list of check_id, from_factor, to_factor, reason)

3. `propose_factor_fixes() -> list[CheckFix]` -- based on audit, generate fixes:
   - SECT5.GOV.* checks mapped to F10 -> reassign to F9 (auto-fix: obvious section-factor mismatch)
   - SECT3.FIN.* and SECT3.DIST.* checks mapped to F3/F7 -> reassign to F8 (review: might be intentional)
   - Checks with empty factors list in financial/governance sections -> propose factor assignment
   - Apply heuristic: check_id prefix to factor mapping rules:
     * SECT1.* -> F1 (Prior Litigation) or context-dependent
     * SECT2.* -> F4 (IPO/SPAC/M&A) or F5 (Guidance)
     * SECT3.FIN.* -> F8 (Financial Distress)
     * SECT3.AUD.* -> F3 (Restatement/Audit)
     * SECT4.MKT.* -> F2 (Stock Decline) or F6 (Short Interest) or F7 (Volatility)
     * SECT5.GOV.* -> F9 (Governance)
     * SECT5.COMP.* -> F10 (Officer Stability)
     * SECT6.* -> F1 (Prior Litigation)

4. `propose_anomaly_fixes(anomalies: list[CalibrationAnomaly]) -> list[CheckFix]` -- process anomalies:
   - ZERO_FIRE BUGs: propose threshold review (flag for manual review, NOT auto-fix -- could be data issue)
   - ALWAYS_FIRE: if threshold is trivially permissive (e.g., ">0" for a always-positive metric), propose tighter threshold. Otherwise flag for review.
   - LOW_EVIDENCE: propose evidence template improvement (manual review)
   - SKIPPED_WITH_DATA: propose mapper fix (flag for review with details of which tickers have data)

5. `apply_fixes(fixes: list[CheckFix], dry_run: bool = True) -> dict[str, Any]` -- if dry_run=True, return summary of what would change. If False, modify checks.json in-place. Only apply auto_applied=True fixes. Return: applied_count, skipped_count, review_needed list.

6. `generate_fix_report(fixes: list[CheckFix]) -> str` -- Markdown report:
   ```
   # Check Fix Report

   ## Auto-Applied Fixes ({count})
   | Check | Type | Before | After | Reason |
   ...

   ## Needs Manual Review ({count})
   | Check | Type | Issue | Suggested Fix |
   ...

   ## Factor Mapping Audit
   ### Current Distribution
   | Factor | Checks | Expected Range |
   ...
   ### Proposed Reassignments
   | Check | From | To | Reason |
   ...
   ```

**Important safety rules:**
- NEVER auto-apply a fix that changes scoring behavior without evidence (e.g., don't auto-adjust thresholds based on gut feel)
- Factor reassignment from F10 -> F9 for SECT5.GOV.* checks IS auto-applicable (clear section-factor mismatch)
- Threshold changes are ALWAYS manual review (could change risk assessment)
- All fixes logged with before/after state
