Create `AnomalyDetector` class that takes CheckAnalyzer output and CalibrationReport to identify calibration anomalies.

**CalibrationAnomaly** Pydantic model (per RESEARCH.md):
- anomaly_type: str (one of: "ZERO_FIRE", "ALWAYS_FIRE", "LOW_EVIDENCE", "TIER_MISMATCH", "SECTION_CONTRADICTION", "SKIPPED_WITH_DATA")
- check_id: str (or "" for tier-level anomalies)
- check_name: str
- description: str (human-readable explanation)
- severity: str ("BUG", "REVIEW", "INFO")
- tickers_affected: list[str]
- suggested_fix: str | None (auto-populated for obvious bugs)

**6 Anomaly Types:**

1. **ZERO_FIRE** (severity: BUG): Checks with 0% fire rate AND skip_rate < 0.5. These are checks that should fire sometimes but never do. Suggested fix: "Check threshold may be too strict, or data mapping broken."

2. **ALWAYS_FIRE** (severity: REVIEW): Checks with 100% fire rate. Some are expected (universally true), but flag for review. Suggested fix: "Threshold may be too permissive. Verify if this is expected universal behavior."

3. **LOW_EVIDENCE** (severity: REVIEW): Checks where evidence_quality is "LOW". These fire but produce generic evidence strings. Suggested fix: "Evidence template needs specificity -- include filing type, date, metric value."

4. **TIER_MISMATCH** (severity: REVIEW): Compare expected_tier (from CalibrationTicker config) to actual_tier (from CalibrationTickerResult). Expected tier is a range like "WIN/WANT" -- actual must fall within that range. If AAPL (expected WIN/WANT) gets WATCH, that's a mismatch. Suggested fix includes both expected and actual.

5. **SECTION_CONTRADICTION** (severity: REVIEW): Detect contradictions between sections. Strategy: compare factor scores. If F8 (Financial Distress) score is near max (8 points, indicating distress) but F1 (Prior Litigation) is 0 for a company like SMCI that has known litigation, flag. Also: if quality_score tier is WIN/WANT but any factor scores at >70% of its max, flag the inconsistency. This is heuristic-based, not perfect.

6. **SKIPPED_WITH_DATA** (severity: BUG): Checks that return SKIPPED but the check's data field path (from check config) has a non-None value in the state. For this, load the full state.json and check if the mapped data field exists. This requires cross-referencing check_mappers.py field paths. Simpler approach: any check that is SKIPPED for some tickers but TRIGGERED/CLEAR for others likely has a data mapping issue for the SKIPPED tickers. Suggested fix: "Check mapper returns None for some tickers but data may be available."

**AnomalyDetector methods:**
1. `__init__(metrics: list[CheckMetrics], report: CalibrationReport, config: list[CalibrationTicker])` -- store all inputs
2. `detect_all() -> list[CalibrationAnomaly]` -- run all 6 detectors, return combined list
3. Individual `_detect_*` private methods for each type
4. `get_anomalies_by_severity(severity: str) -> list[CalibrationAnomaly]` -- filter
5. `get_bug_count() -> int` and `get_review_count() -> int` -- convenience
