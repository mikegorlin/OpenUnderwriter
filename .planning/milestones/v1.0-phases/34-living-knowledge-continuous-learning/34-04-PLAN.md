  <name>Task 1: Build calibration impact simulation and git audit module</name>
  <files>
    src/do_uw/knowledge/calibrate.py
  </files>
  <action>
    Create src/do_uw/knowledge/calibrate.py (~300-400 lines) with:

    1. `get_pending_proposals(conn: DuckDBPyConnection) -> list[ProposalRecord]`:
       - Query brain_proposals WHERE status='PENDING' ORDER BY created_at
       - Return list of ProposalRecord models

    2. `preview_calibration(conn: DuckDBPyConnection, output_dir: Path = Path("output")) -> CalibrationPreview`:
       - Load all pending proposals
       - For each proposal, compute:
         - What's changing (like git diff): field_name, old_value, new_value
         - Which checks are affected
       - Run impact simulation against cached state files (NOT via run_backtest — it
         always loads checks from brain_checks_active with no override parameter):
         - Find all state.json files in output_dir/*/state.json
         - For each state file, load AnalysisState and extract company + extracted data
         - Load current checks via BrainDBLoader.load_checks() once
         - Run execute_checks(current_checks, extracted, company) to get baseline results
         - Apply proposed changes in-memory to the checks list (modify thresholds, add new checks, remove deactivated)
         - Run execute_checks(modified_checks, extracted, company) to get proposed results
         - Diff the two result lists to find what flipped (build BacktestComparison-compatible output)
         NOTE: This avoids the run_backtest limitation by calling execute_checks directly.
       - Return CalibrationPreview with:
         - proposals (list of pending proposals)
         - changes (list of {field, old, new} dicts per proposal)
         - impact (list of {ticker, check_id, current_status, proposed_status} for changed checks)
         - state_files_tested (int)

    3. CalibrationPreview model (BaseModel):
       - proposals: list[ProposalRecord]
       - changes: list[dict[str, Any]] -- [{proposal_id, description, field, old_value, new_value}]
       - impact: list[dict[str, str]] -- [{ticker, check_id, current_status, proposed_status}]
       - state_files_tested: int

    4. `apply_calibration(conn: DuckDBPyConnection, proposal_ids: list[int] | None = None) -> ApplyResult`:
       - If proposal_ids is None, apply ALL pending proposals
       - For each approved proposal:
         a. If NEW_CHECK: promote the INCUBATING check via BrainWriter.promote_check to ACTIVE
         b. If THRESHOLD_CHANGE: update via BrainWriter.update_check with the proposed_changes
         c. If DEACTIVATION: set lifecycle_state='INACTIVE' via BrainWriter.update_check
       - After all changes applied:
         - Export checks.json via BrainWriter.export_json (keeps JSON file in sync)
         - Update proposal status to 'APPLIED' in brain_proposals
         - Mark related feedback as APPLIED via mark_feedback_applied
       - Call _git_commit_calibration with the changed files and a structured summary
       - Return ApplyResult with commit_hash, proposals_applied, checks_modified

    5. `_git_commit_calibration(files_changed: list[str], summary: str, details: str) -> str | None`:
       - Check git status for dirty brain/ files first (warn and abort if unrelated changes exist in brain/)
       - Run `git add` on each file in files_changed (brain/checks.json, brain/brain.duckdb)
       - Run `git commit -m "calibrate: {summary}\n\n{details}"` via subprocess
       - Return commit hash or None on failure
       - All git operations via subprocess.run with check=True wrapped in try/except

    6. ApplyResult model (BaseModel):
       - commit_hash: str | None
       - proposals_applied: int
       - checks_modified: list[str] -- check_ids that were modified
       - feedback_resolved: int -- feedback entries marked APPLIED

    Important: Before applying, verify git working tree is clean for brain/ directory. If dirty, abort with clear message about uncommitted changes. Only stage specific files (brain/checks.json, src/do_uw/brain/brain.duckdb) -- NEVER use `git add -A`.
  </action>
