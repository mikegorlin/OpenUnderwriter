  <action>
Create `docs/` directory at project root. Write `docs/knowledge-system-architecture.md` as a comprehensive presentation document covering:

**1. Knowledge Organization (brain/ and knowledge store)**
- brain/ directory: 5 JSON files (checks.json, patterns.json, red_flags.json, scoring.json, sectors.json) -- flat file predecessor
- Knowledge store: SQLAlchemy ORM in knowledge.db -- the queryable replacement
- ORM models: Check, CheckHistory, Pattern, ScoringRule, RedFlag, Sector, Note, IndustryPlaybook
- BackwardCompatLoader: bridge that auto-migrates brain/ JSON to in-memory store, zero-regression for existing consumers
- Config files in config/: scoring weights, thresholds, industry theories, claim types, etc.
- Relationship: brain/ = source of truth for initial migration; knowledge store = runtime source of truth after migration

**2. Data Flow Through Pipeline**
Walk through exactly how knowledge flows using a concrete check as example (e.g., "Active SCA detected"):
- ACQUIRE: What data sources feed this check (Stanford SCAC via Playwright, 10-K Item 3, CourtListener)
- EXTRACT: How raw data becomes structured (sca_extractor.py -> SecurityClassAction model)
- ANALYZE: How the check executes (check_engine.py evaluates threshold against extracted data)
- SCORE: How check results flow into factor scores (F1_litigation factor, point deductions, red flag gates)
- How compat_loader.py provides checks to the scoring engine

**3. Adding and Improving Checks**
- Check lifecycle: INCUBATING -> DEVELOPING -> ACTIVE -> DEPRECATED
- How to add a check manually (CLI ingest or direct store insertion)
- Required chain for a check to be ACTIVE: required_data, data_locations, scoring_factor, output_section
- Traceability validation: traceability.py validates all links in the chain
- How industry playbooks add checks (playbook_data.py pattern)

**4. Calibration Parameters**
- scoring.json: factor weights, max_points, tier boundaries
- governance_weights.json: 7 governance dimensions
- red_flags.json: gate triggers and score ceilings
- How changes propagate: config -> compat_loader -> ScoreStage
- CheckHistory table records all field changes with who/when/why

**5. Intent Preservation**
- CheckHistory: every modification recorded with reason
- ProvenanceSummary: origin, creation date, version history
- provenance.py: audit trail for who changed what and why
- Deprecation log: deprecated checks preserved with reason, never deleted

**6. System Diagram** (ASCII)
Show the relationship between brain/ -> migrate.py -> knowledge.db -> compat_loader.py -> stages

Read the actual source files to extract accurate details. Reference specific file paths throughout. Use Markdown headers, tables, code examples, and ASCII diagrams. This is a presentation document -- it should be readable by someone unfamiliar with the codebase.
