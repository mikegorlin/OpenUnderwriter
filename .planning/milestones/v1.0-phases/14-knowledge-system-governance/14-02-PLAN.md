Write `docs/knowledge-integration-lifecycle.md` covering the full integration path. Read the actual source files to extract accurate details:

**1. Document Ingestion Pipeline**
- Supported input: .txt and .md files via `do-uw knowledge ingest <filepath> --type <TYPE>`
- 6 document types: SHORT_SELLER_REPORT, CLAIMS_STUDY, UNDERWRITER_NOTES, INDUSTRY_ANALYSIS, REGULATORY_GUIDANCE, GENERAL
- Extraction patterns (from ingestion.py):
  - RISK:/CHECK: prefixed lines -> check_idea items
  - NOTE:/OBSERVATION: prefixed lines -> note items
  - Header bullet extraction (KEY FINDINGS, CONCLUSIONS, etc.)
  - Numbered list extraction (items >10 chars)
- Tag mapping: doc_type -> tag string (e.g., CLAIMS_STUDY -> "claims_study")
- Output: IngestionResult with checks_created and notes_added counts

**2. What Happens to Ingested Items**
- Check ideas become Check ORM objects with:
  - status = INCUBATING
  - section = 0 (unassigned)
  - pillar = INGESTED
  - origin = AI_GENERATED
  - ID format: ING-{timestamp}-{hash}
- Notes become Note objects with title, content, tags, source
- Both stored in knowledge.db via KnowledgeStore

**3. Lifecycle State Machine**
- Full state diagram: INCUBATING -> DEVELOPING -> ACTIVE -> DEPRECATED
- Also: DEVELOPING -> INCUBATING (regression), DEPRECATED -> DEVELOPING (resurrection)
- Valid transitions (from lifecycle.py VALID_TRANSITIONS dict)
- transition_check(): validates transition, updates status, increments version, records in CheckHistory
- record_field_change(): tracks non-status field changes

**4. Graduation Criteria (INCUBATING -> DEVELOPING -> ACTIVE)**
Document what must be true at each stage:
- INCUBATING -> DEVELOPING: Human has reviewed the check idea, assigned a section, defined required_data and data_locations
- DEVELOPING -> ACTIVE: All traceability links validated (required_data exists, data_locations map to state, scoring_factor assigned, output_section mapped), threshold defined, execution tested
- Reference traceability.py validation chain

**5. Worked Example: Sidley Biotech Study**
Walk through the specific example mentioned in the phase goal:
- Input: Sidley Austin biotech SCA study (claims study document)
- Ingestion: `do-uw knowledge ingest sidley_biotech.md --type CLAIMS_STUDY`
- Result: 24 check ideas (RISK:/CHECK: lines) + 21 notes (NOTE: lines and bullet items)
- Current state: All 24 checks are INCUBATING in knowledge store
- What graduation looks like: Pick one check, show the full path from INCUBATING -> assign section/data -> DEVELOPING -> validate traceability -> ACTIVE
- Show the CLI commands that would be used (even if governance CLI is built in plan 14-03)

**6. Human Review Points**
Explicitly identify where human judgment is required:
- Reviewing ingested check ideas for validity
- Assigning section and pillar
- Defining threshold_type and threshold_value
- Mapping to scoring_factor
- Validating against existing checks for redundancy (learning.py find_redundant_pairs)
- Final approval to transition to ACTIVE

**7. Stage Activation**
Which pipeline stages consume knowledge store data and when:
- ANALYZE (check_engine.py): Executes checks with status=ACTIVE from compat_loader
- SCORE (factor_scoring.py): Uses scoring rules from compat_loader
- SCORE (pattern_detection.py): Uses patterns from compat_loader
- BENCHMARK (market_position.py): Uses sector baselines
- Industry playbooks: Auto-activated in RESOLVE based on SIC, adds industry checks

Use concrete file paths, code references, and the worked example throughout. This is an operational guide for someone who will be managing the knowledge lifecycle.
