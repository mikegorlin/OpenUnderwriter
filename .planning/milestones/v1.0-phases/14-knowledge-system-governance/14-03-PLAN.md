  <files>src/do_uw/cli_knowledge_governance.py, src/do_uw/cli.py</files>
  <action>
Create `src/do_uw/cli_knowledge_governance.py` as a new Typer sub-app following the pattern in cli_knowledge.py. Register it as a sub-app of knowledge_app (the existing knowledge Typer app in cli_knowledge.py), NOT a sibling of knowledge_app. Registration path: `knowledge_app.add_typer(governance_app, name="govern")` so commands are invoked as `do-uw knowledge govern <command>`.

Why a new file: cli_knowledge.py is 370 lines. Adding governance commands would push it over 500 lines. New file keeps both under 500.

The governance_app Typer sub-app must include these commands:

**1. `do-uw knowledge govern review` -- Review pending checks**
- List all checks with status=INCUBATING (default) or a specified --status
- Display as Rich table with columns: ID, Name, Origin, Created, Section, Pillar
- Option: --status {INCUBATING,DEVELOPING,ACTIVE,DEPRECATED} to filter
- Option: --limit (default 50)
- Import KnowledgeStore, use store.query_checks(status=status)

**2. `do-uw knowledge govern promote <check_id> <to_status>` -- Promote/demote a check**
- Arguments: check_id (str), to_status (str -- one of INCUBATING, DEVELOPING, ACTIVE, DEPRECATED)
- Option: --reason (required for DEPRECATED, optional for others)
- Uses lifecycle.transition_check() with changed_by="cli"
- On invalid transition, show error with valid transitions for current status
- On success, show the check's new status and version
- Needs store.get_session() for the session parameter
- Validate to_status is a valid CheckStatus value before calling transition_check

**3. `do-uw knowledge govern history <check_id>` -- View check modification history**
- Uses provenance.get_provenance_summary() to get full history
- Displays header: check name, origin, created_at, current version, total modifications
- Displays Rich table of changes: Version, Field, Old Value, New Value, Changed By, Date, Reason
- Option: --field to filter history entries by field name (e.g., --field status)

**4. `do-uw knowledge govern drift` -- View calibration drift**
- Compares current scoring config (scoring.json via config/loader.py or compat_loader) with knowledge store scoring rules
- For each scoring factor: show max_points and weight from config vs store
- Flag differences as "DRIFT" in yellow
- This shows whether the flat-file config and knowledge store have diverged
- Implementation: load scoring.json via Path(__file__).parent / "config" / "scoring.json", compare with store.get_scoring_rules()
- Display as Rich table: Factor ID, Config Weight, Store Weight, Config Max Points, Store Max Points, Status (OK/DRIFT)

**5. `do-uw knowledge govern deprecation-log` -- View deprecated checks**
- Uses provenance.get_deprecation_log()
- Displays Rich table: Check ID, Name, Deprecated At, Deprecated By, Reason

Registration in cli.py:
- In cli.py, add import: `from do_uw.cli_knowledge_governance import governance_app`
- Note: the registration should be on knowledge_app, not the main app. So in cli_knowledge.py, add the import and registration: `from do_uw.cli_knowledge_governance import governance_app` then `knowledge_app.add_typer(governance_app, name="govern")`
- Actually, to avoid circular imports, do the registration in cli_knowledge.py at module level, since cli_knowledge.py already defines knowledge_app

Implementation notes:
- Use `from __future__ import annotations` at top
- Lazy imports for KnowledgeStore and lifecycle/provenance functions inside command functions (follows existing pattern)
- Rich Console for output, Rich Table for tabular data
- Pyright strict compliance: type all functions, use cast() where needed
- Follow existing B008 ruff ignore pattern if Typer defaults need function calls (add to ruff.toml)
