  <action>
    Create `docs/design-decisions.md` as a comprehensive design decision record. The docs/ directory already exists (Phase 14 created it for architecture docs).

    Organize decisions into these categories, drawing from STATE.md's Decisions section and the CLAUDE.md rules:

    **1. Architecture Decisions**
    - Single Pydantic AnalysisState (not multiple state files) -- learned from predecessor's 7+ state representations
    - 7-stage pipeline (RESOLVE -> ACQUIRE -> EXTRACT -> ANALYZE -> SCORE -> BENCHMARK -> RENDER)
    - MCP boundary: MCP tools only in ACQUIRE stage, subagents cannot access
    - Sync rate limiter (time.sleep + Lock), not async
    - SEC REST API primary, MCP integration deferred
    - SQLAlchemy 2.0 + Alembic + SQLite FTS5 for knowledge store
    - No source file over 500 lines (anti-context-rot)
    - Config-driven thresholds (JSON files in config/, never hardcoded)
    - Pipeline config passthrough: dict flows CLI -> Pipeline -> stages
    - Full-document-once: ACQUIRE downloads, EXTRACT parses locally

    **2. Visual Design & Branding**
    - #1A1446 navy + #FFD000 gold color scheme
    - NO green in risk spectrum (nothing is "safe" in underwriting)
    - Risk heat: red (deteriorating), blue (improving), amber (caution)
    - Typography: Georgia headings, Calibri body, Consolas citations
    - Conditional formatting: red >= 10% deteriorating, amber 1-10%, blue >= 10% improving
    - Stock chart thresholds: single-day >= 8% red triangles, multi-day >= 15% orange bands

    **3. Data Integrity**
    - Every data point: source + confidence (HIGH/MEDIUM/LOW)
    - Never generate/guess financial data -- "Not Available" instead
    - Web-sourced data requires 2+ independent sources
    - Blind spot detection: broad web search is first-class, not fallback
    - SourcedValue[T] pattern for typed values with provenance

    **4. Scoring & Analysis**
    - 10-factor composite scoring model with config-driven weights
    - 11 Critical Red Flag gates with score ceilings
    - 17 composite patterns (EVENT_COLLAPSE, DEATH_SPIRAL, etc.)
    - Theory-to-factor mapping: A=[F1,F3,F5], B=[F2,F5], C=[F7,F8], D=[F9,F10], E=[F4]
    - Tier classification: WIN/WANT/WRITE/WATCH/WALK/NO_TOUCH
    - Multiplicative inherent risk: base_rate * cap_multiplier * score_multiplier
    - Band-based risk ordering (not strict pairwise)

    **5. Rendering**
    - Word (.docx) as primary output, PDF and Markdown as secondary
    - Section renderer dispatch via importlib with None fallback
    - Radar chart uses risk fractions (0-1 scale)
    - Meeting prep questions: 4 categories (CLARIFICATION, FORWARD_INDICATOR, GAP_FILLER, CREDIBILITY_TEST)
    - PDF optional (WeasyPrint), Markdown via Jinja2

    **6. Dashboard**
    - FastAPI + htmx + Plotly.js + DaisyUI CDN (no Node.js/build step)
    - Read-only view of AnalysisState, hot-reloads on file mtime change
    - Chart API: GET /api/chart/{name} -> JSON for client-side Plotly rendering
    - empty_figure() for missing data (not broken axes)

    **7. Knowledge System**
    - Check lifecycle: INCUBATING -> DEVELOPING -> ACTIVE -> DEPRECATED
    - BackwardCompatLoader as drop-in ConfigLoader replacement
    - Industry playbooks auto-activated by SIC/NAICS (10 verticals)
    - PricingStore shares SQLite DB with KnowledgeStore
    - Document ingestion: rule-based extraction v1 with pluggable extraction_fn

    **8. Actuarial Pricing**
    - ILF power curve for layer pricing
    - Credibility weighting for market calibration
    - All parameters in actuarial.json, none hardcoded
    - Model output labeled as indicated, not prescriptive

    **9. AI Risk Factor**
    - Separate from 10-factor composite (independent dimension)
    - Threat-level baseline prior: HIGH=7, MEDIUM=5, LOW=3
    - Industry-specific impact models
    - Patent + disclosure + sentiment multi-signal assessment

    **10. Pyright Strict Compliance Patterns**
    - yfinance: `# type: ignore[import-untyped]` + cast()
    - python-docx: All doc/styles/paragraphs typed as Any
    - Plotly: `import plotly.graph_objects as go  # type: ignore[import-untyped]`, figures typed as Any
    - FastAPI: `_templates: Any = templates` for Jinja2 env
    - Lambdas in dataclass fields get Unknown type -- use closures instead
    - lambda: [] for Pydantic list default_factory

    For each decision, include:
    - **Decision**: What was decided
    - **Rationale**: Why this choice
    - **Alternatives rejected**: What else was considered (where applicable)
    - **Phase**: Which phase made this decision

    Format as a clean, scannable Markdown document with category headers, decision sub-headers, and concise entries. Target ~200-300 lines -- comprehensive but not verbose. Not every single micro-decision from STATE.md needs inclusion; focus on decisions that a future developer would need to understand.
