# Phase 10.1: Pricing Model Enhancement — Context

## Problem Statement

The Phase 10 pricing database was built with a simplified model that assumes complete quote data. In reality, D&O insurance market intelligence arrives as fragments — an anniversary date here, one layer's pricing there, a broker name from a conversation. The model needs to reflect how data actually arrives and accumulates over time.

## Decisions (LOCKED — honor these exactly)

### D1: Tower Structure Semantics

D&O towers are built like bricks stacking upward:

1. **Primary layer**: Has a retention (SIR/deductible), a limit, and a premium. Only the primary has a retention.
   - Example: Primary $10M limit, $2.5M retention, $1,000,000 premium
   - Rate = Premium / Limit = 10%
   - Rate per million = Premium / (Limit / $1M) = $100,000

2. **Excess layers**: Stack above the primary. Named as 1st Excess, 2nd Excess, 3rd Excess, etc.
   - Notation: "10xs10" = $10M excess of $10M (limit xs attachment)
   - Each has a limit, premium, and carrier. No retention.
   - Rate = Premium / Limit
   - Rate on Line (ROL) can mean two things:
     - **Relative ROL**: Layer rate / primary rate (e.g., 5% / 10% = 50%)
     - **Absolute ROL**: Premium / Limit (same as rate)
   - Store raw numbers, compute either definition as needed

3. **Side A layers**: Sit ON TOP of the ABC tower, not parallel to it.
   - Example: "10xs100 Lead Side A" = $10M Side A excess of $100M ABC tower
   - Side A = individual director/officer protection only (no entity coverage)
   - ABC = Side A + Side B (company reimbursement) + Side C (entity securities)

4. **Ventilated carriers**: Same carrier can appear on multiple layers (e.g., Hartford on 1st Excess AND Lead Side A). This is normal and important to capture.

5. **Layer sizes**: Typically $5M and $10M increments. $1M and $2.5M possible but rare. Range: $1M to $25M+ per layer.

6. **Commission**: Primary usually has commission; can be net 0%. Track as optional field.

### D2: Minimum Viable Record

**Ticker + anniversary date** is the absolute minimum worth storing. Everything else is optional and adds value incrementally:
- Anniversary date (when the program incepts/renews)
- Broker/brokerage
- Total limits purchased
- Individual layer pricing
- Carrier information
- Terms and conditions

### D3: Broker as Full Contact Card

Capture full broker/brokerage details:
- Brokerage firm name (Marsh, Aon, WTW, etc.)
- Producer/broker name
- Email
- Phone

This turns the pricing database into a quasi-CRM for D&O market contacts.

### D4: Carrier Information Per Layer

Track per carrier:
- Carrier name
- AM Best rating
- Appetite notes (growing/shrinking in this space, relationship context)

### D5: One Carrier Per Layer (Norm)

One carrier per layer is the standard. Quota share exists but is uncommon. Support quota share as an optional field, not a first-class modeling concept.

### D6: AI-Assisted Document Ingestion

Users will upload diverse artifacts:
- **Excel tower spreadsheets** — varied formats, every broker different, can be top-to-bottom or bottom-to-top
- **PDF program summaries** — broker-produced 1-2 page program overviews
- **Quota share binders** — full binder documents with coverage terms, schedules, endorsements
- **Policy documents** — actual policy forms

Use LLM/AI to parse these artifacts and extract structured pricing data (tower structure, layers, carriers, premiums, terms). This is a meaningful parsing challenge but well-suited to LLM capabilities.

### D7: Inference with Transparency

When partial tower data exists:
- **Infer** missing layer pricing from known rate relationships (e.g., if you know primary and 1st excess, you can estimate 2nd excess)
- **Mark inferred values clearly** — INFERRED vs. VERIFIED source attribution
- Never present inferred data as if it were real market data

### D8: Year-over-Year Tracking

Track how programs evolve:
- Carrier rotations (who moved in/out of which layers)
- Pricing movement (hardening/softening per layer and per program)
- Broker changes
- Limit/retention changes
- Goal: See directionality around renewals and carrier behavior

### D9: Coverage Scope

D&O liability (ABC + Side A) is the focus for Phase 10.1. Architecture should allow future expansion to other management liability lines (EPL, Fiduciary, Crime) but do not build them now.

### D10: Ultimate Analytics Goal

The pricing database should eventually answer:
- What the **market** is pricing for this type of risk (from accumulated data)
- What the risk **should** cost (actuarial estimate from Phase 12)
- The **delta** between market and actuarial — this is where underwriting value lives

## Claude's Discretion (implementation choices)

### CD1: Schema Design
How to restructure the ORM models. Likely need a `Program` entity above `Quote` to represent a company's D&O program across years. Policy year as a grouping concept. Must support the minimum viable record (just ticker + anniversary).

### CD2: Migration Strategy
How to migrate existing Phase 10 data to the enhanced schema without breaking existing functionality. Alembic migration approach.

### CD3: Document Parsing Pipeline
Architecture for AI-assisted document ingestion. How to handle varied Excel formats, PDF extraction, and structured output. Integration with existing knowledge store ingestion infrastructure (Phase 9).

### CD4: Inference Engine Design
How to implement rate inference from partial tower data. What rate relationships to use, confidence levels for inferred values, when inference is appropriate vs. when data is too sparse.

### CD5: Analytics Adaptation
How to update MarketPositionEngine and trend detection to work meaningfully with partial data, inferred values, and the richer program/year/layer hierarchy.

## Deferred Ideas (out of scope for 10.1)

- **Other management liability lines** (EPL, Fiduciary, Crime) — future phase
- **Actuarial pricing model** — Phase 12
- **Interactive dashboard for pricing data** — Phase 11 may include this
- **Automated competitive intelligence** — scraping market data from external sources
- **Carrier relationship management** — full CRM features beyond basic contact storage
