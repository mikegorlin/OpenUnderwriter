# Phase 10.1: Pricing Model Enhancement - Research

**Researched:** 2026-02-09
**Domain:** Insurance pricing schema evolution, document ingestion, LLM-based extraction
**Confidence:** HIGH (schema design, migration), MEDIUM (document ingestion, inference engine)

## Summary

Phase 10.1 transforms the Phase 10 pricing database from a "complete quote required" model to a "partial data as the norm" model that reflects how D&O insurance market intelligence actually arrives. The core challenge is schema evolution: adding Program as a grouping entity above Quote, restructuring towers to match real D&O semantics (Primary + Excess + Side A layers), adding broker/carrier contact tracking, and making nearly every field optional except ticker + anniversary date.

The document ingestion pipeline is the most technically ambitious part: parsing varied Excel tower spreadsheets, PDF program summaries, and policy documents using LLM-based extraction (via the `instructor` library with Anthropic Claude) to produce structured Pydantic models. This builds on Phase 9's ingestion infrastructure but targets a completely different output domain (pricing vs. knowledge store checks).

The migration strategy must preserve all existing Phase 10 data while adding new tables and making existing required columns nullable. SQLite's ALTER TABLE limitations require Alembic's batch mode for column modifications but standard `add_column` works for new nullable columns.

**Primary recommendation:** Design the schema around a Program -> PolicyYear -> TowerLayer hierarchy where Program is the minimum viable record (ticker + anniversary), PolicyYear groups a specific renewal's tower/pricing/broker data, and TowerLayer stores individual layer details. Make all pricing/carrier/broker fields nullable. Use Alembic migration 003 with batch mode for existing table changes and standard DDL for new tables.

### Pattern 1: Minimum Viable Record (D2)
**What:** A Program record can exist with just ticker + anniversary_date. All other fields are nullable and accumulate over time.
**When to use:** Always -- this is the fundamental design principle.
**Example:**
```python
# Minimum record: just learned about a company's D&O program
class Program(Base):
    __tablename__ = "programs"
    id: Mapped[int] = mapped_column(Integer, primary_key=True, autoincrement=True)
    ticker: Mapped[str] = mapped_column(String, nullable=False, index=True)
    company_name: Mapped[str | None] = mapped_column(String, nullable=True)  # optional!
    anniversary_month: Mapped[int | None] = mapped_column(Integer, nullable=True)
    anniversary_day: Mapped[int | None] = mapped_column(Integer, nullable=True)
    broker_id: Mapped[int | None] = mapped_column(Integer, ForeignKey("brokers.id"), nullable=True)
    created_at: Mapped[datetime] = mapped_column(DateTime, nullable=False)
    updated_at: Mapped[datetime] = mapped_column(DateTime, nullable=False)
```

### Pattern 2: Data Completeness Tracking
**What:** Every PolicyYear tracks its data completeness level so analytics can weight appropriately.
**When to use:** When computing market statistics, filter or weight by completeness.
**Example:**
```python
class DataCompleteness(StrEnum):
    """How much of the policy year data we have."""
    FRAGMENT = "FRAGMENT"    # Just anniversary or one field
    PARTIAL = "PARTIAL"      # Some layers/pricing but gaps
    COMPLETE = "COMPLETE"    # Full tower structure with all premiums
```

### Pattern 3: Source Attribution Per Value (D7)
**What:** Track whether each data point is verified (from document), inferred (from rate relationships), or estimated (rough guess).
**When to use:** For layer pricing when partial tower data exists.
**Example:**
```python
class DataSource(StrEnum):
    """Provenance of a data value."""
    VERIFIED = "VERIFIED"      # From document or manual entry
    INFERRED = "INFERRED"      # Computed from known rate relationships
    ESTIMATED = "ESTIMATED"    # Rough estimate, low confidence
```

### Pattern 4: Side A On Top Of ABC (D1)
**What:** Side A layers sit on top of the ABC tower, not parallel. Model with layer_type enum.
**When to use:** When representing full D&O tower structures.
**Example:**
```python
class LayerType(StrEnum):
    """Type of insurance layer in the tower."""
    PRIMARY = "PRIMARY"      # Has retention, first layer
    EXCESS = "EXCESS"        # ABC excess layers (1st, 2nd, 3rd...)
    SIDE_A = "SIDE_A"        # Side A layers sitting on top of ABC tower
```

### Pattern 5: Backward-Compatible Quote Wrapper
**What:** Keep existing QuoteInput/QuoteOutput working by mapping to new schema internally.
**When to use:** During migration period. Existing CLI commands and CSV import continue to work.
**Example:** `add_quote()` creates a Program (if not exists) + PolicyYear + TowerLayers internally.

### Anti-Patterns to Avoid
- **Everything required:** The old model required total_limit, total_premium, company_name, etc. The new model must make nearly everything nullable.
- **Flat quote model:** Don't keep a single Quote table for everything. Separate into Program/PolicyYear/Layer hierarchy.
- **Carrier embedded in layer:** Don't store carrier details on TowerLayer. Use a separate Carrier table with FK so the same carrier can appear on multiple layers (D4, D5).
- **Parallel Side A:** Don't model Side A layers as a separate parallel tower. They sit on top of the ABC tower per D1.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Excel parsing | Custom cell-by-cell reader | openpyxl + LLM extraction | Every broker's spreadsheet is different; LLM handles format variation |
| PDF text extraction | regex on raw text | pdfplumber for tables, LLM for structure | PDFs have complex layouts; pdfplumber handles table detection |
| Structured LLM output | Manual JSON parsing + retries | instructor library | Handles Pydantic validation, retries on parse failure, streaming |
| Rate inference | Ad-hoc formulas | Configurable rate decay curves from config/ | Rate relationships (excess vs primary) follow known patterns |
| Schema migration | DROP + recreate | Alembic batch mode migration | Preserves existing data |

**Key insight:** The document parsing challenge is fundamentally an LLM task. Every broker formats their spreadsheets differently, PDF layouts vary wildly, and policy documents use inconsistent terminology. Rule-based parsing will never cover the variance. Use LLM extraction with Pydantic output schemas.

## Common Pitfalls

### Pitfall 1: SQLite ALTER TABLE Limitations
**What goes wrong:** SQLite cannot ALTER TABLE to change column types, add NOT NULL constraints, or drop columns.
**Why it happens:** SQLite only supports ADD COLUMN (with nullable or default) for ALTER TABLE.
**How to avoid:** Use Alembic's `batch_alter_table` for any column modifications. For new nullable columns on existing tables, standard `op.add_column()` works fine.
**Warning signs:** Migration fails with "Cannot add a NOT NULL column with default value NULL".

### Pitfall 2: Breaking Existing QuoteInput API
**What goes wrong:** Changing required fields to optional in QuoteInput breaks all existing CLI commands and CSV import.
**Why it happens:** Pydantic validation will reject data that previously worked.
**How to avoid:** Create new ProgramInput/PolicyYearInput models alongside existing QuoteInput. Keep QuoteInput working as a convenience wrapper that creates Program+PolicyYear+Layers internally. Phase out gradually.
**Warning signs:** Existing tests for add-quote and import-csv start failing.

### Pitfall 3: Over-Indexing on Complete Data
**What goes wrong:** Analytics engine requires complete pricing data, returning INSUFFICIENT for any partial records.
**Why it happens:** Current `get_rates_for_segment()` assumes every quote has a valid program_rate_on_line.
**How to avoid:** Update analytics to work with whatever data is available. A PolicyYear with only 2 known layers can still contribute layer-level rate data even if total program ROL is unknown.
**Warning signs:** Market position always shows "INSUFFICIENT DATA" even after adding many partial records.

### Pitfall 4: LLM Extraction Without Guardrails
**What goes wrong:** LLM hallucinates plausible-looking pricing data that gets stored as VERIFIED.
**Why it happens:** LLMs are good at generating realistic insurance pricing numbers.
**How to avoid:** Always mark LLM-extracted data as `INFERRED` or `AI_EXTRACTED` source. Require human review before changing to VERIFIED. Include confidence scores from the LLM.
**Warning signs:** Suspiciously complete data appearing from document ingestion.

### Pitfall 5: Migration 003 Ordering
**What goes wrong:** New tables with FKs to existing tables, or existing tables needing new FKs to new tables.
**Why it happens:** Alembic executes DDL sequentially; FK targets must exist before FK creation.
**How to avoid:** Create new standalone tables first (brokers, carriers), then add FK columns to existing/new tables. Use batch_alter_table for existing table modifications.
**Warning signs:** "no such table" errors during migration.

### Pitfall 6: File Line Count Violations
**What goes wrong:** pricing_store.py (464L) and pricing_analytics.py (435L) are already close to the 500-line limit. Adding new functionality pushes them over.
**Why it happens:** Natural code growth without splitting.
**How to avoid:** Plan the split early. pricing_store.py should split into pricing_store.py (core CRUD) and pricing_store_programs.py (program/policy_year operations). pricing_analytics.py should split into pricing_analytics.py (market position) and pricing_analytics_trends.py (trend/YoY analysis).
**Warning signs:** Any file approaching 450 lines needs a split plan before adding code.

### Example 4: Excel Tower Spreadsheet Parsing
```python
# Source: openpyxl docs (https://openpyxl.readthedocs.io/en/stable/)
from openpyxl import load_workbook

def extract_tower_from_excel(filepath: Path) -> str:
    """Read Excel file and convert to text for LLM extraction.

    Rather than trying to parse the varied formats programmatically,
    we convert to a text representation and let the LLM handle
    format interpretation.
    """
    wb = load_workbook(filepath, read_only=True, data_only=True)
    text_parts: list[str] = []

    for sheet_name in wb.sheetnames:
        ws = wb[sheet_name]
        text_parts.append(f"=== Sheet: {sheet_name} ===")
        for row in ws.iter_rows(values_only=True):
            cells = [str(c) if c is not None else "" for c in row]
            if any(c.strip() for c in cells):
                text_parts.append("\t".join(cells))

    return "\n".join(text_parts)
```

### Example 5: Rate Inference from Partial Tower
```python
# Source: Project-specific domain knowledge from D1 decision
def infer_layer_rate(
    primary_rate: float,
    layer_number: int,
    layer_type: str,
    decay_config: dict[str, float],
) -> tuple[float, str]:
    """Infer a layer's rate from the primary rate and position.

    Returns (inferred_rate, confidence_note).

    D&O tower pricing follows a decay curve:
    - 1st Excess is typically 40-60% of primary rate
    - 2nd Excess is typically 30-45% of primary rate
    - Higher layers continue to decrease
    - Side A layers have their own pricing dynamics
    """
    if layer_type == "SIDE_A":
        # Side A has different dynamics, lower confidence
        factor = decay_config.get("side_a_factor", 0.25)
        return (
            primary_rate * factor,
            f"INFERRED: Side A estimated at {factor:.0%} of primary"
        )

    # Excess layer decay
    decay_key = f"excess_{layer_number}"
    factor = decay_config.get(decay_key, max(0.15, 0.60 - 0.10 * layer_number))
    return (
        primary_rate * factor,
        f"INFERRED: Excess layer {layer_number} at {factor:.0%} of primary"
    )
```

## Open Questions

1. **Quote table: keep, rename, or replace?**
   - What we know: Existing tests, CLI, and pipeline integration use Quote/QuoteInput/QuoteOutput. Breaking them all at once is high risk.
   - What's unclear: Whether to keep quotes as a legacy table alongside programs/policy_years, or to refactor quotes into policy_years directly.
   - Recommendation: Keep quotes table as-is for Phase 10.1. Add new programs/policy_years/brokers/carriers tables. Map new CLI commands to new tables. Deprecate quote-centric commands gradually in a future phase. The existing `add_quote()` function continues to work by creating a Program + PolicyYear internally.

2. **LLM API key management for instructor**
   - What we know: instructor needs an Anthropic API key. The system already uses MCP tools for LLM interactions.
   - What's unclear: Whether to use the MCP Anthropic connection or a separate API key for instructor.
   - Recommendation: Use a separate ANTHROPIC_API_KEY env var for instructor. The ingestion pipeline is a CLI command, not a pipeline stage, so MCP boundaries don't apply. Document the env var requirement.

3. **Document ingestion: sync vs async**
   - What we know: The project uses sync patterns (time.sleep + Lock rate limiter). instructor supports both.
   - What's unclear: Whether large document parsing will be slow enough to need async.
   - Recommendation: Use sync. A single document extraction call takes 2-10 seconds. Not worth async complexity for a CLI command that processes one document at a time.

4. **Rate decay configuration**
   - What we know: D&O excess layer pricing follows predictable decay curves from primary rate. The exact percentages vary by market conditions.
   - What's unclear: What the right default decay factors should be.
   - Recommendation: Store defaults in config/rate_decay.json. Ship with conservative defaults (50%, 35%, 25%, 20% for excess layers 1-4). Allow override per-program. Flag all inferred values clearly.

5. **How to handle the PolicyYear / Quote coexistence during migration**
   - What we know: Existing quotes have no program_id. New data should go into programs/policy_years.
   - What's unclear: Whether to backfill existing quotes into programs or leave them in the legacy table.
   - Recommendation: Migration 003 should create programs/policy_years from existing quotes (group by ticker, infer anniversary from effective_date). Existing quotes keep their IDs. New policy_years get back-references to the original quote_id for traceability.

## CD Recommendations

### CD1: Schema Design
Use the Program -> PolicyYear -> TowerLayer hierarchy documented above. Add Broker and Carrier as separate tables with FK relationships. Key decisions:
- Program.ticker is the natural key (indexed, not PK -- use auto-increment ID for FKs)
- PolicyYear replaces Quote for new data but coexists with existing quotes
- TowerLayer gains layer_type, layer_label, data_source, carrier_id, policy_year_id
- All pricing fields nullable (premium, rate_on_line, etc.)
- DataCompleteness enum on PolicyYear for analytics filtering

### CD2: Migration Strategy
Create Alembic migration 003 with three phases:
1. **DDL Phase:** Create brokers, carriers, programs tables. Add columns to tower_layers via batch_alter_table.
2. **Policy Year Phase:** Create policy_years table with FK to programs.
3. **Data Migration Phase:** Group existing quotes by ticker, create programs, create policy_years from quotes, link tower_layers. This is a DML step within the migration.

Use `render_as_batch=True` in env.py (already configured). New nullable columns on existing tables use standard `add_column`. Column type changes or NOT NULL modifications use batch mode.

### CD3: Document Parsing Pipeline
Architecture:
1. **File Detection:** Determine file type (Excel, PDF, text) from extension
2. **Text Extraction:** Convert to text representation
   - Excel: openpyxl -> cell-level text dump with sheet/row structure
   - PDF: pdfplumber -> extracted text with table detection
   - Text/MD: direct pass-through
3. **LLM Extraction:** instructor + Claude Haiku for speed/cost -> ExtractedProgram Pydantic model
4. **Validation:** Check extracted data for sanity (limits in reasonable ranges, dates valid)
5. **Storage:** Map ExtractedProgram to Program/PolicyYear/TowerLayer ORM models
6. **Provenance:** Mark all values as AI_EXTRACTED data source

New CLI command: `do-uw pricing ingest <filepath>` with options for document type hints.

### CD4: Inference Engine Design
Create a `PricingInferenceEngine` class:
- Input: Partial tower (some layers known, others missing)
- Process: Apply rate decay curves from config/rate_decay.json
- Output: Estimated rates for missing layers, each marked INFERRED
- Confidence: Based on how many known data points anchor the inference
- Config-driven: All decay factors in JSON, not hardcoded

Inference rules:
- If primary rate known, can estimate all excess layers via decay curve
- If two adjacent layers known, can interpolate missing layers between them
- If only excess layers known (no primary), can back-calculate primary from known layer + its position
- Side A pricing requires primary + total ABC tower as inputs

### CD5: Analytics Adaptation
Update MarketPositionEngine to:
- Accept DataCompleteness filter (include COMPLETE only, or also PARTIAL)
- Weight contributions by completeness level
- Support layer-level analytics across policy_years (not just quotes)
- Add carrier rotation detection: compare adjacent policy_years for same program
- Add pricing movement tracking: rate changes between policy_years for same program
- Handle nullable pricing fields gracefully (skip records missing the needed field)

Split pricing_analytics.py (435L) into:
- pricing_analytics.py (market position engine -- ~250L)
- pricing_analytics_trends.py (trend analysis, YoY tracking -- ~250L)

Similarly split pricing_store.py (464L) into:
- pricing_store.py (core CRUD, quote compat -- ~300L)
- pricing_store_programs.py (program/policy_year operations -- ~250L)
