Add 5-link traceability chains to every check result. Each check gets explicit metadata tracking: where its data came from (DATA_SOURCE), how it was extracted (EXTRACTION), how it was evaluated (EVALUATION), where it appears in the output (OUTPUT), and how it affects the score (SCORING). A CLI audit command identifies incomplete chains.

Purpose: Enables auditing of the full data pipeline for any check result. Surfaces gaps in extraction or scoring mapping that indicate checks needing attention.

Output: Traceability fields on CheckResult, population logic in engine and mappers, CLI audit command, tests.
</objective>

    ```python
    # Traceability chain (5 links)

        default="",
        description="Link 3: Evaluation method (e.g., 'tiered_threshold:red>25%')",
    )
    trace_output: str = Field(
        default="",
        description="Link 4: Where result appears in worksheet (e.g., 'SECT3:financial_health')",
    )
    trace_scoring: str = Field(
        default="",
        description="Link 5: How result affects score (e.g., 'F1:-3.5pts')",
    )
    ```
    Also add a helper property or method:
    ```python
    @property
    def traceability_complete(self) -> bool:
        """Whether all 5 traceability links are populated."""
        return all([
            self.trace_data_source,
            self.trace_extraction,
            self.trace_evaluation,
            self.trace_output,
            self.trace_scoring,
        ])

    @property
    def traceability_gaps(self) -> list[str]:
        """Return names of missing traceability links."""
        gaps = []
        for field_name in ["trace_data_source", "trace_extraction",
                           "trace_evaluation", "trace_output", "trace_scoring"]:
            if not getattr(self, field_name):
                gaps.append(field_name.replace("trace_", "").upper())
        return gaps
    ```

    **check_engine.py** -- When the engine evaluates a check and creates a CheckResult, populate:
    - `trace_evaluation`: Describe the evaluation method used. For tiered thresholds: `"tiered_threshold:{threshold_type}:{level}"`. For boolean: `"boolean:{check_id}"`. For info: `"info_display"`. This comes from the check definition's threshold type and the evaluation logic path taken.
    - `trace_scoring`: From the check definition's `factors` list. Format: comma-joined factor IDs, e.g., `"F1,F3"`. If no factors (CONTEXT_DISPLAY check), use `"none:context_only"`.
    - `trace_output`: From the check definition's `section` field. Format: `"SECT{section}:{pillar}"`.

    Look for the code path where CheckResult objects are constructed in check_engine.py. There should be one or a few places where `CheckResult(...)` is called. Add the trace fields at those construction points. The check definition dict has `section`, `factors`, `threshold.type` -- use these to populate trace fields.

    IMPORTANT: check_engine.py is 579 lines. Do NOT add significant logic. The trace population should be 1-3 lines per CheckResult construction, reading directly from the check dict. If there's a helper needed, keep it to a small inline function (<10 lines).

    **check_mappers.py** -- Check mappers resolve data locations to actual values. When a mapper resolves a data value for a check, it knows:
    - Where the data came from (the data_location key, e.g., "extracted_financials.current_ratio")
    - Which extraction module produced it (derivable from the data_location prefix)

    Look for where mappers return resolved values. At the point where data is successfully resolved, set `trace_data_source` and `trace_extraction` on the check context or return them alongside the value. The exact approach depends on how mappers pass data back to the engine:
    - If mappers return a simple value, they can return a tuple (value, trace_data_source, trace_extraction)
    - If mappers set values on a dict, add trace keys to the dict
    - If CheckResult is constructed after mapping, populate trace fields at construction time

    The DATA_SOURCE link format: `"{source_type}:{specific_location}"` e.g., `"SEC_10K:xbrl_current_ratio"`, `"MARKET:yfinance_stock_price"`, `"LIT:stanford_scac"`.
    The EXTRACTION link format: `"{extractor_module}"` e.g., `"xbrl_extractor"`, `"market_signals_extractor"`, `"litigation_extractor"`.

    For the initial implementation, it's acceptable to derive these from the check definition's `required_data` and `data_locations` fields rather than tracking actual runtime resolution. The check dict already has `required_data: ["extracted_financials"]` and `data_locations: {"current_ratio": "extracted_financials.current_ratio"}`. Use these to populate trace_data_source and trace_extraction at CheckResult construction time in the engine, alongside trace_evaluation/trace_output/trace_scoring. This avoids modifying the mapper return signatures.
