# Requirements: D&O Underwriting Worksheet System

**Defined:** 2026-02-25
**Core Value:** Surface every red flag and risk signal that exists in public data for any publicly traded company, benchmarked against industry peers, so an underwriter can make the most informed D&O decision possible.

---

## v1.1 Requirements

Requirements for the Brain-Driven Acquisition & Data Coverage milestone. Continues from v1.0 (119 requirements, all satisfied).

### Gap Search (Brain-Driven Acquisition Feedback Loop)

- [x] **GAP-01**: System audits all 68 "Data mapping not configured" SKIPPED checks and classifies each into routing-gap, intentionally-unmapped, or aspirational bucket before any gap-search code is written
- [x] **GAP-02**: System identifies L2/L3 acquisition-tier checks with DATA_UNAVAILABLE status after structured acquisition and generates targeted web search queries from each check's brain metadata (name, required_data)
- [x] **GAP-03**: System executes gap-targeted web searches within a hard per-run cap (≤15 searches) that draws from the existing 50-search budget, with L1 checks ineligible for web fallback
- [x] **GAP-04**: System enforces an evidence quality gate: web search results may produce at most LOW-confidence advisory results; a gap-search result can never directly set a check to TRIGGERED status without keyword-presence confirmation
- [x] **GAP-05**: System re-evaluates previously SKIPPED checks after gap search, updating CheckResult to TRIGGERED or CLEAR with confidence=LOW and source="WEB (gap)" where web evidence was found
- [x] **GAP-06**: Gap search results are persisted to AcquiredData.brain_targeted_search (a Pydantic field on AnalysisState) so results are available across pipeline stages without re-searching

### Check Mapping (Data Mapping Completeness)

- [x] **MAP-01**: System adds FIELD_FOR_CHECK routing entries for all bucket-a checks (estimated ~40) where structured field data exists in ExtractedData but the routing entry is missing, reducing SKIPPED count
- [x] **MAP-02**: System adds extraction logic for bucket-b checks (estimated ~28) where the required data exists in source filings but no extractor currently reads it into ExtractedData
- [x] **MAP-03**: System expands DEF 14A LLM extraction schema (DEF14AExtraction) to include board diversity, director tenure, expertise, and attendance fields; validated against AAPL, RPM, and TSLA before routing is declared complete

### Output Quality (QA Audit & Threshold Context)

- [x] **QA-01**: QA audit table source column displays the actual filing reference or "WEB (gap)" URL for each evaluated check (fixes template bug: check.get('filing_ref') → check.get('source'))
- [x] **QA-02**: QA audit table value column displays the actual datum evaluated for each check (audit of check_evaluators.py to ensure result.value is populated for all threshold types)
- [x] **QA-03**: CheckResult model gains a threshold_context field populated from the brain check's threshold YAML (red/yellow/clear text) at evaluation time in _apply_traceability()
- [x] **QA-04**: TRIGGERED findings in red_flags.html.j2 and qa_audit.html.j2 display the human-readable threshold criterion alongside the finding value (e.g., "red: Prior SEC enforcement action within 5 years")
- [x] **QA-05**: Full regression validation on AAPL, RPM, and TSLA confirms: SKIPPED count decreases from baseline 68, TRIGGERED count on AAPL (clean company) does not increase, output quality approved on human review

## Out of Scope

Explicitly excluded from v1.1. Documented to prevent scope creep.

| Feature | Reason |
|---------|--------|
| Full ANALYZE re-run after gap search | Retriggers LLM extraction ($0.30-$1.00/run, 10-20 min delay); pipeline is explicitly one-pass |
| LLM parsing of web search snippets | Doubled LLM cost; 150-char truncated snippets are structurally inadequate for reliable LLM extraction |
| Gap search for all 68 SKIPPED checks per run | 68 uncapped searches = 68% of monthly Brave budget in a single run |
| Gap search findings as standalone worksheet section | Confidence dilution; false parity with filing evidence |
| Web UI / dashboard | CLI-first; dashboard prototype from Phase 11 not production-ready; not needed for v1.1 goals |
| Portfolio-level analytics | Different product for a different user; deferred to post-v2 |
